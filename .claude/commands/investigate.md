# /investigate — Iron Law 버그 근본 원인 분석

## Iron Law
**근본 원인 없이 수정 없다.**
증상만 치료하고 원인을 모르면 같은 버그가 반드시 재발한다.

## 트리거 조건
- "왜 이게 안되지", "버그 찾아줘", "이 에러 왜 나"
- "디버그해줘", "investigate", "에이전트가 자꾸 실패해"
- 에러 메시지를 붙여넣었을 때

---

## Phase 1: 증상 수집

에러 메시지 원문 보존. 추측으로 수정 시작 금지.

```bash
# 최근 실패한 BotLog (에이전트 관련 버그)
npx tsx -e "
import { createRequire } from 'module';
const require = createRequire(import.meta.url);
const {prisma} = await import('./agents/core/db.js');
const logs = await prisma.botLog.findMany({
  where: {status: 'FAILED'},
  orderBy: {createdAt: 'desc'},
  take: 10,
  select: {botType: true, action: true, details: true, createdAt: true}
});
logs.forEach(l => console.log(l.createdAt.toISOString().slice(0,16), l.botType, l.action, '|', (l.details??'').slice(0,100)));
await prisma.\$disconnect();
" 2>/dev/null || echo "BotLog 조회 불가 — 일반 버그일 수 있음"

# 최근 커밋 히스토리
git log --oneline -15

# 최근 변경된 파일
git diff HEAD~5 --stat 2>/dev/null || git diff --stat
```

수집할 정보:
- 에러 메시지 **원문** (요약 금지)
- 최초 발생 시점 (언제부터?)
- 재현 조건 (항상 발생? 특정 조건?)
- 최근 변경 사항과 시점 일치 여부

---

## Phase 2: 분석

수집한 증상을 바탕으로 분석:

1. **최근 커밋 관련성**: 에러 발생 시점과 가장 가까운 커밋은 무엇인가?
2. **파일 변경 추적**: 관련 파일이 최근에 변경됐는가?
   ```bash
   git log --oneline --follow -10 -- [의심 파일 경로]
   ```
3. **환경 변수**: 환경변수/외부 API 설정이 변경됐는가?
4. **패턴 매칭**: 비슷한 과거 실패가 BotLog에 있는가?

---

## Phase 3: 가설 (최대 3개)

각 가설에 대해 아래 형식으로:

```
가설 N: [1줄 설명]
검증 방법: [어떻게 확인할지]
검증 실행: [실제 확인]
결과: 확인됨 / 불일치
```

가설이 확인되면 Phase 4로. 불일치면 다음 가설.
**3가지 모두 실패하면 → BLOCKED 형식으로 에스컬레이션.**

---

## Phase 4: 구현

**근본 원인 확인 후에만 수정.**

수정 전 체크:
- [ ] 원인이 명확히 특정됐는가?
- [ ] 수정이 다른 기능에 영향을 주는가?
- [ ] 회귀 방지 방법이 있는가?

수정 후 체크:
```bash
npx tsc --noEmit    # 타입 깨지지 않았는가?
npx eslint . --ext .ts,.tsx  # 린트 통과
```

## Silent Failure 전용 (에러 없는데 결과 이상할 때)

> 트리거: "에러 없는데 이상해", "됐는데 안 보여", "테스트 통과했는데 실제론 안 돼"

### 핵심 원칙
**에러 없음 ≠ 성공.** 파이프라인 각 단계가 예외 없이 실행됐다고 해서 최종 사용자 화면이 올바른 것은 아니다.

### 1. 파이프라인 전체 명시 (코드 보기 전)
```
[사용자 행동] → [API 호출] → [데이터 변환/저장] → [렌더링] → [브라우저 리소스 요청] → [사용자 화면] ← 진짜 기준
```
각 화살표마다: "이 단계의 성공을 어떻게 확인하는가?"

### 2. 끝에서 역방향 검증
최종 실패 지점에서 거슬러 올라가라. 정방향보다 빠르다.
```
"이미지가 안 보인다" → HTML에 img 있는가? → src URL이 유효한가? → 브라우저가 그 URL fetch 가능한가? ← 끊긴 곳
```

### 3. 코드베이스에서 유사 해결책 먼저
```bash
grep -r "proxy\|HACK\|WORKAROUND\|브라우저.*실패\|직접.*실패" src/ -n
```
이미 같은 문제를 해결한 코드(주석 포함)가 있을 가능성이 높다.

### 4. "A는 되고 B는 안 된다" 처리
A의 작동을 독립 재확인 → A/B 차이점 표 작성 → 렌더링/표시 단계 차이에 집중

### 증상별 의심 포인트
| 증상 | 먼저 확인 |
|------|---------|
| 에러 없는데 화면 없음 | 렌더링/URL 변환 단계 |
| 테스트 통과, 실제 안 됨 | 테스트가 최종 출력(사용자 화면) 기준인가? |
| A는 되고 B는 안 됨 | A 독립 재확인 + 표시 단계 차이점 |
| 특정 환경(모바일)만 안 됨 | CSP, 브라우저 리소스 정책 |
| 일부 콘텐츠만 안 됨 | 같은 패턴의 다른 코드 경로 확인 |

### 재발 방지 체크리스트
- [ ] 이 수정을 같은 패턴의 다른 코드에도 적용했는가? (`grep`으로 확인)
- [ ] Playwright/E2E 테스트가 DOM 존재가 아닌 실제 렌더링(naturalWidth > 0)을 검증하는가?
- [ ] Workaround 코드에 "이 처리가 필요한 이유 + 적용해야 할 다른 곳" 주석을 달았는가?
