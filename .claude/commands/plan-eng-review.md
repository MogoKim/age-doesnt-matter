# /plan-eng-review — 엔지니어링 매니저 아키텍처 리뷰

코딩 시작 전 아키텍처를 확정하는 스킬.
**항상 코드 작성 전에** 실행. 잘못된 아키텍처는 나중에 고치면 10배 비용.

## 트리거 조건
- "엔지니어링 리뷰", "아키텍처 검토", "설계 검토"
- "코딩 시작 전 확인해줘"
- 새 에이전트 추가, 새 API 라우트, DB 스키마 변경 전

---

## STEP 0: 범위 챌린지 (항상 먼저, 생략 불가)

아래 6가지 질문에 답한 후 진행:

1. **기존 코드 재활용**: 이미 있는 함수/패턴으로 해결 가능한가? grep으로 확인
2. **최소 변경**: 핵심 목표에 꼭 필요한 파일만 포함했는가?
3. **복잡도 체크**: 8개 파일 초과 또는 2개 이상 새 클래스/서비스 도입?
   - 그렇다면 → AskUserQuestion으로 범위 축소 협의 (강제)
4. **라이브러리 확인**: 새 패턴/인프라 도입 시 context7으로 최신 문서 먼저 확인
5. **보류 작업 교차**: `memory/project_deferred_tasks.md`와 충돌하는가? 번들 기회는?
6. **완전성 확인**: 지름길인가? AI가 있으면 완전한 구현도 15분 차이. 완전한 버전 선택

**8파일 룰 위반 시 출력:**
```
⚠️ SCOPE ALERT: 현재 계획이 [N]개 파일을 건드림 (한도: 8개)
초과 파일: [목록]
범위 축소 방안:
1. [방안 1 — 어떤 파일 제거 가능]
2. [방안 2]
진행하시겠습니까? (범위 축소 후 진행 권장)
```

## STEP 1: 아키텍처 리뷰

### 컴포넌트 경계 + 데이터 흐름 다이어그램
복잡한 흐름이면 ASCII 다이어그램 **필수**:

```
예시:
[사용자] → POST /api/posts → [route.ts] → [createPost action]
                                              ↓
                                        [prisma.post.create]
                                              ↓
                              에러 시 → [AppError] → 400 응답
                              성공 시 → [revalidatePath] → 리다이렉트
```

### 보안 체크
- [ ] 이 API 라우트에 `getServerSession()` 인증이 있는가?
- [ ] 사용자 입력을 그대로 DB에 넣지 않는가? (Prisma ORM이면 OK, Raw SQL 금지)
- [ ] 환경변수 하드코딩 없는가?
- [ ] 관리자 전용 라우트에 관리자 권한 체크가 있는가?

### 에이전트 코드라면 추가 체크
- [ ] `agents/cron/runner.ts` HANDLERS에 등록 예정인가?
- [ ] 워크플로우 `.github/workflows/agents-*.yml` case문 추가 예정인가?
- [ ] BotLog 기록 (`this.log()`) 있는가?
- [ ] `// DISPATCH ONLY` 또는 `// LOCAL ONLY` 주석이 필요한가?
- [ ] COO 외 에이전트가 DB write를 하지 않는가?

## STEP 2: 코드 품질 리뷰

섹션별 이슈 발견 시 STOP → AskUserQuestion 후 계속

### TypeScript
- [ ] `any` 타입 사용 없는가? (CLAUDE.md 절대 금지)
- [ ] 적절한 타입 가드 / Zod 검증이 있는가?
- [ ] 타입 단언(`as Type`)보다 타입 가드 선호

### 코드 구조
- [ ] DRY 위반: 비슷한 로직이 이미 다른 파일에 있는가? (grep 확인)
- [ ] 에러 처리: `AppError` / `NotFoundError` / `ForbiddenError` 사용하는가?
- [ ] 서버 컴포넌트 기본, `'use client'` 불필요하게 붙이지 않는가?

### 시니어 UI 규칙 (UI 변경 시)
- [ ] 터치 타겟 52×52px 이상
- [ ] 폰트 최소 15px (본문 18px 베이스)
- [ ] 버튼 높이 52px (모바일)
- [ ] `"시니어"`, `"어르신"`, `"노인"` 표현 없는가?

## 신뢰도 점수 (모든 발견에 적용)

| 점수 | 의미 |
|------|------|
| 9-10 | 코드를 직접 읽어 확인. 구체적 버그 재현 가능 |
| 7-8 | 높은 신뢰. 패턴 기반 |
| 5-6 | 중간. 오탐 가능. "확인 필요" 표시 |
| 3-4 | 낮음. 부록에만 표시 |

**발견 형식**: `[P1] (신뢰도: 9/10) src/lib/actions/posts.ts:42 — 인증 누락`

우선순위:
- **P0**: 보안 취약점 / 데이터 손실 위험 → 즉시 차단
- **P1**: 기능 버그 / 규칙 위반 → 코딩 전 수정
- **P2**: 성능 / 코드 품질 → 코딩 중 반영 권장

---

## 완성 상태 프로토콜

- **DONE** — 모든 단계 완료. 각 주장에 증거 제시. 코딩 시작 가능
- **DONE_WITH_CONCERNS** — 완료했지만 알아야 할 이슈 존재. 이슈 목록 명시
- **BLOCKED** — 진행 불가. 차단 이유 + 시도한 내용
- **NEEDS_CONTEXT** — 필요한 정보 명시 후 AskUserQuestion
