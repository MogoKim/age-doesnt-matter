# 에이전트 코드 수정·구현 전 필수 검수 체크리스트

> **이 규칙의 존재 이유**: P1~P5가 이미 구현돼 있었는데 코드 안 보고 구현 계획을 세운 일이 있었다.
> 에이전트 코드를 새로 짜거나 수정하기 전에 반드시 이 순서를 따른다.

---

## Step 1. 핸드오버 문서 §13 먼저 확인

`docs/handover-cafe-pipeline.html` §13 "P1~P5 구현 현황" 섹션을 열어보고,
구현하려는 기능이 이미 있는지 확인한다.

---

## Step 2. 코드에서 직접 확인

```bash
# 구현하려는 함수명 또는 기능 키워드로 검색
grep -rn "함수명\|기능키워드" agents/cafe/ --include="*.ts"

# 예: 계절 필터를 추가하려고 할 때
grep -rn "SEASONAL\|season\|계절" agents/cafe/ --include="*.ts"

# 예: 키워드 quota를 추가하려고 할 때
grep -rn "quota\|overlap\|keyword" agents/cafe/ --include="*.ts"
```

grep 결과에 구현체가 나오면 → **코드 직접 읽기** (Read 도구), 없으면 신규 구현.

---

## Step 3. 병렬 함수 비교 (같은 기능 다른 케이스)

같은 처리를 하는 함수가 여러 개일 때 (예: generatePost / generateKillerPost):

1. 버그 있는 함수만 보지 말고 정상 동작하는 유사 함수도 Read 도구로 읽는다
2. 두 함수의 차이점 = 버그 있는 함수에 빠진 것
3. 정상 함수의 패턴을 그대로 적용하면 대부분 해결된다

---

## Step 4. 수정 범위 파악

같은 로직을 처리하는 다른 파일/함수가 있는지 확인한다.
한 곳만 고치면 다른 곳은 여전히 버그인 경우가 많다.

```bash
# 수정할 로직이 다른 파일에도 있는지 확인
grep -rn "수정할패턴" agents/ --include="*.ts" -l
```

---

## 자주 실수하는 패턴

| 실수 | 올바른 접근 |
|------|-----------|
| "이 기능 없는 것 같으니 만들자" | grep + 핸드오버 문서 먼저 확인 |
| 버그 있는 함수만 본다 | 동일 처리하는 다른 함수도 비교 |
| 한 파일만 수정 | 같은 로직 다른 파일도 확인 |
| 코드 가정으로 계획 수립 | Read 도구로 실제 코드 확인 후 계획 |
