# 파이프라인 묵시적 실패 디버깅 방법론
> Silent Pipeline Failure — 에러 없이 실행됐는데 결과가 이상한 모든 버그

## 이 방법론이 필요한 증상
- "에러 없이 실행됐는데 화면에 결과가 안 나온다"
- "업로드/저장은 됐는데 표시가 이상하다"
- "A는 작동하는데 B는 안 된다 (거의 같은 코드인데)"
- "QA 테스트는 통과했는데 실제 사용자는 못 쓴다"

---

## 핵심 공리

> **에러 없음 ≠ 성공.**  
> 파이프라인의 각 단계가 예외 없이 실행됐다고 해서 **최종 사용자 화면이 올바른 것은 아니다.**

이번 사례:
- presign API → 200 ✅
- R2 PUT → 200 ✅
- editor.setImage({src}) → DOM에 img 삽입 ✅
- 브라우저 img src 로드 → **실패** ❌ (에러 없음, 깨진 아이콘만 표시)
- QA 테스트 `imgCount=1` → **통과** ✅ (DOM 존재만 확인했기 때문)

---

## 디버깅 5단계

### Step 1: 파이프라인 전체 명시
코드를 보기 전에, Input → 최종 사용자 화면까지 모든 단계를 명시적으로 나열한다.

```
[사용자 행동]
→ [API 호출] → 응답?
→ [데이터 변환/저장] → 결과?
→ [렌더링] → HTML?
→ [브라우저 리소스 요청] → 로드?
→ [사용자 화면] ← 여기가 진짜 기준
```

각 화살표마다 "이 단계의 성공을 어떻게 확인할 수 있는가?" 질문.

---

### Step 2: 끝에서부터 역방향으로 검증
최종 실패 지점에서 시작해 거슬러 올라간다. 정방향(처음부터 추적)보다 빠르다.

```
"이미지가 안 보인다"
→ HTML에 <img> 있는가? → YES (이 위는 정상)
→ <img> src가 유효한 URL인가? → YES
→ 브라우저가 그 URL을 fetch할 수 있는가? → NO ← 여기서 끊김
→ 이제 "왜 이 URL에 접근 못하는가"만 집중
```

**역방향이 중요한 이유**: 정방향으로 가면 "업로드 로직은 맞다" 결론이 나서 업로드 코드를 계속 파게 된다. 실제 문제는 렌더링 단계였다.

---

### Step 3: 코드베이스에서 유사 해결책 먼저 검색
가설을 세우기 전에, 이미 같은 문제를 해결한 코드가 있는지 확인한다.

```bash
# 비슷한 workaround/fix 검색
grep -r "proxy\|프록시" src/ -n
grep -r "브라우저\|browser" src/ -n        # 한국어 주석 포함
grep -r "TODO\|FIXME\|HACK\|WORKAROUND" src/ -n
grep -r "직접.*실패\|실패.*직접" src/ -n
# 비슷한 기능이 어떻게 구현됐는지 확인
grep -r "Magazine\|magazine" src/ -l       # 이번 경우 proxyMagazineImages 발견
```

**이번 사례**: `sanitize.ts`의 `proxyMagazineImages` 주석에 **이미 정답이 적혀 있었다**:
```
"브라우저에서 R2 직접 요청이 실패하므로 /_next/image 프록시 경유"
```
이것을 첫날 찾았다면 30분에 해결됐다.

---

### Step 4: "A는 되고 B는 안 된다" 증상 처리
이 증상은 **강력한 미끼**다. "A가 작동하니까 A와 관련된 것은 원인이 아니다"는 논리로 빠지게 한다.

**올바른 처리**:
1. A의 작동을 **독립적으로 재확인**한다 (사용자가 방금 올린 건가? 시드봇 콘텐츠인가?)
2. A와 B의 **차이점 표**를 만든다
3. 차이점 중 표시(렌더링) 단계에 집중한다

| | 이미지 | 영상 |
|---|---|---|
| 업로드 API | `/api/uploads/presign` | `/api/uploads/video/presign` |
| editor 삽입 | `setImage({src})` | `insertContent({type:'video'})` |
| sanitize 처리 | **없음** | 없음 |
| 최종 표시 | `<img src="R2_URL">` | `<video src="R2_URL">` |

→ "둘 다 같은 처리인데 이미지만 안 되면, 영상도 실제로 유저가 방금 올린 건지 먼저 확인해야 한다."

---

### Step 5: 테스트는 최종 사용자 화면 기준으로 작성

| | 잘못된 테스트 | 올바른 테스트 |
|---|---|---|
| 이미지 업로드 | `imgCount=1` (DOM 존재) | 이미지가 실제로 로드됐는가 |
| API 호출 | 200 응답 | 응답 데이터로 UI가 올바르게 표시됐는가 |
| 저장 | DB row 존재 | 다시 불러왔을 때 화면에 올바르게 표시됐는가 |

```typescript
// ❌ DOM 존재만 확인
const imgCount = await page.locator('.ProseMirror img').count()
expect(imgCount).toBeGreaterThan(0)

// ✅ 실제 로드 확인
const img = page.locator('.ProseMirror img').first()
await expect(img).toBeVisible()
const naturalWidth = await img.evaluate((el: HTMLImageElement) => el.naturalWidth)
expect(naturalWidth).toBeGreaterThan(0)  // 0이면 로드 실패
```

## 재발 방지 — 코드 작성 시 체크리스트

새 기능을 추가하거나 버그를 수정할 때:

### 부분 수정 적용 시 (가장 흔한 실수)
- [ ] 이 수정이 적용된 케이스 외에 **같은 패턴이 있는 다른 코드**가 있는가?
  - 이번 사례: magazine에만 적용 → regular posts에도 적용 필요했음
  - 확인 방법: `grep -r "유사함수명\|유사로직" src/ -n`

### 파이프라인 수정 시
- [ ] 파이프라인의 **최종 단계(사용자 화면)**까지 검증했는가?
- [ ] Playwright/E2E 테스트가 **중간 단계가 아닌 최종 출력**을 검증하는가?

### Workaround 추가 시
- [ ] 이 워크어라운드가 필요한 **다른 코드 경로**가 없는가?
- [ ] 주석에 "이 처리가 필요한 이유"와 "적용해야 하는 다른 곳"을 명시했는가?

```typescript
// ✅ 좋은 주석 패턴
/**
 * 브라우저에서 R2 직접 요청이 실패하므로 /_next/image 프록시 경유
 * [APPLIES TO] 사용자 게시글(regular posts)도 동일 처리 필요
 * → proxyR2Images() 함수 참고
 */
```

## 빠른 참고 — 증상별 의심 포인트

| 증상 | 먼저 확인할 것 |
|---|---|
| 에러 없는데 화면에 안 나옴 | 렌더링/표시 단계의 URL/데이터 변환 |
| 테스트 통과했는데 실제로 안 됨 | 테스트가 최종 출력 기준인가? |
| A는 되는데 B는 안 됨 | A의 작동 독립 재확인 + 표시 단계 차이점 |
| 특정 환경(모바일)에서만 안 됨 | CSP, 브라우저 리소스 정책, 네트워크 요청 차단 |
| 예전엔 됐는데 최근에 안 됨 | 부분 수정이 전체에 적용됐는지 확인 |
| 일부 콘텐츠만 안 됨 | 동일 처리가 다른 경로에도 있는지 확인 |
