# 서비스 개발 후 내가 계속 확인해야 할 것 (봇 운영 & 데이터 전략)

> **작성 목적**: 웹사이트(Phase 1~3) 개발 완료 후, 운영 초기(Phase 4~5)에 적용할 100명 규모의 가상 유저(AI 에이전트) 운영 전략과 이전에 논의된 아이디어를 잊지 않고 구현하기 위해 기록합니다.
> **주의사항**: 일반적인 웹서버의 '멀티스레드'가 아닌, 비동기 큐(Message Queue)와 스케줄러(Cron)를 활용하는 백그라운드 아키텍처입니다.

---

## 🚀 1. 핵심 아키텍처: 어떻게 100명이 활동하는가?

**잘못된 접근**: 웹 서버 내에서 100개의 스레드를 띄워서 무겁게 돌리는 방식 (서버 부하 및 차단 위험)
**올바른 접근**: 봇들이 각자 자신의 스케줄(타이머)에 맞춰 깨어나, API를 호출하고 자는 '이벤트 기반 스케줄러' 방식

1. **시간차 무작위 반응 (Delay & Jitter)**
   - 글이 올라오면 봇 A는 3분 뒤, 봇 B는 12분 뒤, 봇 C는 45분 뒤에 점진적으로 댓글을 달도록 타이머를 부여하여 매크로 느낌을 지웁니다.
2. **역할 분담 체계**
   - **수집 봇(Crawler)**: 하루 1~2번 최신 트렌드를 조사하는 정탐꾼.
   - **에디터 봇**: 조사해온 원문을 요약해서 '트렌드 보고서' 작성.
   - **알바생 봇 (100명)**: 트렌드 보고서를 기반으로 각자의 페르소나(예: 60대 은퇴남성, 55세 주부 등)에 맞춰 글과 댓글을 다는 실무 봇.

## 🔍 2. 트렌드 학습(가스라이팅) 전략

타 커뮤니티의 실시간 분위기를 우리 웹사이트의 봇들에게 이식해 "진짜 사람 사는 곳"처럼 보이게 하는 전략.

1. **수집 타겟 (난이도 구분)**
   - 네이트판, 펨코 (공개형, 수집 용이): 핵심 타겟
   - 네이버 카페 '우아한 갱년기' 등 (폐쇄형, 까다로움): 로그인 세션 필요 (추후 고도화)
2. **로컬 전용 크롤러 아키텍처 (Mac Mini + Puppeteer 워커)**
   - 100명의 봇이 각자 타 사이트에 들어가는 것이 아닙니다. (IP 전면 차단 위험)
   - **온프레미스(On-premise) 방식 채택**: 사무실/집에 24시간 켜져 있는 남는 PC(맥미니 등)를 **'수집 전용 워커'**로 지정합니다.
     - **안티봇 우회**: 클라우드(AWS/Vercel) IP가 아닌 일반 가정/사무용 가정용 인터넷(ISP) IP를 쓰기 때문에 네이버, 펨코 등의 보안 시스템(Cloudflare 등)을 완벽하게 패스합니다.
     - **비용/성능 제로화**: 퍼피티어(크롬을 통째로 띄우는 무거운 작업)를 돌려도 클라우드 서버 요금이 나오지 않으며, 우리 메인 웹사이트 트래픽과도 완전히 분리됩니다.
   - 맥미니가 매일 퍼피티어로 오늘의 트렌드(베스트 글)를 긁어온 뒤, 클라우드의 본 데이터베이스(Supabase API)로 전송(Insert)해 줍니다.
3. **지식 주입 (Context Injection)**
   - 요약된 트렌드를 100명의 봇에게 프롬프트(Context)로 던져줍니다.
   - 봇은 자신의 성향(페르소나)과 주입된 트렌드를 섞어서 극도로 자연스러운 글을 작성합니다.

## 📈 5. 데이터 링겔 (Data-Driven PDCA 루프) 세팅: GA4 & GTM 친화적 설계

이 모든 AI 자동화가 "무성한 잡초"가 아니라 "정성스럽게 가꾼 정원"이 되려면, 반드시 다음과 같은 무한 반복 루프가 필요합니다.

`Data ➔ 문제정의/아이데이션 ➔ 가설 ➔ (대표님의 피드백) ➔ 실험 ➔ 결과 ➔ 다시 Data...`

AI 에이전트(특히 CEO 봇과 애널리스트 봇)가 이 루프를 돌기 위해서는 **"양질의 데이터"**가 필수적이며, 이를 위해 프론트엔드 코드(Next.js)단부터 GA4와 GTM(Google Tag Manager) 친화적으로 설계되어야 합니다.

### 왜 GA4 / GTM이 필수인가요?
- 에이전트가 "어제 A 주제의 글이 반응이 좋았네? 내일은 이쪽으로 봇들을 투입하자!"라고 **판단(결과 도출)**하려면, 사람들이 스크롤을 얼마나 내렸는지, 어느 버튼을 많이 눌렀는지 정확한 지표가 필요합니다.

### 어떻게 코드를 수정해야 할까요? (추후 개발 가이드)
1. **모든 뼈대 컴포넌트에 Data-Attribute 추가**
   - 버튼과 링크마다 GTM이 쉽게 잡을 수 있도록 특정 속성 부여.
   - 예: `<button data-gtm-id="btn_post_like" data-gtm-tag="health_board">공감</button>`
2. **이벤트 루프 심기 (dataLayer Push)**
   - 중요한 이벤트(글 작성, 가입 완료 등)가 발생할 때마다 `window.dataLayer.push()`를 통해 GA4에 구체적인 정보를 전송하도록 Next.js 이벤트 핸들러를 수정.
3. **AI 피드백 통로 모으기**
   - 수집된 GA4 결과 데이터를 구글 시트 조작이나 애널리틱스 API 등을 통해 CEO 봇에게 매일 아침 던져줌으로써 **[가설 ➔ 실험 ➔ 피드백]**의 사이클이 비로소 끊기지 않고 돌아가게 됩니다.
