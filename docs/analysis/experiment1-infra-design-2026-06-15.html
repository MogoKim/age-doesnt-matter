1. 5:5 균등 배정 — 해시를 바꿉니다 갭①

🔴 지금 문제 방문자를 A/B로 가르는 배정 해시가 약합니다(assign.ts의 글자 코드 단순 합산). 기기 ID(UUID)의 글자값을 더하면 값이 가운데로 몰려, 절반으로 나눠도 5:5가 안 맞고 한쪽으로 치우칠 수 있습니다.
A 58%
B 42%
↑ 치우치면 실험 결과를 신뢰할 수 없습니다(표본 불균형).
✅ 해결 (djb2 해시로 교체 — 창업자 승인) 검증된 균등 분배 해시(djb2)로 바꿉니다. 글자를 더하지 않고 곱셈으로 섞어 값이 고르게 퍼집니다.
A 50%
B 50%
  • 안전한 이유: 지금 진행 중인 실험이 0개라(어드민에서 확인됨) 공용 해시를 바꿔도 깨질 실험이 없습니다. 앞으로의 모든 실험이 5:5로 정확해집니다.
  • 검증 방법: 가짜 기기 ID 1만 개를 만들어 배정을 돌려 49~51% 안인지 확인하는 임시 스크립트(통과 후 삭제). 배포 후 실측 배정 비율도 어드민에서 모니터링.
  • 재방문 일관성: 한번 A로 배정된 기기는 다시 와도 A(기기 ID 기반 + 로컬 캐시). 실험 도중 variant가 바뀌어 데이터가 오염되지 않습니다.

2. 시작 시점 데이터 컷 — 과거 데이터를 배제합니다 갭②

📌 창업자 요구 "시작 기간을 기점으로 정확한 데이터가 나와야지 과거 데이터 들고 오면 안 된다."

두 겹으로 보장합니다:

① 노출 이벤트는 실험 배포 후에만 발생→시작 전 데이터는 애초에 존재하지 않음
② 집계에서 "시작일 이후"로 한 번 더 자름→start = max(최근 N일, 실험 시작일)
운영 주의 실험을 등록한 뒤 어드민에서 상태=진행중 + 시작일을 입력해야 이 컷이 작동합니다. (어드민 /admin/ab-tests에 이미 있는 기능) → 시작일 이전 트래픽은 절대 안 섞입니다.

3. 3화면·D1 측정을 새로 깝니다 갭③ · 가장 큰 작업

🔴 지금은 "가입" 밖에 못 봅니다 현재 어드민 A/B는 "노출 → 가입 전환율"만 측정합니다. 그런데 실험 1의 핵심 지표는 3화면 도달률·D1 재방문입니다 — 이걸 보는 기능이 아예 없습니다. 새로 만들어야 합니다.

어떻게 측정하나 (이미 쌓이는 데이터 활용)

노출→글 상세 들어오면 "이 사람은 A(또는 B)였다" 1회 기록
3화면 도달(기본)→그 사람이 노출과 같은 날 화면을 3개+ 봤나 (기존 D1 분석과 비교 가능)
추가 탐색(보조)→노출 "이후" 글을 더 봤나 / 카드를 클릭했나 (B카드가 실제 다음 글을 만들었나)
D1→그 사람이 다음날 다시 왔나 (다음날 페이지뷰 있나)
광고→세션당 페이지뷰(노출 기회 proxy) · 애드센스는 전체 수익 가드레일
지표정의역할
3화면 도달률 (기본)같은 sessionId, 노출과 같은 KST 날짜 page_view ≥ 3주지표 · 기존 D1 분석과 비교 가능
노출 이후 추가 탐색 (보조)노출 이후 같은 sessionId page_view +1 이상 또는 카드 클릭(inline) 발생B카드가 실제로 다음 글 클릭을 만들었나
D1 재방문율노출 다음날(KST) page_view ≥ 1주지표(최종) · 95% 유의 검정
세션 페이지뷰A·B 각 그룹 세션당 화면 수 평균주지표 · 광고 노출 기회 proxy
카드 클릭(inline)related_post_click(position='inline')주지표 · B 전용(이미 있는 기능)
애드센스 RPM·노출애드센스 대시보드(variant 분리 불가, 전체값)가드레일 — 큰 하락 시 롤백/재검토
⚖️ 애드센스 수익은 "직접 답"이 아니라 "가드레일" 현재 인프라로는 A/B별 애드센스 RPM을 분리 측정하지 못합니다(애드센스 대시보드는 전체값). 그래서 수익은 세션당 페이지뷰(노출 기회 proxy)로 비교하고, 애드센스 RPM·노출은 전체 수익이 크게 빠지지 않는지 지키는 가드레일로 둡니다. 위너는 3화면·D1·페이지뷰·카드클릭으로 판정하고, 애드센스가 크게 하락하면 롤백/재검토합니다.
핵심 화면수·재방문의 원천 데이터(page_view)는 이미 모든 페이지에서 쌓이고 있습니다. 새로 만드는 건 "실험 그룹별로 그걸 묶어 보여주는 집계 + 어드민 카드"뿐입니다.

4. B카드 화면 구현 — "코드가 꼬이지 않게" 갭④

⚙️ 제약 글 상세 페이지는 미리 만들어 캐시(force-static)됩니다 → 서버는 이 방문자가 A인지 B인지 모릅니다. 그래서 브라우저(클라이언트)에서 A/B를 가릅니다.
  • 다음글 3개 데이터는 서버가 이미 뽑아 둔 관련글에서 상위 3개를 그대로 씁니다(새 추천엔진 불필요).
  • 새 컴포넌트 하나(NextPostsInline)만 추가 — 광고① 다음 자리에 둡니다. 그 안에서 "나는 A냐 B냐" 확인:
    • B → 다음글 3개를 보여줌 글 진입 즉시(창업자 결정)
    • A → 아무것도 안 보여줌(현행 유지)
    • 단 노출 기록은 A·B 둘 다 남김(비교 분모를 정확히).
  • 화면 안 튀게(CLS 방지): 판정 전엔 빈 상태, 확정 후 표시 — 기존 배너들이 쓰는 검증된 패턴 그대로.
  • 코드 격리: 변경은 새 컴포넌트 1개 + 글 상세에 한 줄 삽입뿐. 광고·댓글·기존 로직은 안 건드립니다 → 다른 기능과 안 꼬임.
엣지케이스 관련글이 3개 미만인 글은 있는 만큼만 보여주고(0개면 B=A 동일), 분석에서 따로 표기합니다.

5. 위너 결정되면 나머지 즉시 드랍

📌 창업자 요구 "위너가 결정되면 나머지는 드랍 바로 가능하게." → 과거 실험(TWA 게이트 등)에서 쓴 검증된 2단계 방식.
1단계 · 즉시 중단→진 쪽 비중을 0으로(registry) → 신규 방문자는 더 이상 그쪽에 안 배정 (어드민 상태 변경이 아니라 비중 0/코드 배포)
2단계 · 완전 종료→실험 정의 제거 + 위너 코드만 남김(분기 삭제) + 결론 기록
위너최종 처리
B 승다음글 3개를 항상 표시로 고정(A/B 분기 제거)
A 승새 컴포넌트를 제거(현행으로 복귀)

두 경우 모두 tsc·build 통과 확인 후 반영. 실험 흔적이 깔끔히 사라집니다.

위험대비
5:5 안 맞음djb2 교체 + 1만개 시뮬 49~51% + 배포 후 실측 모니터링
과거 데이터 섞임노출 이벤트 시작 후 발생 + 시작일 컷 이중
화면 튐(CLS)판정 전 빈 상태 → 확정 후 표시(기존 패턴)
다음날 추적 끊김쿠키 삭제 시 D1 일부 누락 가능 — 단 A·B 동일 영향이라 비교는 유효(분석서 명시)
다른 기능과 충돌변경 = 새 컴포넌트 1개 + 한 줄 삽입으로 격리. 광고·댓글·인증 무변경

7. 2차 보완 — 다시 점검해서 막은 구멍 4개 중요

1차 설계 후 코드를 한 번 더 뜯어보니 실험을 망가뜨릴 수 있는 구멍 4개가 더 있었습니다. 전부 막는 설계를 포함합니다.

① 배포 ≠ 시작 — "올려둬도 시작 전엔 아무 일 안 일어나게" 가장 치명적

🔴 구멍 앞서 말한 "시작일 컷"은 어드민 집계에만 적용됩니다. 정작 방문자 화면(클라이언트)에는 시작 시각을 막는 장치가 없어, 실험 코드를 배포하는 순간 시작일 전에도 A/B 배정·노출이 일어납니다 → 시작 전 데이터가 섞입니다.
✅ 해결 실험에 시작 시각(startsAt)을 넣고, 지금 시각이 시작 전이면 배정·노출·B카드 표시를 전부 건너뜁니다.
실험 코드 미리 배포→시작 시각 전 = 아무 일 없음(현행과 동일)→시작 시각부터 노출 시작
→ 배포와 실험 시작을 분리. 클라 시작 시각 = 어드민 시작일을 동일하게 맞춰 데이터가 정확히 시작점부터.
📌 적용 순서(중요) 이 시작 시각 체크는 로컬에 저장된 기존 배정(캐시)을 읽기 전, 가장 먼저 적용됩니다. 시작 전이면 캐시가 있어도 variant 반환·노출·B카드 표시가 전부 없습니다.
또 하나 어드민의 실험 상태(진행/종료)는 "기록·집계용"일 뿐 방문자 화면을 막는 장치가 아닙니다. 진행 중 즉시 중단이 필요하면 비중을 0으로 바꾸거나(registry) 코드를 배포해야 합니다.

② 노출 데이터 유실 방지 — "분모가 빠지면 실험이 망가진다"

🔴 구멍 노출 기록(exp1_exposure)이 요청 제한(rate limit)에 안 걸리게 면제돼 있지 않습니다. 노출은 글 볼 때마다 발생하는데, 다른 이벤트와 한도(분당 30)를 공유해 초과분이 버려지면(429) 비교의 분모(전체 노출 수)가 빠져 3화면·D1 비율이 왜곡됩니다.
✅ 해결 exp1_exposure를 필수 측정 이벤트로 분류해 rate limit 면제 목록에 추가. (가입·관련글 클릭 등은 이미 면제돼 있음 — 같은 처리.)

③ 어드민 화면 즉시 반영 — "저장은 정확한데 화면이 10분 늦던 것"

🔴 구멍 실험 상태(시작/진행)를 바꾸면 DB 저장은 즉시 정확하지만, 집계 화면은 10분 캐시에 막혀 늦게 반영됩니다(저장 시 캐시를 안 깨서).
✅ 해결 진행 중 실험 집계는 저장 즉시 캐시를 무효화(태그 방식)하거나 짧은 캐시로. → 데이터는 항상 정확, 화면도 바로 반영.

④ B카드 문구 정합성 — 데이터와 안 어긋나게

📌 구멍 본문 직후 보여줄 글은 "전체 인기글"이 아니라 같은 게시판의 비슷한 글(관련글)입니다. 그래서 "다음 인기글"이라 쓰면 데이터와 어긋납니다.
