1. 회원 전용 가입인사 플로우 ✔ 정책①

✅ 기존 구조가 정책과 정확히 맞음 createPost(posts.ts:23-25)는 auth() 없으면 "로그인이 필요합니다" 반환. Post.authorId는 필수(NOT NULL). → 가입인사 글도 같은 경로 = 회원만. 별도 권한 로직 불필요.

한 줄 인사 전용 폼 (경량)

  • 전용 경량 폼: 제목 없이 한 줄, category='가입인사' 고정, boardType=STORY. 기존 글쓰기 페이지(write/page.tsx) 아님(허들↓).
  • 제출 → createPost(또는 전용 액션 `createGreetingPost`) → ① Post 생성 ② User.firstGreetingAt/PostId set(3번) ③ enqueueUserPostWave 자동 환대(기존).
  • 입구: B위젯(가입 직후 홈 최상단·72h·미작성) + 신입환영 섹션(상시).

2. 비회원 = 환영댓글만 + 가입유도 ✔ 정책②

  • 비회원은 가입인사 글을 쓰지 않는다(Post 스키마 변경 0, 무거운 마이그레이션 회피).
  • 신입환영 섹션의 신입글에 "환영 댓글"만 작성: 기존 createGuestComment(닉네임+비번+Turnstile) 재사용.
  • 댓글 후 가입 유도: "환영 댓글 고마워요! 내 글 쓰고 알림 받으려면 가입" → 카카오 가입 깔때기.
  • 비회원은 알림 못 받음(userId 없음) — 이 약점을 가입 전환 트리거로 사용.
역할 분담 회원 = 가입인사 글 작성(자산감 강) / 비회원 = 환영 댓글 참여(가벼움) + 가입 전환. 둘 다 "참여 → 리텐션/전환".

3. firstGreeting 이력 필드 — schema·migration·write·read·rollback 정책③

schema (User 모델)

model User {
  ...
  firstGreetingAt     DateTime?   // 첫 가입인사 작성 시각 (없으면 미작성)
  firstGreetingPostId String?     // 그 글 id (삭제돼도 이력 유지 → 위젯 재노출 방지)
}

기존 pwaInstalledAt·nicknameChangedAt 등 nullable DateTime 패턴과 동일. 신규 컬럼 2개.

migration

  • ADD COLUMN (nullable) 2개 — 무중단(기존 행은 NULL=미작성). npx prisma migrate. 인덱스 불필요(User PK 조회).

write path

  • 가입인사 글 생성 액션에서 firstGreetingAt이 NULL일 때만 `firstGreetingAt=now, firstGreetingPostId=post.id` set(최초 1회 고정). 두 번째 인사글은 이력 안 바꿈.
  • 트랜잭션: Post create + User update 원자적(또는 Post 후 User update, 실패해도 비크리티컬).

read path

  • B위젯 노출 판별: `User.createdAt < 72h` AND `firstGreetingAt == null`. → 글을 삭제·숨김해도 firstGreetingAt이 남아 재노출 안 됨(정책③ 핵심).
  • 판별 데이터는 세션(JWT)에 firstGreetingAt 포함 권장(홈 server auth 금지 — 6번). source of truth = DB.
🆕 v0.92 — 작성 직후 위젯 즉시 숨김 (방식 1개로 확정) auth.ts:79-84는 trigger==='update' 시 tokenRefreshedAt=0으로 "다음 요청"에 DB 재조회 — 즉 firstGreetingAt을 JWT에 넣어도 작성 직후 즉시 클라 세션엔 반영 안 됨. 그래서:
순서처리역할
① 주 방식작성 성공 → 로컬 상태(setHidden)로 위젯 즉시 숨김즉각 UX(현재 화면)
②localStorage first_greeting_done 플래그같은 기기 새로고침 시 JWT 갱신 전이라도 재노출 방지
③unstable_update() 호출(tokenRefreshedAt=0)JWT firstGreetingAt 다음 로드 반영(다른 기기·장기, DB가 진실)
채택 안 함: eligibility API(매 홈 로드 요청 = 홈 성능·6번 server auth 금지 원칙 위배).

rollback

롤백 종류영향
코드만 롤백컬럼 nullable이라 안전 — 코드가 컬럼 안 읽으면 무시됨. 데이터 손실 0
컬럼 DROP별도 마이그레이션. 이력 데이터 손실 → 롤백 시엔 컬럼 유지 권장(미사용 컬럼은 무해)

4. A/B exposure — A가 UI 안 그려도 기록 핵심

✅ exp1에서 검증된 패턴 그대로 NextPostsInline: variant 확정 시 A·B 모두 1회 노출 기록(분모), variant==='' 만 skip, rendered는 B 여부로 구분.
  • eligible 기준 기록: B위젯/신입환영 컴포넌트가 마운트되어 "노출 자격"(가입 72h내·미작성 등)을 만족하면, A는 UI를 안 그려도 trackEvent('first_greeting_exposure',{variant, surface, rendered}) 1회.
  • 이래야 분모(A·B 노출 세션)가 공정 → 참여율·D1 비교가 정확. (A만 안 기록하면 분모 왜곡 = exp1과 동일 함정)
  • firedRef로 세션당 1회. variant 미확정/시작 전('')은 skip.

8. wave 댓글 품질 — 가입인사 전용 prompt 분기

  • generateUserPostComment(generator.ts:604)는 일반 글 가정. "오늘 가입했어요" 인사글에 일반 댓글 prompt면 어색할 수 있음(예: 인사글에 공감/조언 댓글).
  • 설계: user-post-wave-processor가 post.category==='가입인사' 감지 → 전용 환영 prompt로 분기(generateGreetingWelcome 또는 generateUserPostComment에 kind 인자). "환영해요~", "반가워요 ^^" 류 자연스러운 환영 멘트.
  • 개발 시 실제 생성 댓글 샘플 검수(인사글 5건 → wave 댓글 자연스러운지) 후 prompt 조정.

PASS 기준 (개발 검증)

  • 배정 A/B 49~51%(djb2 시뮬 1만).
  • A도 exposure 기록(eligible 기준, UI 미렌더 시에도).
  • 가입인사 0건 누출 — 전체·베스트·홈 trending/stories·관련글·검색에서 가입인사 글이 안 보임(5번 전수).

  • 삭제 후 재노출 0 — 글 삭제해도 firstGreetingAt 유지로 위젯 안 뜸.
  • wave 환대 적절 — 가입인사 글에 환영 댓글(전용 prompt) 자연스러움.
  • 홈 성능 — server auth() 없음, 홈 캐시 유지.
  • 🆕 작성 직후 위젯 즉시 숨김 — 작성 성공 시 로컬 상태로 즉시 사라짐 / 같은 기기 새로고침 후 재노출 0(localStorage) / 다른 기기·장기는 firstGreetingAt(DB·JWT). eligibility API 미사용.
