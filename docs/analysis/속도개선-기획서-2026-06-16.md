# 속도 개선 기획서 — 리전 미스매치 해소

> 작성일: 2026-06-16 · 작성: Claude Code · 대상: 창업자 검토용
> 한 줄 요약: **사용자(한국)·서버 함수(미국)·DB(시드니)가 전부 다른 대륙에 있어** 로그인·동적 화면이 느립니다. 가장 가성비 높은 해법은 **서버 함수를 DB와 같은 리전(시드니)으로 옮기는 것**(설정 한 줄).

---

## 1. 배경

- 우나어는 **한국 40·60대 여성**이 주 사용자이고, 대부분 **모바일(TWA 앱)**로 접속한다.
- 최근 전체 공지(종 알림) 클릭 → `/best` 이동을 테스트하던 중, **이동은 되지만 "너무 느려서" 안 되는 것처럼 보이는** 현상을 확인했다.
- 캐시되는 익명 페이지(홈·베스트 첫 화면 등)는 빠른데, **로그인 상태·동적 화면·캐시 미스**(알림, 베스트 탭/페이징, 글 작성, 공지 등)에서 체감 지연이 크다.

## 2. 문제 원인 (측정 근거)

### 핵심: 삼중 리전 미스매치
| 구성요소 | 현재 위치 | 비고 |
|---|---|---|
| 👤 사용자 | **한국** | KT 등 국내 |
| ⚡ 서버 함수(Vercel) | **미국 동부(iad1)** | 리전 설정 없음 → 기본값 |
| 🗄️ DB(Supabase) | **시드니(aws-1-ap-southeast-2)** | 함수와 1만 km |

→ 한 요청이 **한국 → 미국 함수 → 시드니 DB(쿼리 N번) → 미국 → 한국**을 왕복한다. 최악의 경로.

### 측정 데이터 (2026-06-16)
| 측정 대상 | 결과 | 의미 |
|---|---|---|
| DB 쿼리 1회 왕복 | **208 ms** | 함수↔DB가 태평양 횡단 |
| 홈/베스트/커뮤니티 (캐시됨) | TTFB **0.1~0.2초** | CDN이 가려줌 → 빠름 |
| **`/api/best` 캐시 미스** | TTFB **1.7초** 🔴 | 함수가 시드니 DB에 여러 쿼리 → 폭발 |

- **캐시가 가려주는 익명 화면은 빠르지만**, 캐시가 없는 곳(로그인/동적/미스)은 **쿼리 수 × 200ms**를 그대로 맞는다.
- 페이지 하나가 DB 쿼리 4~5개를 직렬로 하면 그것만 **0.8~1.0초+**. 여기에 한국↔미국 왕복까지 더해진다.

### 왜 지금까지 안 드러났나
- 익명 방문(SEO·신규)은 CDN 캐시라 빠르게 느껴졌다.
- **로그인 회원의 실사용 경험**(알림·베스트 탐색·글쓰기)에서만 지연이 누적돼, 측정 전엔 "감"으로만 느껴졌다.

## 3. 목적

- **로그인 회원의 동적 화면 응답을 1.7초대 → 0.3초대로** 단축한다(캐시 미스·동적 경로 기준).
- 체감 속도를 올려 **재방문·체류·전환(가입/구독)**을 개선한다(시니어 친화 UX는 속도가 곧 이탈 방지).
- "느려서 기능이 안 되는 줄 아는" 오인을 제거한다(이번 공지 클릭 사례).

## 4. 해결 방안 (우선순위)

### ✅ 1순위 — 서버 함수를 시드니로 이전 (즉효·저위험·설정 한 줄)
- `vercel.json`에 `"regions": ["syd1"]` 지정 → 함수를 **DB와 같은 시드니**에 둔다.
- 효과: 쿼리 왕복 **208ms → ~2ms**. `/api/best` 미스 **1.7초 → ~0.3초** 예상.
- 보너스: **시드니가 미국 동부보다 한국에 더 가깝다**(한국→시드니 ~150ms < 한국→미국 ~200ms) → 사용자↔함수 지연도 오히려 개선. **모든 축에서 이득.**
- 전제: **Vercel Pro 플랜**이어야 리전 선택 가능(Hobby는 iad1 고정).

### 🔭 2순위 — DB를 서울로 이전 + 함수도 서울(icn1) (장기·최적·큰 작업)
- 사용자·함수·DB 전부 서울 → 총 지연 ~50ms(궁극 최적).
- 단 Supabase는 리전 인플레이스 변경 불가 → **새 프로젝트(서울) 생성 후 데이터 이전** = 대형 마이그레이션(다운타임·검증 필요). 1순위 효과 확인 후 별도 검토.

### 🧩 3순위 — 페이지당 쿼리 수 축소 / 동적 경로 캐시 (보조)
- compose·알림 등 직렬 쿼리 병렬화·캐시. 1순위 적용 후 남는 병목만 선별 대응.

## 5. 기대 효과
- 동적/캐시미스 응답 **약 5배 단축**(1.7s→0.3s 수준).
- 알림 클릭→이동, 베스트 탐색, 글쓰기 등 **실사용 체감 즉각 개선**.
- 추가 비용 거의 없음(리전 변경은 요금 동일, Pro 한정).

## 6. 리스크 · 전제
- **Vercel 플랜 확인 필수**(Pro여야 1순위 가능). Hobby면 2순위(DB 이전) 또는 3순위로.
- 함수 리전 변경 후 **재배포 + 측정**으로 효과 검증(미스 TTFB 재측정).
- 현재 **Codex가 perf(gtag/광고 지연로드) 작업 중** → 1순위(리전 설정)는 영역이 안 겹침. 충돌 없음.
- 외부 호출이 미국 리전을 전제한 곳이 있는지 점검(대부분 영향 없음 — Supabase·R2·외부 API는 리전 무관 호출).

## 7. 결론 · 추천
**1순위(함수 → 시드니, `vercel.json` 한 줄)**가 압도적 가성비. 위험 낮고 효과 큼.
→ 진행 시 순서: ① Vercel 플랜 Pro 확인 → ② `vercel.json regions: ["syd1"]` 추가 → ③ 재배포 → ④ `/api/best` 미스 TTFB 재측정으로 효과 확인 → ⑤ 남는 병목은 3순위로.
장기적으로는 **2순위(서울 이전)**가 한국 사용자에게 최적이나, 마이그레이션 규모상 1순위 효과를 본 뒤 결정한다.
