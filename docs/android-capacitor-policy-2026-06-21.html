2. TWA 레거시 vs Capacitor 현재 구조

2.1 TWA / Bubblewrap 기존 구조 (레거시)

  • Trusted Web Activity(TWA) = Chrome 커스텀 탭 기반으로 PWA를 전체화면 표시. Bubblewrap CLI로 생성.
  • 웹과 같은 Chrome 쿠키 저장소를 공유 → 로그인/세션이 브라우저와 연속.
  • 네이티브 플러그인 계층이 없음 → Firebase native SDK, 딥링크 브릿지 등 활용 제약.
  • assetlinks.json 기반 도메인 검증(Digital Asset Links).

2.2 Capacitor Android 현재 구조

  • Capacitor Android 앱이 WebView로 server.url(production 웹)을 로드.
  • 네이티브 플러그인 계층 확보 → @capacitor-firebase/analytics, @capacitor/browser, @capacitor/app 등 사용. package.json
  • WebView 쿠키 저장소는 시스템 브라우저와 격리 → 카카오 OAuth는 시스템 브라우저 + 1회성 handoff token 구조 필요.

2.3 기존 twa-policy.html에서 더 이상 맞지 않는 항목

레거시 전제현재
Chrome 쿠키 공유로 로그인 연속WebView 쿠키 격리 → handoff token 로그인
assetlinks.json 도메인 검증이 핵심Capacitor는 server.url 직접 로드(별도 검증 체계)

2.4 계속 참고 가능한 항목

  • Play Console 운영(트랙, 단계적 출시, vitals) 일반 원칙.
  • Kakao OAuth redirect URI 관리 원칙(아래 §8에서 갱신·확장).

공식 Capacitor 공식 문서는 server.url을 live-reload(개발)용으로 규정하며 production 사용을 권장하지 않는다. production 빌드에 server.url을 남기면 앱 스토어 리젝 위험이 있다고 명시한다.

우나어 선택 우나어 웹은 Next.js SSR(서버 의존: API 라우트·미들웨어·서버액션·next/image) 구조라 정적 번들(next export) 불가 → Capacitor 권장 방식(webDir 정적 번들)을 쓸 수 없다. 그래서 server.url로 production 웹을 로드하는 것을 의도적으로 선택했다. 이는 공식 권장과 다르며 리스크 관리 대상이다(특히 iOS App Store 4.2/4.7 리젝 가능성은 별도 트랙). Android는 현재 production 게시·운영 중.

5. Firebase / GA4 / Google Ads 정책

5.1 앱 이벤트 수집 원칙 코드 근거

  • 앱(Capacitor native)에서는 Firebase native Analytics만 사용 → GA4 Android app stream(481670969)로 전송.
  • 앱에서는 gtag / web stream 호출 금지. GtagLoader가 isAppNative면 gtag 스크립트 미로드, gtm.ts의 sendEvent가 앱이면 즉시 return(전 헬퍼 단일 차단점).

5.3 GA4 vs Google Ads 해석

  • GA4 = 전체 사용자 행동 확인용. 앱 이벤트가 실제 찍히는지는 GA4 실시간/DebugView로 본다.
  • Google Ads = 광고 유입 attribution 기준. 광고 클릭/설치로 들어온 사용자의 전환만 집계.
  • 친구가 직접 Play Store 검색/링크로 설치하거나 창업자 직접 테스트 가입은 GA4/Play엔 보여도 Ads 전환엔 0일 수 있다(정상).
  • Ads 전환 확인은 광고 유입 후 보통 수 시간~24~48시간 지연을 감안한다.
실측 확인됨 (2026-06-21) GA4 실시간에서 앱 가입 시 sign_up이 집계됨을 확인 → native 이벤트 경로 작동. (DebugView "디버그 기기 0"은 setprop이 재설치로 꺼진 것이지 이벤트 미전송이 아니다.)

6. Google Ads AC(App Campaign) 운영 정책

6.1 광고 집행 전 확인

  • GA4 실시간 또는 DebugView로 앱 sign_up/first_open 이벤트가 찍히는지만 확인.
  • Google Ads에 Android login / Android sign_up 전환이 import 되어 있는지 확인.
  • 온보딩·홈 첫 전환 체감 속도 점검(§10).

6.2 소액 AC 집행 후 확인

  1. AC 광고를 소액으로 켠다.
  2. 광고 클릭/설치/가입이 실제 발생하도록 기다린다.
  3. Google Ads > 목표 > 가입 > Android sign_up 전환 수 확인.
  4. 보고 지연으로 당일 0일 수 있다 → 24~48시간 후 재확인.

6.3 Ads가 0일 때 — 고장으로 단정하지 않는 기준

관측해석

GA4엔 sign_up 있음 + Ads 0 + 광고 집행 후 48h 경과조사 대상 read-only audit(아래)

7. AAB / Play Console 배포 정책

7.1 버전 규칙 공식

  • versionCode는 매 AAB 업로드마다 증가(정수, 단조 증가). 한 번 쓴 값 재사용 불가 → Play가 거부.
  • versionName은 사용자 표시용 문자열(예: 1.0.14). 운영 기준: native/기능 변경 시 patch/minor 증가, 명확히 추적.
  • 현재: versionCode 20 / versionName 1.0.15 (다음 native 빌드는 21부터).

7.2 Play App Signing & 업로드 키 공식 코드/운영

  • Play App Signing 사용 중 — Google이 앱 서명키를 보관·관리. 개발자는 업로드 키로만 AAB를 서명하고, Google이 최종 재서명한다.
  • 업로드 키 keystore: /Users/yanadoo/migration-backup/android/android.keystore, alias android (인증서 CN=Yongseok Kim).
  • 코드 근거 build.gradle에 signingConfig가 없음 → bundleRelease는 unsigned AAB를 만든다. 업로드 전 jarsigner로 업로드 키 서명 필요(과거 "모든 번들에 서명해야 합니다" 사고 원인).
# vc20 서명 예시 (창업자가 keystore 비번 직접 입력)
cd android/app/build/outputs/bundle/release
jarsigner -keystore /Users/yanadoo/migration-backup/android/android.keystore \
  -sigalg SHA256withRSA -digestalg SHA-256 app-release.aab android
jarsigner -verify app-release.aab   # "jar verified." 확인
구버전 산출물 업로드 금지 app-release-signed.aab(과거 vc17, Firebase Analytics 없음)는 업로드 금지. 업로드 대상은 항상 최신 빌드 app-release.aab(서명 후). 두 파일이 폴더에 공존하니 빌드 시각·버전 확인.

9. 광고 / 수익화 정책

앱(WebView)에서 AdSense 게재 금지 — 공식 정책

공식 Google 정책상 AdSense는 소프트웨어 애플리케이션(앱)에 통합할 수 없다(AdMob에는 적용 안 됨). 앱 내 광고는 AdMob 또는 WebView API for Ads + Google Mobile Ads SDK로만 정책 준수가 가능하다. WebView에 AdSense를 직접 띄우는 것은 정책 위반·계정 리스크.

9.1 현재 코드상 앱 AdSense 차단 코드 근거

  • AdSenseScriptLoader.tsx: window.Capacitor가 있으면 return → 앱에서 AdSense 스크립트 미로드.
  • AdSenseUnit.tsx: isCapacitor면 push/observer 미실행 + mounted && isCapacitor면 aside(광고 영역·"광고" 라벨·예약 공간·Coupang fallback)까지 전부 null 렌더.
  • → 앱에서 adsbygoogle 마커가 나오지 않음(계정 보호). 웹/TWA는 기존대로 정상 노출(회귀 0).

9.2 정책

  • 앱 광고와 웹 광고를 분리 판단. 앱은 AdSense OFF, 웹은 AdSense/Coupang 유지.
  • AdMob은 후순위 — 도입 시 Google Mobile Ads SDK + 정책 검토 + Play 광고 ID 선언 갱신 등 별도 설계.
  • 쿠팡 CPS 배너의 앱 노출 여부는 제휴 약관·UX 관점 별도 검토(2-1 범위 밖, 정책 트랙).
  • 계정 보호 관점 금지: 앱 WebView에서 AdSense 다시 켜지 않는다.

10. 성능 운영 정책

10.1 적용 이력 코드 근거

  • 온보딩 닉네임 입력 개선: 입력 1회당 중복 setState 제거(idle→error 중간 전환 제거, setStatusIfChanged/setErrorIfChanged 가드, 규칙 계산 useMemo). 커밋 47672e6 → main 1ee6849.
  • 홈 첫 렌더 비필수 작업 defer: PopupRenderer /api/popups fetch를 idle/1200ms defer, NotificationBadge 최초 unread-count를 idle/1000ms defer, PushPermissionToast의 signup 토스트를 1800ms로 지연(나머지 500ms 유지). 커밋 cb1d89a → main db4288b.
  • 온보딩 전환 stutter 감소: native sign_up/onboarding_complete를 await Promise.all([...])로 병렬 유지하며 전환 매끄럽게(ea89fd2).

10.2 원칙

  • 광고 집행 전 첫 경험(온보딩·첫 홈) 속도 우선. 우리 또래 사용자 기준 즉시 반응감·입력 안정성.
  • 가입 전환 이벤트 유실 방지 await는 유지. 라우팅 전 보장하되 가능한 작업은 병렬화/전환 후로 defer.
  • src/web 성능 패치는 AAB 재빌드 불필요(Vercel 배포로 반영).
  • 홈 서버 쿼리 구조 개선은 별도 read-only 진단 후 진행(홈 page는 타 세션 작업 영역일 수 있음 — 수정 전 git status 확인).

14.1 read-only 성능 진단

14.2 AAB 빌드 전 점검

read-only로 vc 빌드 전 점검해라. 코드 수정 금지.

google-services.json app_id 주입, src 변경이 AAB 필요한지(native/plugin/config 변경 여부),
서명 필요(unsigned AAB + jarsigner 업로드 키)인지.
보고: AAB 재빌드 필요 여부 + versionCode 다음 값 + 서명 절차.

14.3 GA4 / Ads 전환 read-only audit

read-only로 Android Firebase Analytics first_open/sign_up 경로와 Google Ads import 상태를 재검증해라. 코드 수정 금지.
구분 보고: 1) GA4 app stream에 Android sign_up/first_open 있는가 2) Ads에 import 됐는가
3) Ads 0이 attribution 부재인지·GA4 부재인지·import 설정 문제인지 4) 친구/직접 설치가 Ads 전환에 안 잡히는 이유.

14.4 Kakao OAuth 변경 전 점검

read-only로 Kakao OAuth 변경 영향을 점검해라. 코드 수정 금지.
확인: auth.config.ts/auth.ts, redirect URI(production/www/localhost 영구 유지), KOE006/KOE010 이력,
Client Secret rotation 영향. 영구 redirect URI 삭제 금지. 변경 시 /api/health/auth 200 + 실기기 검증 계획 보고.

14.5 main cherry-pick 배포 (poc 전체 merge 금지)

<커밋해시> 1커밋만 main에 반영해라. worktree로 dirty 보존. poc 전체 merge/PR 금지. force push 금지.
절차: git fetch → git worktree add --detach ../tmp origin/main → git cherry-pick <해시>
→ (충돌 시 멈추고 보고) → 변경 파일이 의도한 것뿐인지 확인 → typecheck/lint/build
→ PASS면 git push origin HEAD:main (non-FF면 멈추고 보고) → worktree 정리 → Vercel 배포 확인.
보고: 커밋 해시 / 변경 파일 / 검증 결과 / push 성공 / Vercel 확인 필요 / AAB 재빌드 필요 여부.
