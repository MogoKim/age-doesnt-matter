2. TWA 레거시 vs Capacitor 현재

2.1 TWA / Bubblewrap (레거시)

  • TWA = Chrome 커스텀 탭 기반으로 PWA를 전체화면 표시. Bubblewrap CLI로 생성. Chrome 쿠키 저장소 공유 → 로그인 연속.
  • 네이티브 플러그인 계층 없음 → Firebase native SDK·딥링크 브릿징 제약. assetlinks.json 도메인 검증.

2.2 Capacitor Android 현재

  • Capacitor Android 앱이 WebView로 server.url(production 웹)을 로드. 화면/로직 대부분은 웹(Next.js SSR).
  • 네이티브 플러그인(현재 6개 + 커스텀 1개) package.json · capacitor.build.gradle: @capacitor-community/admob, @capacitor-firebase/analytics, @capacitor-firebase/app, @capacitor-firebase/messaging, @capacitor/app, @capacitor/browser + 앱-로컬 커스텀 AdMobNative(Native Advanced 오버레이).
  • WebView 쿠키는 시스템 브라우저와 격리 → Kakao OAuth는 시스템 브라우저 + handoff token 구조(Â§8).
본 문서가 우선하는 기준 Android 앱 관련 판단에서 본 문서가 twa-policy.html·구 vc20본보다 우선한다. 충돌 시 본 문서를 따른다.

5. Firebase / GA4 / Ads / FCM 정책

5.1 앱 이벤트 수집 코드 근거

  • 앱에서는 Firebase native Analytics만 사용 → GA4 Android app stream(481670969)로 전송. 앱에서 gtag/web stream 호출은 금지(GtagLoader·gtm.ts가 isAppNative면 차단).
  • 앱 native 이벤트: login_start·login·sign_up·onboarding_complete. first_open/session_start/screen_view는 native SDK 자동.

5.2 sign_up 정의

sign_up = 온보딩 마지막 "우나어 시작하기"까지 완료한 시각 OnboardingForm.handleComplete()에서만 발화. 카카오 로그인만으로는 login만 발화, sign_up은 아님. 남성 차단(정책)은 sign_up이 아니다. 라우팅 전 await 보장 유지(이벤트 유실 방지).

5.3 FCM 푸시 vc19~ 신규 코드 근거

  • 토큰 수집: 앱에서 AppFcmRegister가 @capacitor-firebase/messaging으로 토큰 획득 → /api/push/fcm-token에 저장(Prisma FcmToken).
  • 서버 발송: lib/push/fcm-sender.ts가 firebase-admin으로 발송. 어드민 발송/스케줄 발송 라우트 존재.
  • 권한: Android 13+ POST_NOTIFICATIONS 런타임 요청(Manifest 선언). 전 구간(수집→발송→기기 알림) 검증 PASS 이력.
  • 후속 포그라운드 표시·탭 딥링크 등 수신 핸들러 보강 항목은 별도 트랙.

5.4 GA4 vs Google Ads 해석

  • GA4 = 전체 사용자 행동. Google Ads = 광고 전환 attribution(광고 클릭/설치로 온 사용자만).
  • 지인/직접 설치는 Ads 전환 0일 수 있음(정상). Ads 전환은 광고 집행 후 수 시간~24~48h 지연 가정.

6. Google Ads AC(App Campaign) 이슈 정책

  • 광고 집행 전: GA4 실시간/DebugView로 앱 sign_up/first_open 발화 확인 + Ads에 Android login/sign_up 전환 import 여부 확인.
  • Ads 0일 때 고장으로 단정 금지: GA4 있음+Ads 0+미집행=정상. GA4 있음+Ads 0+집행 48h 경과=조사 대상(read-only audit).
  • read-only audit 기준: ① GA4 app stream에 Android sign_up/first_open 있나 ② Ads import 되나 ③ Ads 0이 attribution 부재/GA4 부재/import 문제인지 ④ 지인·직접 설치 분리.

7. AAB / Play Console 배포 정책

7.1 버전 규칙 공식

  • versionCode는 매 AAB 업로드마다 단조 증가. 이미 업로드된 값 재사용 불가(Play 거부).
  • 현재 versionCode 23 / versionName 1.0.18. 다음 native 빌드는 24부터.
  • 버전 이력: vc20(1.0.15) → vc21(1.0.16, 하단 배너 도입) → vc22(1.0.17, Native Advanced 도입/내부테스트) → vc23(1.0.18, Native 확장·배너 제거·production).

7.2 Play App Signing & 업로드 키 공식 코드

  • Play App Signing 사용 → Google이 앱 서명키 보관. 개발자는 업로드 키로만 AAB 서명, Google이 재서명.
  • build.gradle에 signingConfig 없음 → bundleRelease는 unsigned AAB. 업로드 전 jarsigner로 업로드 키 서명 필수.
# AAB 서명 (창업자가 keystore 비번 직접 입력, 채팅에 쓰지 않음)
jarsigner -keystore ~/migration-backup/android/android.keystore \
  app-release.aab <alias>
jarsigner -verify app-release.aab   # "jar verified." 확인
# (자체 서명 업로드 키라 cert-chain 경고는 정상)
구버전 산출물 업로드 금지 항상 최신 빌드 app-release.aab(서명 후). 파일이 여러 개면 빌드 시각·버전 확인.

7.3 빌드 환경 코드

  • CLI 빌드 시: JDK 21(Capacitor 8 요구) + Android SDK. ./gradlew :app:bundleRelease. compileSdk/targetSdk 36, minSdk 24.
  • native 의존성: admob 플러그인의 play-services-ads는 implementation(비노출)이라 Native 광고 클래스용으로 app 모듈에 play-services-ads:24.9.+ 명시.

7.4 Play Console 광고 ID 선언

앱이 AdMob Native Advanced 광고를 실제 게재하고 Firebase가 광고 ID를 사용하므로, Play Console 데이터 보안/AD_ID 권한 선언을 광고 사용 기준으로 유지(Manifest AD_ID 권한과 정합).

8. Kakao OAuth 이슈 정책

8.1 앱 로그인 구조 (handoff token) 코드 근거

  • WebView 쿠키 격리 → 앱은 시스템 브라우저 OAuth 후 com.agenotmatter.app://auth?token=...로 복귀.
  • AppDeepLinkHandler.tsx가 token을 받아 NextAuth Credentials app-handoff provider로 WebView 세션 발급. handoff는 HMAC+nonce 1회 소비 검증(app-handoff.ts). 발급 시작 /api/app-login/start.
  • 남성 차단/상태 검증은 시스템 브라우저 OAuth(kakao) 단계에서 수행, handoff는 세션 발급 레이어.

8.2 요구 유지 redirect URI 삭제 금지

http://localhost:3000/api/auth/callback/kakao
https://age-doesnt-matter.com/api/auth/callback/kakao
https://www.age-doesnt-matter.com/api/auth/callback/kakao

  • 알려진 코드: KOE006(preview redirect URI 콘솔 미등록) · KOE010(ID/Secret 키 박스 불일치).
Client Secret rotation은 별도 계획 재발급 시 인증 영향 → Vercel prod/preview env + 로컬 .env.local 교체 후 /api/health/auth 200 · 실기기 로그인 · AUTH_FAILURE 모니터링.

9. 광고 정책 (AdMob Native + AdSense) vc21~23 대폭 변경

핵심 원칙

공식 AdSense는 앱(WebView)에 게재 불가(계정 리스크). 앱 광고는 AdMob(Google Mobile Ads SDK)으로만. 앱 = AdMob / 웹·TWA = AdSense 분리 유지.

9.1 앱: AdMob Native Advanced 코드 근거

  • 구현: 커스텀 Capacitor 플러그인 AdMobNativePlugin.java가 AdLoader로 NativeAd 로드 → NativeAdView(native_ad_view.xml)를 WebView 위 오버레이로 표시. 웹의 placeholder(NativeAdSlot.tsx)가 좌표를 측정해 setRect로 전달.
  • 표시 보장: NativeAdSlot이 IntersectionObserver + scroll rAF + 정착 timeout으로 뷰포트 진입 시 반드시 표시. no-fill이면 해당 슬롯만 접힘. 라우트 이탈 시 destroy. 화면당 개수 제한 없음(웹 배치와 동일하게 다중 노출).
  • 광고 단위: Native Advanced ADMOB.NATIVE_INFEED = ca-app-pub-4117999106913048/8732877425. App ID ~8809650070(strings.xml). App ID meta는 Manifest에 선언.
  • 노출 위치 10곳: home-feed-1·home-feed-2·list-feed(커뮤니티/베스트/매거진/일자리 공통 PostListWithAds 4번째 뒤)·search-feed·{community,magazine,jobs}-detail-{inarticle,bottom}. 상세 광고는 관련글/댓글을 가리지 않는 위치.
  • 웹/TWA: 위 슬롯은 NativeAdSlot의 fallback으로 기존 AdSense가 그대로 렌더(앱만 Native).

9.2 하단 anchored 배너 — 도입 후 폐지 코드 근거

  • vc21에서 @capacitor-community/admob 하단 anchored 배너 도입 → vc23 정책 전환으로 폐지.
  • AdMobBanner.tsx는 이제 teardown 전용: 앱은 라이브 웹을 로드하므로 새 웹 배포로 showBanner를 없애도 기존 앱의 네이티브 배너가 잔존 → mount 시 removeBanner() 1회 호출로 콜드 재시작 없이 제거.
  • 금지 하단 anchored 배너 재도입.

9.3 웹 AdSense 차단 기준 수정됨 코드 근거

  • AdSenseScriptLoader.tsx: 차단 기준 = cap?.isNativePlatform?.() === true. (과거 window.Capacitor 존재만으로 return하던 것은 버그 — 일부 웹 환경에 window.Capacitor가 주입돼 있고 isNativePlatform=false인데도 AdSense가 안 떴음. 수정·프로덕션 검증 완료.)
  • 앱 AdSense는 여전히 OFF: AdSenseUnit.tsx가 isCapacitor면 렌더 null(계정 보호). 쿠팡 CPS는 별도 유지.

9.4 범위 밖 / 미사용

  • interstitial / app opening / reward: 미도입(금지). Native Advanced만 운영.
  • ADSENSE.LIST_PAGINATION_BOTTOM = 'REPLACE_WITH_SLOT_ID': 슬롯 미발급 → 웹/앱 모두 미노출. 발급 시 별도 적용.

10. 성능 이슈 정책

  • 적용됨(main 반영): 온보딩 닉네임 입력 리렌더 감소(1ee6849), 홈 첫 렌더 비필수 요청 defer(db4288b), sign_up/onboarding_complete를 await Promise.all([...])로 병렬 유지.
  • 앱 광고 성능: NativeAdSlot은 scroll rAF 스로틀 + IntersectionObserver로 위치 갱신. 다중 슬롯 동시 노출 시 저사양 기기 스크롤 부드러움 모니터링 권장.
  • 원칙: 광고 집행 전 첫 경험(온보딩·첫 홈) 속도 우선. sign_up/onboarding_complete의 라우팅 전 await 보장은 유지.
  • src/web 성능 패치는 AAB 재빌드 불필요(Vercel 배포로 반영).

14. Claude Code 작업 지시 템플릿 (복붙용)

14.1 문서만 안전 배포(병렬 개발 오염 0)

docs/ 문서 1개만 최신화해 배포해라. 코드 파일 수정 금지.
절차: git fetch origin main → 격리 worktree/브랜치 off origin/main
→ 문서 파일 1개만 작성 → git add docs/파일.html (파일명 명시, git add . 금지)
→ git diff --cached --name-only 로 1개만인지 검증 → commit → push → PR(base main)
→ checks에서 변경파일감지=docs-only 확인 → (지시 시) merge.
보고: staged 목록(1개) / PR URL / checks docs-only 여부.

14.2 앱 광고 온디바이스 QA(read-only)

read-only로 Android 앱 AdMob Native 광고를 실기기 QA해라. 코드/AAB/광고 클릭 금지.
확인: 홈 feed-1/2, list-feed, search-feed, 상세 in-article/bottom 노출,
하단 배너 없음, 관련글/댓글 미가림, 스크롤 밀림/잔상/크래시 없음, 웹은 AdSense.
release 빌드는 logcat plugin 로그 없음 → 스크린샷 시각 검증. 설치 출처가 Play인지 확인.

14.3 AAB 빌드 전 점검(read-only)

read-only로 vc 빌드 전 점검해라. 코드 수정 금지.
확인: build.gradle versionCode(다음 24)/versionName, play-services-ads 의존성,
capacitor.config server.url(production), google-services app_id 주소,
src 변경이 AAB 필요한지(native/plugin/res/config 여부), 서명(unsigned + jarsigner 업로드 키).
보고: AAB 재빌드 필요 여부 + versionCode 다음 값 + 서명 절차.
