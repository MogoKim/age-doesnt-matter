1

채널이란 — 누가 어디로 들어오나

같은 웹사이트지만 "어떻게 실행됐는지"에 따라 채널이 갈립니다.

🌐

웹 (브라우저)

안드 Chrome · 아이폰 Safari · 데스크탑. 그냥 주소로 접속하는 보통의 웹.

🍏

PWA

아이폰에서 "홈 화면에 추가"한 형태(standalone). 주소창 없이 앱처럼. 사실상 아이폰용.

🤖

TWA

안드 플레이스토어 앱(com.agenotmatter.app). 우리 웹을 안드 앱 껍데기로 감싼 것. 안드 전용.

💬

인앱브라우저

카톡·네이버·인스타·구글앱 안에서 열리는 브라우저. 설치·일부 기능 제약 → "외부 브라우저로 열기"를 유도.

🤖 안드로이드

웹 또는 TWA(앱)

🍏 아이폰

웹 또는 PWA(홈 화면)

💻 데스크탑

웹만
왜 이렇게 나뉘나? 안드에는 플레이스토어가 있어 "앱(TWA)"으로 설치할 수 있지만, 아이폰엔 우리 앱이 없어 "홈 화면 추가(PWA)"가 유일한 앱화 방법입니다. 그래서 설치·유도 정책이 채널마다 달라집니다.
개발자용 — PWA/TWA 인프라 (manifest · assetlinks · service worker)
  • public/manifest.json: display=standalone, prefer_related_applications=true → 안드 Chrome은 PWA 대신 플레이스토어 앱을 권함. related_applications = play / com.agenotmatter.app
  • public/.well-known/assetlinks.json: TWA 도메인 소유 검증(sha256 지문) — 이게 있어야 안드 앱이 주소창 없이 우리 도메인을 띄움
  • public/sw.js: 오프라인 캐시 CACHE_NAME='unao-v3' + 웹푸시 수신(push) / 클릭(notificationclick) / 구독갱신(pushsubscriptionchange)

★ 공통 vs 차이 — 채널별로 뭐가 같고 뭐가 다른가

대부분은 모든 채널이 똑같습니다(같은 웹 코드). 다른 건 딱 3가지: 가입 유도 방식 · 앱 설치 유도 · 일부 측정.

항목공통(모든 채널 동일)채널별 차이
회원 인증 카카오 1초 로그인 — 모든 채널 동일 인앱브라우저만 카카오 로그인이 막혀 "외부 브라우저로 열기" 우회
콘텐츠·글쓰기 글/매거진/일자리/댓글/글쓰기 — 전 채널 100% 동일 차이 없음

앱/설치 유도 — (채널마다 방식이 다름) 안드 = 플레이스토어(TWA) / 아이폰 = 홈 화면 추가(PWA) / 데스크탑 = 없음  (현재 전체 OFF)
푸시 알림 가입·글쓰기·댓글·방문 시 "알림 받을래요?" 권유. "받을게요"=푸시 허용+마케팅 수신 동의 아이폰은 PWA로 설치했을 때만 웹푸시 가능(브라우저 정책). 인앱·웹아이폰(미설치)은 미노출
네비게이션 상단 아이콘 메뉴 + 플로팅 "✏️글쓰기" — 전 채널 동일 차이 없음
측정(분석) 모든 이벤트에 채널값(browser_env) 기록 채널값이 달라짐(twa-android / ios-safari / android-chrome …). TWA는 sticky로 정확도 보강
왜 TWA만 게이트? 앱(TWA)을 일부러 설치한 사람은 관심·동기가 높아, 가입을 먼저 권하는 편이 이탈보다 득이 클 수 있습니다. 그래서 "어떤 방식이 가입 후 재방문을 높이나"를 실험 중입니다(6번). 반대로 웹은 가볍게 들렀다 갈 수 있어 정독을 방해하지 않는 하단 배너를 씁니다.

채널을 어떻게 알아내나 (감지)

화면 크기 + 브라우저 종류(UA) + 어디서 왔는지(referrer) + 과거 기억(sticky)을 종합해 판별합니다.

중요: 같은 일을 하는 감지 함수가 3종이고 역할이 다릅니다(혼동 주의).

🔑 TWA sticky(_twa_confirmed): "한 번 앱이면 계속 앱으로 기억". 카카오 로그인을 다녀오며 referrer(어디서 왔는지)가 사라져도 TWA로 유지됩니다. 측정 정확도 보강(C1 수정으로 getBrowserEnv에도 적용).

기능별 정책 상세

5-1. 회원가입 유도 🟢 ON · 항상

비회원에게 가입을 권유하는 3가지 자리. 모두 로그인 안 한 사람에게만 보입니다.

  • ① 가입 배너(콘텐츠 페이지 하단): 글/매거진/일자리/베스트 페이지에서, 글을 거의 다 읽으면(85% 정독 또는 60초 경과) 화면 맨 아래에 뜸. 문구는 "나만 이런 게 아니었네?"(공감형, 단일 고정). 한 세션에 1번, 최대 4번까지만.
  • ② 홈 가입카드: 홈 중간의 "우리 또래 이야기, 여기 다 있어요" 카드.
  • ③ 글 하단 CTA: 게시글 맨 아래 "같이 나눠보세요 / 1초 만에 가입하기".
  • 인앱브라우저(카톡·네이버 등): 카카오 로그인이 막히므로 버튼이 "외부 브라우저로 열기"로 바뀌어 Chrome/Safari로 안내.
  • 자동 트리거: 광고 등 외부 유입(utm)으로 들어오면 5초 카운트다운 후 배너를 1번 자동으로 띄움.
⚠️ 과거의 "A/B/C 문구 실험 · 20초 · 50% 스크롤"은 끝났습니다(폐기). 사용자 테스트로 공감형(C) 문구가 승자로 확정돼, 지금은 단일 문구 + 정독 후(read_complete) 노출로 고정입니다. 배리언트 배정·측정 코드도 제거됐습니다.

5-2. TWA 진입 게이트 🟢 ON · 실험 중 · 끌 수 있음

안드 앱(TWA)에 처음 들어온 비회원에게 가입을 권하는 화면. 어떤 방식이 효과적인지 실험 중입니다.

  • A (없음): 아무것도 안 띄움 — 대조군
  • B (소프트): 글 3개를 본 뒤 하단 시트로 권유
  • C (하드): 첫 화면을 통째로 가입 화면으로 ("먼저 둘러볼게요" 탈출구 포함)
  • 측정: 어느 방식이 가입 후 재방문(D1/D7)을 높이나
  • 끄는 법: Vercel에서 NEXT_PUBLIC_TWA_GATE_ENABLED=false (미설정 = ON)
실제 운영 화면
C(하드) 게이트 — 안드 앱 첫 화면 전체. 작게 "먼저 둘러볼게요" 탈출구
개발자용 — 게이트 코드
  • TwaEntryGate.tsx — 최상단 if (process.env.NEXT_PUBLIC_TWA_GATE_ENABLED === 'false') return 가드. 노출 조건 isTWA + 비로그인. variant는 getExperimentVariant('twa01_entry_gate')
  • B variant: 본 글 수 누적 ≥ 3에서 시트. 배정/노출은 localStorage twa_gate_assigned
  • src/lib/experiments/registry.ts — 실험 SSOT, twa01_entry_gate(A/B/C)

5-3. 앱 설치 유도 🔴 현재 OFF

⚠️ 지금은 아무에게도 안 보입니다. 마스터 스위치 NEXT_PUBLIC_PWA_INSTALL_ENABLED가 미설정(OFF)이라, 설치 유도 코드 전체가 즉시 멈춥니다. 아래는 "켜면 이렇게 동작한다"는 설명입니다(그래서 실제 스크린샷 대신 목업).

켜지면, 이미 충분히 둘러본 사람에게 안드는 플레이스토어, 아이폰은 홈 화면 추가로 설치를 권합니다. TWA·PWA(이미 설치)·인앱(설치 불가)에는 안 띄웁니다.

5-4. 푸시 알림 — 구독 수집 · 마케팅 동의 · 광고 발송 🟢 ON

예전엔 "댓글 단 뒤"에만 물어봤고 그마저 버그로 안 떴습니다(구독자 0). 이제 4가지 순간에 물어보고, 받으면 답글 알림(리텐션)과 합법 광고를 보냅니다.

언제 뜨나 — 4개 트리거 (모두 로그인 회원·미구독에게만)

  • ① 가입 직후 — "🌱 환영해요! 알림 받으시겠어요?"
  • ② 글쓰기 직후(전환 최고) / 댓글 직후 — "🔔 답글 오면 바로 알려드릴까요?"
  • ③ 회원 방문(글 1개 정독 후 자동) — "🔔 내 글 소식, 알림 받으시겠어요?"
  • 서브문구에 "소식·혜택"을 명시 → "받을게요" 탭 = 푸시 허용 + 마케팅 수신 동의. 버튼 [받을게요]/[나중에].
  • 차단: 비회원 / 이미 허용·차단 / 쿨다운(거절 1일·2회+30일) / PWA·어드민 팝업 표시중 / 미지원 기기. 세션당 1회.
  • 일자리(job) 알림은 여전히 미연결(문구만 있고 부르는 곳 없음 — 부채 C6).
⚠️ 과거 버그 수정: 토스트는 (main) 레이아웃에 한 번 붙어 안 사라지는데, 예전엔 "트리거 확인"을 마운트 때 1번만 해서 같은 페이지/앱내 이동 시 글쓰기·댓글 토스트가 안 떴습니다(구독 0의 진짜 원인). 이제 이벤트(unao:push-trigger/unao:engaged)로 즉시 재평가해 정상 노출됩니다.

마케팅 수신 동의 (정보통신망법 §50)

  • "받을게요" = 마케팅 동의: 문구에 "우나어 소식·혜택"을 명시했으므로 동의 근거(informed consent). 구독 성공 시 marketingOptIn=true + Agreement(MARKETING,'1.0') 기록(온보딩과 version 단일). ⚖️ 동의 묶음은 법무 확인 완료.
  • 권한(기술) ≠ 동의(법) — 헷갈리기 쉬움:
메시지OS 푸시 권한(구독)마케팅 동의(법)
정보성/서비스(답글·등급)✅ 필요(기술)❌ 불필요
광고성(혜택·이벤트)✅ 필요✅ 필요(§50)
앱 안 종(🔔) 알림❌ 불필요❌ 불필요

즉 정보성(답글)은 마케팅 동의 없이 보내도 되지만, "폰 OS 푸시" 자체는 정보성이든 광고든 권한(구독)이 필수입니다(앱 안 종 알림만 예외).

발송 2종 (서비스 vs 광고)

종류받는 사람표기·제한
서비스 알림구독자 전원(댓글·답글·등급)없음 · 자동 발송
광고/마케팅구독 + marketingOptIn=true만(광고) 표기 + 야간 21~08 차단 + 어드민 수동 발송
⚖️ 새 글·좋아요·스크랩 자동 푸시는 없음(남발 방지). 광고는 어드민 /admin/push에서 "유형: 광고" 선택 시 위 규제 자동 적용. 실제 광고 발송 전 법무 최종확인 권장.

현재 ON/OFF + 스위치 (운영 핵심)

기능현재스위치(env)끄고 켜는 법
회원가입 유도🟢 ON(없음·항상)코드 변경만
TWA 게이트🟢 ONNEXT_PUBLIC_TWA_GATE_ENABLEDVercel서 =false → OFF (미설정=ON)
앱 설치 유도🔴 OFFNEXT_PUBLIC_PWA_INSTALL_ENABLEDVercel서 =true → ON (미설정=OFF)
푸시 토스트(구독수집 4트리거)🟢 ONFEATURE_PUSH_TOAST ⚠️ 무효코드 변경 필요(아래 경고 — 새 4트리거도 이 한계 상속)
웹푸시 발송(서비스)🟢 ONFEATURE_WEB_PUSH (서버)Vercel서 =false → OFF
광고 푸시 발송동의자만코드 게이트(자동)마케팅 미동의자엔 미발송 + (광고)표기 + 야간 21~08 차단(§50)
⚠️ 푸시 토스트는 지금 끌 수 없습니다(부채). FEATURE_PUSH_TOAST는 NEXT_PUBLIC_ 접두사가 없는데, 이 값을 쓰는 PushPermissionToast는 클라이언트 컴포넌트라 빌드 시 값이 안 박혀 항상 ON으로 읽힙니다(죽은 스위치). 끄려면 코드 수정 필요. 반면 웹푸시 발송(FEATURE_WEB_PUSH)은 서버에서 동작하므로 정상 OFF 가능.
🔎 위 "현재" 값은 로컬 .env.local 4개 키 모두 미설정 = 기본값 기준입니다(설치유도 OFF는 프로덕션 번들에서도 실측 확인). 운영 변경 시 Vercel 환경변수가 진실이니, ON/OFF 단정 전 Vercel을 확인하세요.
