우나어, 웹·PWA·TWA가 어떻게 돌아가나요?

사람이 들어오는 3가지 문(웹·PWA·TWA)마다 화면과 권유가 어떻게 달라지는지, 지금 실제로 뭐가 켜져 있고 어떤 실험이 돌고 있는지, 그리고 지금 알려진 문제를 한 장에 정리했습니다. 코드를 몰라도 읽히게 썼고, 기술 내용은 접어뒀습니다(▸ 눌러서 펼치기).

🚪 들어오는 문은 3개 (웹 · PWA · TWA)
🔀 권유 시스템은 2개 — 가입유도(켜짐) · 설치유도(꺼짐)
🧪 지금 A/B 실험 3개가 동시에 돌아가는 중
1. 들어오는 문 3개 2. 권유 2시스템(중요) 3. 안드·아이폰 여정 4. 누가 뭘 보나(표) 5. 실험 3개 6. 지금 알려진 문제 7개 7. 관련 문서·갱신규칙 8. 개발자용 기술
1

사람이 들어오는 "문"이 3개 있어요

같은 우나어라도, 어떤 문으로 들어왔는지에 따라 보이는 화면이 달라집니다.

문무엇누가
웹그냥 인터넷 브라우저(크롬·사파리)로 사이트를 봄안드폰 · 아이폰 · 컴퓨터 모두
PWA"홈 화면에 추가"로 바탕화면 아이콘을 깔아 앱처럼 씀주로 아이폰 사람
TWA플레이스토어에서 우리 앱을 깔아서 씀 (com.agenotmatter.app)안드로이드 사람만

🤖 안드로이드 사람

웹 → (앱 깔면) TWA

플레이스토어 앱이 있어서, 앱을 깔면 TWA로 들어옵니다.

🍏 아이폰 사람

웹 → (홈에 추가하면) PWA

아이폰엔 플레이스토어가 없어서, 홈 화면 추가(PWA)가 "앱처럼 쓰기"의 전부예요.

🧩 쉽게: 안드 = 웹 또는 TWA / 아이폰 = 웹 또는 PWA. (PWA는 사실상 아이폰용, TWA는 안드용)
2

가장 중요 — 권유가 "2개의 다른 시스템"이에요

이게 그동안 헷갈렸던 핵심입니다. 우리가 사람에게 하는 권유는 두 종류이고, 각각 따로 켜고 끕니다.

① 가입 권유 시스템

🟢 지금 켜짐 (작동 중)

"회원가입 하세요"(카카오). 끄는 스위치가 따로 없고 항상 돕니다.
— 가입 배너, 홈 중반 카드, 글 하단 가입버튼, 안드 앱 진입 게이트

② 앱 설치 권유 시스템

🔴 지금 꺼짐 (안 나옴)

"앱 깔아보세요". 스위치(PWA_INSTALL_ENABLED)가 꺼져 있어 아무에게도 안 뜹니다.
— 설치 팝업, 하단 배너, 푸터 버튼, 글 하단 설치버튼

💡 그래서 "가입은 권하는데 앱 설치는 안 권하는" 상태예요. 설치 유도의 자세한 동작은 app-install-policy.html 문서에 있지만, 그 문서는 "스위치 ON"을 가정해 쓰였으니 지금은 전부 안 뜬다는 점을 같이 기억하세요.
✅ 안드 앱(TWA)으로 들어온 사람에겐 "가입 진입 게이트"라는 강한 가입 권유가 따로 돕니다(아래 5번 실험). 이건 설치 스위치와 무관하게 켜져 있어요.
3

안드·아이폰 사람이 겪는 길

🤖 안드로이드

웹으로 들어옴 → 콘텐츠 보면 가입 배너 → 카카오 가입.
앱(TWA)으로 들어오면 → 진입 게이트(아래 실험)로 가입 권유 → 카카오 가입.
(가입 후 "앱 설치 권유 → 플레이스토어"는 스위치 꺼져서 지금 안 뜸)

🍏 아이폰

웹(사파리)으로 들어옴 → 콘텐츠 보면 가입 배너 → 카카오 가입.
PWA(홈에 추가)로 들어와도 → 가입 안 했으면 동일하게 가입 배너가 뜸.
(가입 후 "홈 화면 추가 권유"는 스위치 꺼져서 지금 안 뜸)

⚠️ 아이폰 PWA 사람에게도 가입 배너가 뜨는 건 의도된 설계입니다 — PWA는 "앱 진입 게이트" 대상이 아니라 "웹 배너" 대상으로 분류돼 있어요.

5

지금 돌고 있는 A/B 실험 3개

"실험"은 같은 상황에서 A안·B안을 사람마다 다르게 보여주고 뭐가 더 가입을 잘 시키는지 비교하는 것. 결과는 어드민 /admin/ab-tests에서 봅니다.

실험무엇을 비교누구에게
가입 배너 문구혜택형 vs 재미형 vs 공감형 — 어떤 문구가 가입을 잘 시키나웹·PWA 비회원
가입 배너 타이밍글 읽는 중(20초/50%) vs 거의 다 읽은 후(85%) — 언제 띄울까웹·PWA 비회원
앱 진입 게이트없음(A) vs 살짝(B,글 3개 후) vs 강하게(C,첫 화면) — 막아서면 진성 가입↑?안드 앱(TWA) 비회원
📌 실험의 "정답 정의"와 세부 문구는 코드 한 곳(src/lib/experiments/registry.ts)에 모여 있어요. 이 문서는 거기를 가리키기만 합니다(중복 금지).
6

지금 알려진 문제 7개 (코드 한 줄씩 검수 결과)

🚦 신호등으로 급한 정도를 표시했습니다. 위 2개가 급하고, 나머지는 "당장 탈은 없지만 정리하면 좋은 것".

🔴 1번 (제일 급함) 앱 유저 숫자가 실제보다 낮게 보일 수 있어요

안드 앱(TWA) 사람이 카카오 로그인을 하면, 그 사이에 "이 사람 앱 유저"라는 꼬리표가 떨어져 "그냥 웹 사람"으로 잘못 기록될 수 있어요. → 어드민의 "앱 유저 재방문율(D1/D7)" 숫자가 진짜보다 낮게 나옵니다.

개발자용 — 원인·근거

TWA 판정이 4곳에서 다름. 게이트는 useAppEnvironment.isTWA(referrer + _twa_confirmed sticky)로 OAuth 복귀해도 유지되지만, 집계는 이벤트의 browser_env=getBrowserEnv()(referrer만)에 의존 → referrer 소실 시 android-chrome으로 기록. 집계(admin.experiments-web.ts isTwa)는 browser_env==='twa-android'로만 TWA 판정.

제안: sign_up/page_view의 browser_env를 sticky 포함 판정으로 보강하거나, 집계 TWA 판정에 referrer/sticky 보조신호 추가.

🟡 2번 안드 앱 "진입 게이트"를 끌 스위치가 없어요

진입 게이트가 사람을 막아서는데, 급히 끄고 싶어도 끄는 버튼이 없습니다. 끄려면 개발자가 코드를 직접 고쳐야 해요. (끄는 스위치 비슷한 게 만들다 만 채로 방치)

개발자용 — 원인·근거

flags.twa(src/lib/feature-flags.ts)가 정의됐으나 사용처 0(dead). TwaEntryGate.tsx는 환경변수 가드 없이 항상 작동. 제안: if (!flags.twa) return 연결.

🟡 3번 어드민 가입률 % 는 "정확한 값"이 아니라 "대략값"이에요

실험에서 "보여준 수"는 방문 단위, "가입한 수"는 사람 단위로 세요. 단위가 섞여서, 그 % 를 정확한 수치로 믿으면 안 되고 A안 vs B안 비교용으로만 봐야 해요.

개발자용 — 원인·근거

admin.experiments-web.ts: 노출=sessionId 분모, 전환=userId 분자(인앱→외부 sessionId 단절 우회 목적의 의도적 설계). 게이트는 이미 funnel서 제외하고 재방문 지표로 분리 처리됨(올바름). 제안: 어드민/문서에 "근사치" 라벨 명시.

🟡 4번 같은 규칙이 두 군데에 복사돼 있어요

"사람마다 A안/B안 어느 걸 보여줄지 정하는 규칙"이 한 곳에 모여야 하는데, 가입 배너만 자기 복사본을 따로 씁니다. 지금은 똑같아서 괜찮지만, 나중에 한쪽만 고치면 어긋날 수 있어요.

개발자용 — 원인·근거

SignupPromptBanner.tsx의 getOrAssignVariant/getTriggerVariant가 SSOT(src/lib/experiments/assign.ts) 대신 자체 해시 사용. 값은 동등(signup_variant 키 공유). 제안: assign.ts로 통일.

⚪ 5번 안 쓰는 설정값 + 기본값이 서로 반대

켜고 끄는 설정값 몇 개가 만들어만 두고 안 쓰여요. 게다가 같은 설정이 한 파일에선 "기본 켜짐", 다른 파일에선 "기본 꺼짐"으로 반대로 적혀 혼란을 줍니다. (지금 동작엔 영향 없음)

개발자용 — 원인·근거

src/lib/env.ts FEATURE_* 3개 기본 'false' vs feature-flags.ts 기본 ON(!=='false'). 실사용은 feature-flags.ts만. 제안: env.ts 쪽 제거 또는 단일화.

⚪ 6번 "일자리 알림 받을래요?" 권유가 영영 안 떠요

댓글 달면 "알림 받을래요?"는 뜨는데, 일자리용 알림 권유는 띄우는 코드만 있고 부르는 데가 없어서 한 번도 안 떠요.

개발자용 — 원인·근거

setPushToastTrigger('comment')만 호출(CommentInput.tsx), 'job' 호출처 없음. 제안: 일자리 페이지/저장 시점에 setPushToastTrigger('job') 연결.

⚪ 7번 환경 판별 코드가 두 벌이에요

"이 사람 안드냐 아이폰이냐" 판별하는 코드가 거의 똑같이 두 군데 복사돼 있어요(딱 한 가지, 안드 앱 구분만 달라요). 지금은 영향 없지만 나중에 한쪽만 고치면 어긋날 수 있어요.

개발자용 — 원인·근거

detectEnv()(AddToHomeScreen) vs getBrowserEnv()(gtm.ts) UA 분기 복제. 유일 차이=TWA(referrer). detectEnv는 설치유도(현재 OFF)에만 쓰여 무해.

✅ 검수에서 "문제일까?" 의심했지만 괜찮았던 것들: 실험 그룹 배정의 무작위성(균등 OK), 봇/내부트래픽 거르기(일관 OK), 게이트 A그룹 0% 표시(어드민이 올바르게 분리 처리). 헛걱정 안 하셔도 됩니다.
