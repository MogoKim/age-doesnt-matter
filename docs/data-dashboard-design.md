# 우나어 실시간 데이터 대시보드 — 설계서

> 작성: 2026-06-07 | 상태: 설계 확정, 구현 대기
> 목적: 어드민에서 봇을 제외한 실고객 데이터를 실시간으로 보고, 매일 의사결정한다.
> 관련: `agents/scripts/insights.ts`(봇 제외 4대 지표 CLI), `docs/analysis/customer-behavior-2026-06-07.md`(현황 분석)

---

## 1. 왜 만드는가

현재 어드민 메인 대시보드는 KPI(방문/가입/PV)는 실시간으로 보여주지만 **리텐션(재방문)을 보여주지 못한다.** 리텐션은 `npm run insights`(터미널)로만 볼 수 있어, "실시간 화면"과 "리텐션 분석"이 분리돼 있다.

데이터가 말하는 우나어의 현실:
> **양동이에 물을 붓고 있는데 바닥에 큰 구멍이 나 있다.**
> 매일 방문 3,000+, 그러나 첫 방문 96%가 하루만 보고 영영 안 옴.

이 단계에서는 "물 더 붓기(광고)"보다 "구멍 막기(리텐션)"가 압도적으로 중요하다. 따라서 대시보드의 중심도 **리텐션**이어야 한다.

대시보드의 목적: **"매일 아침 30초 보고 → 오늘 뭐에 집중할지 결정"** — 숫자는 적고 명확하게.

## 2. 지표 체계 (확정)

### 🎯 북극성 지표 (North Star): 주간 재방문 실고객 수
- **정의**: 해당 주(월~일, KST)에 활동/방문한 실고객 중, 가입이 그 주 이전인 사람 수 (= "다시 온 진짜 손님")
- **봇 제외**: `providerId`가 순수 숫자(`^\d+$`)인 진짜 카카오 유저만
- **해석**: 오르면 구멍이 막히는 중(건강) / 떨어지면 양동이가 새는 중(경고)
- **보조 표시**: 지난주 대비 증감(↑↓), 최근 8주 추이 막대

### 북극성을 떠받치는 3영역 (AARRR 축약: 획득→활성화→유지)

| 영역 | 지표 | 핵심 질문 | 현재값(6/7) |
|---|---|---|---|
| **① 들어오기 (획득)** | 채널별 세션·가입전환율·재방문율 | "좋은 손님이 어디서 오나" | Kakao 30% ≫ Google 12% ≫ 직접입력·YouTube 1% |
| **② 첫 경험 (활성화)** | 가입→온보딩→첫 활동(글/댓글) 전환 | "가입하고 실제로 쓰나" | 활동자 17.5% |
| **③ 다시 오기 (유지)** | D1·D7·D30 코호트 리텐션 + 세션 리텐션 | "단골이 늘고 있나" | 세션 D7 ~1%, login 2일+ 34% |

### 화면 레이아웃(개념)
```
┌─ 🎯 주간 재방문 실고객 수: 15명 (지난주 12 ↑) ──┐
│  최근 8주 추이 막대                              │
└────────────────────────────────────────────┘
┌─ ① 들어오기 ──┬─ ② 첫 경험 ──┬─ ③ 다시 오기 ──┐
│ 채널 효율 표   │ 활성화 퍼널   │ D1/D7/D30 곡선  │
└──────────────┴─────────────┴──────────────┘
```

## 3. 측정의 한계와 선행 보강 (중요)

### 현재 한계
- **`login` 이벤트 트래킹은 2026-05-15부터** 정확 → 그 이전 가입자의 재방문 측정은 보수적.
- 분석 스크립트(`_retention-*.ts`)의 봇 필터가 부정확(`bot_`만 제외) → `insights.ts`의 `^\d+$` 기준이 정답.

### ✅ 측정 보강 불필요 — 데이터로 검증함 (2026-06-07)
당초 "`page_view`에 userId가 5%만 붙는다 → 보강 필요"로 추정했으나, **실측 결과 정상이었다.**
`api/events/route.ts:56`이 서버에서 `auth()` 세션으로 userId를 이미 부착하고 있고:
- **로그인 이후 같은 세션 page_view의 90.1%에 userId 부착** (정상 작동)
- 전체 page_view의 11.7%만 userId가 붙는 건 단지 **방문자 88%가 비로그인**이기 때문 (구조상 정상)

→ **트래킹 코드 수정 불필요.** 로그인 유저 단위 재방문은 정확히 측정 가능.
- (선택, 우선순위 낮음) User에 `isBot` 컬럼 추가 → providerId 규칙 의존 제거.

## 4. 구현 방식 (기술 설계)

### 4-1. 집계 전략: 1차는 "서버 쿼리 + 캐시", 데이터 커지면 "매일 집계"로 승격
- **현재 규모(실고객 63명, 30일 EventLog 수천 건)에서는 `insights.ts` 쿼리가 수 초면 끝난다.** 따라서 매일 집계 인프라(에이전트 변경)는 과투자다.
- **1차 구현(권장)**: 어드민 페이지가 서버에서 직접 쿼리하고 `unstable_cache`(30분)로 감싼다. 에이전트 핵심 로직(`agents/cdo`)을 건드리지 않아 안전하고 빠르게 출시.
- **승격 시점**: EventLog가 수십만 건이 되어 30분 캐시 갱신마다 페이지가 느려지면, 그때 CDO(`agents/cdo/kpi-collector.ts`, 매일 22:00)에 집계를 추가해 BotLog `details` JSON 또는 전용 테이블에 저장하고, 페이지는 저장값만 읽도록 전환.
- 화면에 "기준: 최근 N일 / 캐시 30분" 명시.

### 4-2. 저장 위치 선택지
- (A) BotLog `details` JSON 확장 (현행 KPI와 동일 패턴) — 추가 마이그레이션 0, 빠른 시작. **권장.**
- (B) 전용 테이블 `DashboardSnapshot` 신설 — 정석이나 스키마 마이그레이션 필요(HANDOFF 영역).
- 1차는 (A)로 시작, 데이터 쌓이고 추이 쿼리 필요해지면 (B)로 승격.

### 4-3. 봇 제외 로직 일원화
- `insights.ts`의 `isRealUser(providerId) = /^\d+$/.test()` 기준을 그대로 사용.
- 단, agents/(집계)와 src/(표시)는 런타임 import 불가 → 각자 동일 기준을 정의(주석으로 "단일 기준" 명시)하거나, 향후 공용 상수화 검토.

### 4-4. 새 페이지
- `src/app/admin/(panel)/insights/page.tsx` — 북극성 + 3영역 카드.
- 네비게이션 `admin-nav.ts`에 "📈 인사이트" 추가 (대시보드 바로 아래).
- 저위험(읽기 전용 표시). 단 CDO 집계 로직 변경은 agents 핵심 영역 → 신중 진행.
