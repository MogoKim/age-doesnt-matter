# 리텐션 실험 설계서 (2026-06)

> **목적**: 흩어진 리텐션 질문 — ① A/B/C variant가 재방문에 주는 효과 ② 앱푸시·알림톡 개입 효과 ③ 어떤 문구가 영향 ④ 푸시 타이밍 — 을 측정 가능한 실험으로 정리한다.
> **전제**: 맨바닥이 아니다. 이미 `src/lib/experiments/` 프레임워크 + ITT 리텐션 쿼리 + 웹푸시가 돌고 있다. 이 문서는 **그 위에 무엇을 어떻게 얹을지**를 설계한다.
> **이 문서가 하는 것**: "무엇을 / 어떤 variant로 / 어떤 지표로 / 어떻게 측정할지"의 구조.
> **이 문서가 안 하는 것**: 표본 수 계산, 통계 검정력, 구체 발송 코드 — 별도 요청 시.

---

## 0. 측정 기반 — 리텐션을 어떻게 정의하는가

실험을 설계하기 전에 "리텐션"이 무엇인지 못 박는다. 현재 코드에 **두 가지 정의가 공존**하고, 둘은 용도가 다르다.

| 정의 | 코드 | 계산 방식 | 쓰는 곳 |
|---|---|---|---|
| **고정 코호트 생존곡선** | `getRetentionQuadrants()` (`admin.retention.ts`) | 기준일 + N일 **이후** 재방문 존재. 분모 = 세그먼트 전체(같은 사람 끝까지 추적). D1≥D3≥…≥D30 단조 보장 | **전체 추세** — TWA/웹 × 회원/비회원 4분면 건강도 |
| **rolling window ITT** | `getGateITT()` (`admin.experiments-web.ts`) | 배정/가입 +N일 **내** 재방문. 가입 1h 후 이벤트만(자기클릭 제외). 봇 제외 | **실험 비교** — variant 간 차이 |

> **설계 원칙: 실험 간 비교는 rolling ITT, 서비스 전체 건강도는 코호트 곡선.** 둘을 섞어 읽지 않는다.

### 0-1. 측정 단위 정의 (한 표로)

| 항목 | 정의 | 코드 근거 |
|---|---|---|
| TWA(앱) 판정 | EventLog `properties.browser_env === 'twa-android'` | `isTwa()` |
| 봇 제외 | EventLog `isBot = false` | 전 쿼리 공통 |
| 재방문 신호 | `page_view`(+ 회원은 `login`) 이벤트 | `retention.ts` L72 |
| 회원 식별 | `User.providerId` 가 순수숫자(`^\d+$`) = 진짜 카카오 가입자 | `isRealUser()` |
| 비회원 식별 | `userId IS NULL` + `sessionId` | `retention.ts` L96 |
| 가입자 리텐션 | `userId` 기반 재방문 (OAuth로 sessionId 끊겨도 유효) | `getGateITT` `signupD1/D7` |
| "자기 클릭" 제외 | 가입/배정 **1시간 이후** 재방문만 카운트 | `+ 3600_000` 필터 |

### 0-2. 창업자 요구 "D1·D2·D3 … D7 daily"

- **현재**: 실험 쿼리는 D1/D3/D7만(`getGateITT`), 코호트는 D1/D3/D7/D14/D30.
- **설계**: 실험 비교용으로 **D1~D7 daily 전부**를 보려면 `getGateITT` 의 `[1,3,7]` 루프를 `[1,2,3,4,5,6,7]` 로 확장하면 된다(구조 동일, 측정 정의 불변).
- ⚠️ **경고**: TWA 가입자 표본이 작다(코드 곳곳 "표본 작을 때 % 착시 방지용 명수 병기"). daily로 쪼개면 각 칸 분모가 더 작아져 **%는 노이즈**가 된다. → **반드시 명수(`d1ReturnCount`) 병기**, daily는 "곡선 모양 보기"용, 의사결정은 D1/D3/D7 기준.

## 1. 실험 A — variant별 리텐션 (지금 돌고 있는 게이트 실험)

지금 `twa01_entry_gate` (A 현행 / B soft게이트 / C hard게이트)가 실제로 배정·측정 중이다. 새 실험이 아니라 **이미 있는 데이터를 리텐션 관점으로 읽는** 것.

| 항목 | 내용 |
|---|---|
| **목적** | 가입 게이트 강도(없음/소프트/하드)가 가입 후 재방문에 주는 효과 측정 |
| **배경** | 앱 설치자는 재방문 동기는 높은데 가입률은 낮음. "가입 먼저"가 진성 회원→재방문↑인지 |
| **가설** | 함정 = 가입률(강제라 당연 높음). **정답 = 가입 후 재방문**. hard 게이트가 가입률은 올려도 재방문이 안 따라오면 실패 |
| **variant** | A 현행(대조) / B soft(글 2~3개 열람 후 등장) / C hard(첫 화면 가입) — `registry.ts` 전문 보존 |
| **노출/배정** | `twa_gate_assigned` (배정 = ITT 분모) / `twa_gate_view` (실제 노출) |
| **전환** | `sign_up` (properties.twa_gate_variant + userId) |
| **리텐션 지표** | 배정 기준 D1/D3/D7 재방문 + **가입자 D1/D7**(userId 기반) |
| **지금 측정 가능?** | ✅ `getGateITT()` 에 이미 다 있음 |
| **갭** | ① D1~D7 daily 분해(0-2 참고) ② 어드민 카드에 리텐션 곡선 노출 |

**읽을 때 주의**(코드에 명시된 함정):
- 배정 분모(sessionId) 와 가입 분자(userId)는 **모집단 단위가 다름** → `signupRate`는 참고용 근사.
- A(현행)는 게이트를 안 띄워 `twa_gate_view` 노출이 0 → B/C와 노출 분모 직접 비교 불가. **그래서 배정(assigned)을 공통 분모로 통일**한 게 ITT.

## 2. 실험 B — 푸시 개입 (안 돌아온 사람을 부른다)

> 핵심 질문: "재방문 안 한 사람에게 푸시를 보내면 돌아오는가?"

이건 A와 결이 다르다. A는 *진입 경험* 실험, B는 *재호출(re-engagement)* 실험. **반드시 holdout(안 보내는 군)** 이 있어야 인과를 말할 수 있다.

| 항목 | 내용 |
|---|---|
| **목적** | 이탈 신호(D1 미복귀) 사용자에게 푸시 개입 시 재방문 회복 측정 |
| **배경** | 가입은 했는데 안 돌아오는 사람이 다수. 푸시 채널은 구축돼 있으나 재호출에 안 쓰고 있음 |
| **가설** | D1에 안 돌아온 사용자에게 N일차 푸시 → holdout 대비 재방문율 유의하게↑ |
| **대상 모수** | 가입 후 **D1(24h) 재방문 없는** 회원 |
| **배정** | 대상자를 **개입군 / holdout(무발송)** 으로 무작위 분할. `assign.ts` 결정론 해시 재사용 가능(userId 기준) |
| **개입(채널 트리)** | 아래 5장 채널 전략 따름 |
| **노출 이벤트(신규 필요)** | `push_sent` (properties: channel, campaign, variant) |
| **전환** | 발송 후 N일 내 재방문(page_view) + 클릭(UTM `utm_campaign` 도착) |
| **지금 측정 가능?** | ⚠️ 부분 — 발송 자체(`pushService.notify`)는 됨. holdout 배정·`push_sent` 적재·발송후 재방문 쿼리는 신규 |

**서비스성 vs 광고성 (중요 — 규제 직결)**:
- `pushService.notify` 는 `category:'ad'` 면 **marketingOptIn 동의자만 + 야간(21~08 KST) 차단 + `(광고)` 표기** 자동.
- "그냥 보러 오세요" 류 재호출 = **광고성** → 동의자만, 야간 못 보냄. 모수가 확 줄어든다.
- "회원님 글에 댓글 3개 달렸어요" 류 = **서비스성**(`category:'service'`) → 규제 무관, 전원 발송 가능.
- → **설계 선택지**: 재호출을 서비스성 트리거(실제 알림 사유)로 감싸면 도달률↑. 순수 마케팅 카피면 동의·야간 제약.

## 3. 실험 C — 문구(카피) 실험

> 핵심 질문: "같은 개입이어도 어떤 말이 더 돌아오게 하는가?"

개입·타이밍을 고정하고 **카피만** 변형. 과거 `f01_signup_content`(문구 A/B/C, 위너=공감형C 확정) 선례가 있다 — 같은 방식.

| 항목 | 내용 |
|---|---|
| **목적** | 동일 개입에서 카피 톤이 클릭·재방문에 주는 효과 |
| **적용 지점** | (a) 푸시 카피 (b) 게이트 카피 — 둘 다 |
| **variant 예시** | 공감형("요즘 통 안 보이시네요") / 혜택형("새 글 12개가 기다려요") / 호기심형("회원님 글이 인기예요") — *문안 확정은 창업자* |
| **variant 규칙** | `registry.ts` 원칙대로 **카피 전문을 variant.description 에 그대로 보존**(나중에 뭘 테스트했는지 추적) |
| **배정** | 카피 variant properties 로 push_sent 에 기록 |
| **전환** | 클릭률(UTM 도착) + 발송후 재방문 |
| **지금 측정 가능?** | ⚠️ B 인프라(push_sent + 배정) 위에서 properties 한 칸 추가하면 됨 |

## 4. 실험 D — 타이밍 실험

> 핵심 질문: "며칠차 / 몇 시에 보내야 가장 잘 돌아오는가?"

| 항목 | 내용 |
|---|---|
| **목적** | 발송 시점(경과일 × 시간대)이 재방문에 주는 효과 |
| **차원** | 경과일: D1 / D2 / D3 × 시간대: 아침 / 점심 / 저녁 |
| **배정 직교성** | `assign.ts` 의 `hashOffset` 으로 카피(C) 실험과 **직교 배정** — 타이밍과 카피가 섞이지 않게(기존 `f01_signup_timing` 이 `hashOffset:+7` 로 콘텐츠와 직교한 패턴 그대로) |
| **⚠️ 제약** | 광고성 푸시는 **야간 21~08 KST 발송 불가** → "저녁 늦게" 슬롯은 광고성에선 설계 불가. 서비스성만 가능 |
| **전환** | 발송후 N일 재방문 |
| **지금 측정 가능?** | ⚠️ 발송 스케줄러(`ScheduledPush`) + 타이밍 properties + 배정 신규 |

## 5. 채널·발송 전략 (창업자 지정 로직)

재방문 유도 메시지를 **누구에게 어떤 채널로** 보내는가의 우선순위 트리.

```
가입 회원 중 재방문 신호 없음
│
├─ TWA(앱 설치자)            → 앱푸시 (= 웹푸시 VAPID, TWA 환경에서 동작)
├─ 웹 사용자                 → 웹푸시
└─ 가입했으나 푸시 미구독 +   → 카카오 알림톡
   D1(24h) 내 미복귀
```

### 5-1. 채널별 현황

| 채널 | 상태 | 근거 |
|---|---|---|
| 웹푸시(웹) | ✅ 구축 | `pushService.notify`, VAPID |
| 앱푸시(TWA) | ✅ 동일 인프라 | TWA는 웹푸시 표준 사용 |
| 예약 발송 | ✅ | `ScheduledPush` 모델, `/admin/push` |
| **카카오 알림톡** | ❌ **미구축** | 코드·채널·템플릿 전무 |

### 5-2. 알림톡 선행조건 (블로커 — 창업자 결정/액션 필요)

알림톡은 "코드만 짜면 되는" 게 아니다. 아래가 먼저다:

1. **인적정보 필수화 (트레이드오프 — 창업자 결정)**
   - `User.birthYear`, `User.gender` 는 **현재 nullable(선택 동의)**. `phone` 도 선택.
   - 창업자 지정: "카카오에서 성별·연령층 **필수**로 받고 인적정보 알고 난 후 알림톡 발송".
   - ⚠️ **CLAUDE.md "카카오 정보 수집 최소화 / 가입 허들 낮추기" 원칙과 정면 충돌.** 필수 동의는 가입 전환율을 떨어뜨릴 수 있음 → **A 게이트 실험과 상호작용**(게이트도 허들, 인적정보도 허들). 둘 다 키면 가입률 영향 분리 측정 어려움.
   - → **결정 필요**: 전원 필수 vs "알림톡 타겟만 추가 동의" vs 보류.
2. **카카오 인프라 (외부·비용·심사)**
   - 카카오 비즈니스 채널 + 발신 프로필 + **알림톡 템플릿 사전 심사**(영업일 소요).
   - 발송 식별자 = `phone`(현재 선택 수집) 필요.
   - 발송 단가 발생(건당 과금) — 비용 영향 별도 산정.
3. **법적**: 알림톡도 광고성이면 야간/광고표기/동의 동일 적용. 정보성 알림(템플릿 승인 범위)만 무제한.

### 5-3. 정보통신망법 적용 범위

| 발송 성격 | 동의 필요 | 야간(21~08) | (광고) 표기 |
|---|---|---|---|
| 서비스성(댓글·등급 등 실제 사유) | ✗ | 발송 가능 | ✗ |
| 광고성(재호출 마케팅) | ✓ marketingOptIn | **차단** | ✓ |

## 7. 권장 실행 순서 (우선순위 — 수치/표본 설계는 별도)

1. **A 먼저 끝까지 읽기** (무비용): 이미 쌓이는 데이터. D1~D7 daily 분해 + 어드민 노출만 하면 "게이트가 재방문에 도움 되는가" 결론 가능.
2. **B 푸시 개입(웹푸시 holdout)**: 채널 구축돼 있음. `push_sent` + holdout 배정 + 발송후 재방문 쿼리만 추가하면 인과 측정 가능. **재호출을 서비스성 사유로 감쌀지** 먼저 결정(도달률 좌우).
3. **C·D는 B 위에 얹기**: B 인프라(배정·push_sent)가 서면 카피/타이밍은 properties 추가로 자연 확장.
4. **알림톡은 마지막**: 인적정보 필수화 결정(A와 상호작용 주의) + 카카오 채널/심사/비용. 가장 무거운 블로커.

### 창업자 결정 필요 (열린 항목)
