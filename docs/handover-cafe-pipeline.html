전체 파이프라인 흐름

우나어 콘텐츠 시스템은 두 개의 분리된 실행 환경으로 설계됐다. 집에 있는 Mac과 GitHub이 운영하는 클라우드 서버(GHA)다.

Mac이 하는 일 — 데이터 수집: 네이버 카페(wgang · dlxogns01)에서 글·댓글·지표를 실제 브라우저로 수집해 우리 DB에 쌓는다. 네이버가 자동화 봇을 감지·차단하기 때문에 "눈에 보이는 진짜 브라우저"(headless:false)가 반드시 필요하다. 클라우드 서버 환경에서는 이걸 흉내낼 수 없어서 Mac에서만 실행된다.

GHA가 하는 일 — 콘텐츠 발행: Mac이 쌓아 놓은 데이터를 원문 그대로 발행하고, 발행된 글에 봇 댓글을 순차적으로 단다. DB만 접근할 수 있으면 되므로 인터넷 연결만 있는 클라우드에서 24시간 자동으로 돌아간다. Mac 전원 상태와 완전히 무관하다.

핵심 설계 원칙 — 역할 분리의 이유: 수집(Mac)과 발행(GHA)을 분리했기 때문에 Mac이 꺼져 있어도 기존에 수집된 데이터로 발행·댓글이 계속된다. Mac이 하루 중 10시간만 켜져 있어도 GHA는 24시간 내내 콘텐츠를 발행한다. DB가 두 환경을 연결하는 유일한 다리다.

cron-job.org — 정확한 역할과 오해

⚠️ 오해 주의: cron-job.org는 GHA 실행 여부를 감시하거나 미실행을 보완하지 않는다. GHA cron과 cron-job.org는 둘 다 동일한 GitHub runner를 사용하기 때문에, cron-job.org로 dispatch해도 GHA 부하 지연은 그대로 적용된다.

cron-job.org가 실제로 하는 것: GitHub API(POST /repos/.../dispatches)에 workflow_dispatch 이벤트를 HTTP POST로 전송하는 외부 스케줄러다. 요청 자체는 1~2초 안에 성공(200)하지만, 그것이 GHA runner에서 언제 실제로 실행되는지는 보장하지 않는다. GHA 서버 부하가 높으면 최대 30~40분까지 지연될 수 있으며, 이는 cron-job.org가 개입할 수 없는 GitHub 내부 처리 영역이다.

구분내용
현재 운영 현황 큐레이션 20슬롯 전부(08:20~01:15 KST) + Wave 매 5분마다 dispatch 중
총 21개 크론잡이 cron-job.org에서 GitHub API로 dispatch
하는 것 GHA 자체 cron 외에 추가로 workflow_dispatch를 전송 → GHA 자체 cron이 지연되더라도 dispatch 트리거는 이중으로 발사됨
runner.ts 10분 중복방지 로직이 이중 실행을 차단함
하지 않는 것 GHA 실행 여부 감시 / 실패한 GHA 재실행 / GHA 지연 해소
dispatch 성공(HTTP 200) ≠ GHA runner 즉시 실행
GHA 지연 대응 GHA runner 지연은 cron-job.org로 해결 불가. 대신 runner.ts 중복방지 시간을 25분→10분으로 단축하여, GHA가 30분 지연된 경우에도 다음 슬롯(+45분)은 정상 실행되도록 함
왜 cron-job.org를 쓰나? GHA 자체 cron은 GitHub 정책상 실행 순서가 보장되지 않으며 지연이 잦다. cron-job.org는 더 신뢰성 높은 외부 스케줄링을 통해 dispatch 트리거 자체를 놓치지 않기 위한 이중 안전망이다. 단, dispatch 이후의 실행 지연은 GitHub runner 자체 문제이며 cron-job.org가 제어할 수 없다.

중복 수집 방지 3단계

크롤링이 하루 7회 실행된다는 건, 같은 카페를 하루에 7번 방문한다는 뜻이다. 같은 글이 중복으로 DB에 쌓이지 않을까? 실제로 이 문제가 생기면 동일한 원문으로 비슷한 큐레이션 글이 반복 발행되는 치명적 품질 문제가 생긴다.

3단계 방어 구조로 차단한다. 각 단계는 서로 다른 시점(수집 결정 전 → 저장 직전 → 인기글 동기화)에 작동하며, 한 단계가 뚫려도 다음 단계가 막는다.

단계 A — articleId 증분 크롤 (수집 단계)

crawler.ts는 전체글보기 페이지에 접근하기 전, DB에서 해당 카페의 MAX(articleId)를 조회한다. 그보다 articleId가 작거나 같은 글은 크롤링 자체를 하지 않는다.

결과: 이미 저장된 글은 Playwright로 페이지를 방문하지도 않는다. 트래픽 절약 + 처리 시간 단축.

코드 보기 — crawler.ts 증분 크롤 
// crawler.ts — 증분 크롤 핵심 로직
const latestArticleId = await prisma.cafePost.aggregate({
  where: { cafeId: cafe.cafeId },
  _max: { articleId: true },
})
const maxKnownId = latestArticleId._max.articleId ?? 0
// maxKnownId보다 큰 articleId를 가진 글만 수집

단계 B — postUrl unique 차단 (저장 단계)

혹시 articleId 체크를 통과한 글이 있더라도, savePosts 함수에서 postUrl로 DB 조회 후 이미 있으면 저장하지 않는다.

코드 보기 — crawler.ts savePosts 중복 차단 
// crawler.ts savePosts
const existing = await prisma.cafePost.findUnique({ where: { postUrl: post.postUrl } })
if (existing) continue  // 이미 있으면 저장 건너뜀

단계 C — 인기글 popular-sync: 갱신 vs 신규 분기

popular-sync.ts는 인기글 탭을 별도로 수집한다. 이미 DB에 있는 글이면 INSERT 없이 지표만 갱신하고 isPopular=true로 표시한다.

경우처리
Case A: DB에 이미 있는 글 killerScore 재계산 + isPopular=true 업데이트. INSERT 없음.
Case B: DB에 없는 새 글 이미지 의존글(본문 50자 미만 + 이미지만)·공지문이면 isUsable=false, 그 외는 isUsable=true로 신규 INSERT. isPopular=true는 항상 설정.
5

점수 체계 (qualityScore / killerScore)

수집된 카페 글이 모두 큐레이션 후보가 되는 건 아니다. 크롤 즉시 두 가지 점수를 계산해 글의 운명을 결정한다. qualityScore는 "이 글을 DB에 저장할 가치가 있는가, 큐레이션 후보로 쓸 수 있는가"를 판단한다. killerScore는 "이 글이 오늘 카페에서 터진 핫한 주제인가"를 측정해, 큐레이션 주제 선택 시 최우선으로 올린다.
세 개념의 관계 (한눈에)
qualityScore (0~100) → isUsable (true/false) 결정 기준: ≥30이면 true
killerScore (0~100) → 큐레이션 우선순위 결정: ≥55이면 킬러글로 최우선 발행 / ≥75이면 isFeatured 자동 활성화
두 점수는 독립적. isUsable=false여도 killerScore는 계산됨. 그러나 큐레이션 후보 풀에 없으므로 발행 불가.

qualityScore — 저장·후보 기준

크롤 시점에 즉시 계산. 저장 여부와 isUsable 여부를 결정한다.

좋아요·댓글·조회수(40%) + 본문 길이(20%) + 게시판 우선순위(20%) + 최신성(15%) + 미디어(5%) 합산

계산식 보기 — qualityScore 가중치 상세 
qualityScore =
  engagement   × 0.40  (좋아요+댓글+조회)
  contentLen   × 0.20  (본문 길이)
  media        × 0.05  (이미지/영상 수)
  boardPriority× 0.20  (게시판 우선순위)
  recency      × 0.15  (최신성)
qualityScore처리
< 20저장 자체 안 함 (minSave)
20 ~ 29저장하되 isUsable=false
≥ 30isUsable=true → 큐레이션 후보 풀 진입
단, 이미지 의존(본문 50자 미만)·공지문이면 false

killerScore — 킬러글 기준

인게이지먼트 지표에 집중한 별도 점수. 재크롤 시 갱신됨.

댓글 수(55%) + 좋아요(35%) 중심. 55점 이상이면 킬러글로 큐레이션 최우선 대상. 75점 이상이면 isFeatured 자동 활성화.

계산식 보기 — killerScore 가중치 상세 
killerScore =
  scoreEngagement(post)    × 0.55   // 댓글·좋아요 기반 인게이지먼트
  + scoreContentLength(post) × 0.20   // 본문 길이
  + scoreMedia(post)         × 0.05   // 이미지·영상 유무
  + scoreBoardPriority(post) × 0.15   // 게시판 우선순위
  + scoreRecency(post)       × 0.05   // 최신성
→ Math.round(합산값)  // 0~100 정수
killerScore처리
≥ 75킬러글 + Post.isFeatured 자동 true — 발행 즉시 집중 부스트(좋아요 라운드 + 봇댓글 캡 10개) 자동 적용
55 ~ 74킬러글 — 큐레이션 시 selectedTopics 앞에 최우선 삽입
< 55일반 처리
킬러글 처리 방식 killerScore≥50 글은 killerPosts 쿼리로 우선 처리된다. 원문(title/content)을 그대로 발행. AI 재창작 없음 (2026-05-20 제거). imageUrls 있는 글은 content-curator에서 완전 배제 → Sheet 라인(image-router)으로 유도.
killerScore 55점 달성 기준 (lifestyle 게시판, 본문 500자, 24시간 이내 기준)
댓글 8개  OR  좋아요 5개  OR  (댓글 5개 + 좋아요 3개)
조회수만으로는 달성 불가 — 가중치 0.01로 기여가 거의 없음. raw≥50이면 80점 고정(상한선).
isFeatured 자동 활성화 규칙 (2026-05-19 추가) killerScore ≥ 75인 글이 발행되는 순간 Post.isFeatured=true 자동 설정됨. 세 경로에서 동작:
① content-curator.ts 발행 트랜잭션 내
② popular-curator.ts 발행 트랜잭션 내
③ crawler.ts refreshRecentPosts — 재크롤 시 killerScore가 75를 넘으면 연결된 Post 자동 갱신
한 번 isFeatured=true가 되면 killerScore가 75 미만으로 내려가도 유지된다 (false 복구 없음).
⚠️ commentCount=0이면 killerScore 최대 35점 — 셀렉터 버그 영향 (2026-05-20 수정, SHA: b7cf87a)
killerScore 공식에서 댓글 가중치는 0.55로 전체 점수의 절반 이상이다. commentCount가 0으로 잘못 저장되면 이 55%가 통째로 0이 된다. 좋아요 10개짜리 인기글도 killerScore ≈ 35에 그쳐 킬러글 기준(≥55)을 통과하지 못하고 일반 글로 처리된다.

네이버 카페가 u_cbox 위젯으로 댓글 DOM을 교체하면서 기존 셀렉터(.comment_count, .comment_info_count .num 등)가 매칭되지 않아, 수집 즉시 safeNumber()가 fallback '0'을 반환했다. 2026-05-20 14:30 KST 이전 수집분 약 717건이 commentCount=0 상태로 DB에 남아 있다. 수정 후 refreshRecentPosts() 실행 시 이 건들의 killerScore가 자동 재계산된다.
→ 14:30 KST 이후 크롤된 글은 정상. killerScore가 낮게 보이는 과거 글은 재크롤 후 갱신됨.
6

isUsable — 발행 자격 관리

카페 글이 크롤되는 순간, 발행 자격이 결정된다. isUsable=true인 글만 45분마다 실행되는 큐레이션 후보 풀에 들어간다. isUsable=false면 DB에 저장돼 있어도 발행 대상에서 완전히 제외된다.

카페 크롤러는 글을 수집할 때 4가지 방어선을 동시에 적용한다 — 이미지만 있는 글, 게시판 공지문, 동영상 포함 글, 접근차단 안내문. 이 중 하나라도 감지되면 즉시 isUsable=false로 저장된다. 품질 점수(qualityScore) 30점 미만인 글도 마찬가지다.

인기글 탭(popular-sync)에서 수집된 글은 품질 점수 기준 없이 4가지 방어선만 적용한다. 인기글이라는 사실 자체가 품질을 보증하기 때문이다.

게시판 공지문?

isUsable=false

isUsable=false

isUsable의 생애 주기

1
savePosts() — 크롤 즉시 설정
crawler.ts의 savePosts 함수에서 qualityScore를 계산하고 즉시 isUsable을 결정한다.
isUsable = qualityScore >= QUALITY_THRESHOLDS.minUsable && !imageDep && !noticeText && !videoBlocked && !accessBlocked (minUsable = 30)
crawl-only 모드도 동일하게 적용됨. 11:30까지 기다리지 않는다. 이미지 의존·공지문·동영상·접근차단 — 4가지 방어선 중 하나라도 감지되면 점수와 무관하게 isUsable=false로 강제.
2
trend-analyzer.ts — 11:30 full 모드에서 재설정
전체 CafePost를 대상으로 QUALITY_THRESHOLDS.minUsable(=30) 기준 재계산. savePosts에서 설정된 값을 업데이트한다.
이 단계는 "처음 설정"이 아니라 "검증·정정"이다.
3
content-curator.ts — 큐레이션 후 usedAt 설정
발행에 사용된 CafePost는 usedAt에 현재 시각이 기록된다. usedAt이 NULL이어야 큐레이션 후보 풀에 포함된다. (isUsable은 변경 안 됨)
4
popular-sync.ts — Case B 신규글 isUsable 결정
인기글 탭에서 수집된 DB 신규 글(Case B)은 isPopular=true로 표시한다. isUsable은 4가지 방어선(이미지 의존·공지문·동영상·접근차단) 기준으로 결정: 하나라도 감지되면 false, 전부 통과하면 true. "인기글 = 품질 보증"이지만 4가지 방어선에 걸리는 글은 큐레이션에 부적합하므로 예외 처리한다.

// NOTICE_SIGNALS — 2개 이상이면 게시판 공지문으로 판단
const NOTICE_SIGNALS = ['게시판 안내', '인원모집', '무통보 강퇴', '신고게시판', '카톡, 쪽지'] as const

// isImageDependentContent — URL·공백 제거 후 본문 50자 미만
function isImageDependentContent(post: RawCafePost): boolean {
  if (post.imageUrls.length < 1) return false
  const effectiveLen = post.content
    .replace(/https?:\/\/\S+/g, '')
    .replace(/\s+/g, '')
    .length
  return effectiveLen < 50
}

"카톡 문의 주세요", "가입을 환영합니다" 같이 일반 글에 NOTICE_SIGNALS 단어 1개가 등장하는 경우를 허용하기 위해, 2개 이상 동시 출현을 공지문의 기준으로 삼는다.

동영상 포함 글 발행 차단

동영상이 포함된 카페 글은 어떤 경로로 들어와도 발행되지 않습니다. 크롤 시점에 즉시 isUsable=false로 저장되며, 큐레이터가 후보로 선택할 수 없습니다. 스프레드시트에서 직접 입력한 글도 동영상이 감지되면 FAILED 처리됩니다.

개발자용 — 동영상 차단 구현 상세

[CafePost 경로 — 2단계]

1단계 — crawler.ts savePosts:
videoBlocked = post.videoUrls.length > 0 || isVideoPlayerContent(post.content)
isUsable = ... && !videoBlocked — DB 저장 즉시 false.

2단계 — content-curator.ts + popular-curator.ts:
WHERE videoUrls: { isEmpty: true } (Prisma isEmpty 필터) + PZP 2차 필터: STRONG 1개 이상 or WEAK 2개 이상 감지 시 skip. ← 1단계를 통과한 글에 대한 백업 방어선.

[SHEET 경로 — 별도]
sheet-scraper.ts: 동영상 감지 → Sheet row status=FAILED + continue. CafePost.isUsable와 무관.

PZP 신호 (isVideoPlayerContent):
STRONG (1개+): .pzp / pzp-pc / pzp-poster / webplayer-internal-video / 광고 후 계속됩니다 / 디버그 정보 다운로드 / 고화질 재생이 가능한 영상입니다
WEAK (2개+): 재생 속도 / 해상도 / 자막 / 음소거 / 전체 화면 / 자동 (480p) / 0초

접근차단·가입 안내문 방어

네이버 카페 접근이 차단된 글(로그인이나 멤버십 필요 안내문)은 크롤 시점에 감지되어 isUsable=false로 저장됩니다. 스프레드시트 경로에서도 동일한 안내문이 감지되면 FAILED 처리됩니다.

개발자용 — 접근차단 신호 목록 및 구현 상세

[CafePost 경로 — crawler.ts]
ACCESS_BLOCKED_SIGNALS ANY 1개 이상 → accessBlocked=true → isUsable=false
신호 5종: '검색 비허용 게시물' / '가입이 필요합니다' / '카페의 멤버가 되어보세요' / '카페에 가입하면 바로 글을 볼 수 있어요' / '10초 만에 가입하기'

[SHEET 경로 — sheet-scraper.ts]
ACCESS_BLOCKED_SIGNALS (6종, '게시물을 확인하기 위해서는 가입이 필요합니다' 추가) 또는 BOARD_NOTICE_SIGNALS (2종) 감지
→ Sheet row status=FAILED + continue. CafePost.isUsable와 무관.

7일 재크롤 (refreshRecentPosts)

⚠️ 이것은 우나어에 발행된 글에 뭔가를 하는 게 아니다.
네이버 카페에 있는 원본 글의 지표(댓글수/좋아요/조회수)를 갱신하는 것이다.
오늘 오전 7:40에 수집한 글의 댓글 수가 30개였다고 해보자. 같은 날 오후에 댓글이 80개로 늘었다면, DB에는 여전히 30개로 저장돼 있다. 큐레이션 시스템은 "이 글은 반응이 별로 없다"고 잘못 판단한다.

더 결정적인 문제는 killerScore다. 처음 수집 시엔 점수가 낮아서 일반 글이었던 게, 댓글이 급증하면서 킬러글 기준(≥70)을 넘어설 수 있다. 이걸 갱신하지 않으면 "오늘 카페에서 터진 글"이 그냥 묻혀버린다.

이 문제를 해결하기 위해, 모든 크롤링(crawl-only + full) 직후에 최근 7일 인기 글 상위 50건을 다시 방문해 지표를 갱신한다. 변경이 있으면 killerScore를 재계산하고, 승격 조건을 충족하면 킬러글로 올린다.

동작 방식

refreshRecentPosts()는 crawl-only와 full 모드 모두에서 크롤링 직후 실행된다. 최근 7일 이내 인기 글 상위 50건을 Playwright로 재방문해 지표를 갱신하고, 변경된 경우에만 killerScore를 재계산한다.

동작 흐름 보기 — refreshRecentPosts 단계 
refreshRecentPosts()
  → DB에서 최근 7일 이내 CafePost 상위 50건 조회 (commentCount 내림차순)
  → 각 글의 실제 페이지 재방문 (Playwright)
  → likeCount / commentCount / viewCount 재수집
  → 변경된 경우에만 killerScore 재계산 후 UPDATE

killerScore 재계산 공식

공식 보기 — killerScore 재계산 
killerScore =
  scoreEngagement(post)    × 0.55   // 댓글·좋아요 기반 인게이지먼트
  + scoreContentLength(post) × 0.20   // 본문 길이
  + scoreMedia(post)         × 0.05   // 이미지·영상 유무
  + scoreBoardPriority(post) × 0.15   // 게시판 우선순위
  + scoreRecency(post)       × 0.05   // 최신성
→ Math.round(합산값)  // 0~100 정수
⚠️ 재크롤 경로는 위 5-factor 정식 공식이 아니라 단순식을 쓴다
위 5-factor(calculateKillerScore)는 savePosts() 최초 크롤 시점에만 적용된다. refreshRecentPosts()의 재계산은 댓글·좋아요만으로 계산하는 별도 단순식이다 (crawler.ts:1151-1153):
commentScore = min(commentCount × 10, 100) × 0.55 (최대 55점)
likeScore = min(likeCount × 20, 100) × 0.35 (최대 35점)
newKillerScore = round(commentScore + likeScore) → 최대 90점 (본문 길이·미디어·게시판·최신성은 재계산에서 제외)
승격 시나리오 처음 크롤 시 killerScore=50 (킬러글 아님) → 3일 후 댓글이 급증 → 재크롤에서 killerScore=78 → 킬러글(≥55) 승격 + killerScore≥75이므로 isFeatured 자동 활성화 → 다음 큐레이션에서 최우선 주제로 선택되고 집중 부스트까지 적용됨.

Wave 댓글 시스템

큐레이션 글이 발행되는 순간, 댓글이 하나도 없는 빈 글이 된다. 처음 방문한 유저 눈에는 '아무도 관심 없는 글'처럼 보인다. 봇들이 순차적으로 댓글을 달아 "이미 대화가 이뤄지고 있는 글"로 만드는 것이 Wave 시스템의 핵심 목적이다.

'파도(Wave)' 방식을 쓰는 이유는 자연스러움 때문이다. 발행 직후 여러 봇이 동시에 댓글을 달면 누가 봐도 부자연스럽다. 1분 → 5분 → 30분 → 60분으로 나눠 달면, 실제 커뮤니티에서 반응이 퍼지듯 자연스럽게 보인다. 각 wave는 댓글 유형도 다르다 — 공감 → 질문 → 경험 공유 → 다른 관점 순으로, 실제 대화가 발전하는 흐름을 흉내낸다.

인기글 시스템 (popular-sync.ts)

일반 크롤링은 '전체글보기' 탭의 최신 글을 시간순으로 가져온다. 그런데 카페에는 이미 많은 반응을 얻은 글들이 따로 모이는 '인기글' 탭이 있다. 여기 올라온 글은 카페 커뮤니티에서 이미 품질 검증이 된 글이다 — qualityScore를 새로 계산할 필요 없이 바로 큐레이션 후보 풀에 넣어도 된다. 단, 이미지만 있는 글이나 게시판 공지문은 큐레이션에 부적합하므로 예외 처리한다.

이 탭도 네이버 봇 탐지를 우회해야 하기 때문에 headless:false Playwright가 필요하고, Mac에서만 실행된다. Mac이 수집을 마친 뒤 15~20분 후 GHA의 popular-curator가 이 글들을 일반 큐레이션보다 우선적으로 발행한다.

Wave봇 상세

5분 간격으로 24시간 실행되면서 CommentWaveQueue(BOT 포스트)와 UserPostWaveQueue(유저 글)를 처리한다. wave1~4 각각 다른 유형의 댓글(공감형·질문형·경험공유형·다른관점형)을 생성한다. v2-E(현재): topComments 원문 기반 직접 생성, AI 없음. V2 OFF 또는 v2-F fallback(sourceCandidates 부족) 시에만 legacy 경로 (AI 가능). 환경변수: COMMENT_WAVE_V2_ENABLED / COMMENT_WAVE_V2_DATE kill switch로 제어.

SHEET 포스트(스크래퍼 글)의 wave 댓글은 Wave봇(wave-processor.ts)이 아닌 시드봇 scheduler.ts의 processPendingSheetCommentWaves()가 처리한다. BotLog 기반 별도 파이프라인 — §11 "SHEET 포스트 Wave 처리" 참조.

V2-E 티어별 wave count: killerScore≥75 → KILLER {wave1:2, wave2:4, wave3:5, wave4:4} / isPopular·killerScore 55~74 → HOT (슬라이딩) / 그 외 → NORMAL {wave1:1, wave2:1, wave3:2, wave4:1}. v2-F fallback: sourceCandidates(topComments 원문 기반 풀)가 해당 wave의 목표 수에 미달 → legacy path로 자동 라우팅 (BotLog reason=source_not_enough). V2가 ON이어도 원본 댓글이 없으면 자동 fallback.

한 봇이 하루에 달 수 있는 댓글 수는 최대 8개(BOT_DAILY_COMMENT_CAP, 2026-05-30 3→8 상향)로 제한된다. 한 글에 붙을 수 있는 봇 댓글은 tier별 cap(NORMAL 6 / HOT 14 / KILLER 20). 동일 봇이 같은 글에 이미 달았으면 스킵한다. 79명 페르소나 봇(bot-*@unao.bot 이메일 도메인, persona-data.ts)이 순환하며 댓글을 단다. board 가드로 MAGAZINE/JOB 글에는 wave가 처리되지 않는다(wave1~4 done 후 skip).

isUsable=true가 되려면 11:30 full 크롤을 기다려야 하나?
아니다. isUsable은 savePosts()에서 크롤 즉시 설정된다. crawl-only 모드(00:30, 07:40 등)에서도 qualityScore ≥ 30이면 그 자리에서 isUsable=true가 된다. 11:30 full의 trend-analyzer는 재검증(업데이트)이지 최초 설정이 아니다.

30점이다. config.ts의 QUALITY_THRESHOLDS.minUsable = 30. 이전에 60으로 하드코딩된 버그가 있었는데 Fix #2(SHA: e32fdf2, 2026-05-19)로 30으로 복원됐다. 어떤 문서에 60이라고 나와 있으면 그 문서가 틀린 것이다.
crawl-only 모드에서 댓글(topComments)이 수집되나?
설계상 수집된다. 단, 2026-05-19 이전 수집분은 전부 빈 배열이다.

키워드 매칭에서 bestScore > 0이면 desireCategory 풀을 완전히 무시하는 코드가 있었다. LIFE2 글의 키워드(돈, 노후, 투자 등)가 다른 게시판 페르소나의 interest와 매칭되면, LIFE2와 무관한 페르소나가 선택됐다. STORY 페르소나가 LIFE2 글을 쓰는 사례가 이것이다.
수정 후: desireCategory가 MONEY·RETIRE·HOUSING이면 bestScore 체크 전에 해당 카테고리 풀에서 무조건 무작위 선택한다. 키워드 매칭 결과는 무시된다.
웃음방(HUMOR 게시판)에 글이 하나도 안 올라온다. 큐레이터가 고르지 않는 건가?
큐레이터 문제가 아니라 AI 분류 단계 문제였다. 수정됐다(SHA: 8edc882).

큐레이터는 desireCategory='HUMOR' 또는 'ENTERTAIN'인 글만 웃음방(boardType=HUMOR)으로 발행한다. 그런데 수집된 cafePost 3,765건 중 desireCategory='HUMOR'인 글이 단 한 건도 없었다.
