전체 파이프라인 흐름

우나어 콘텐츠 시스템은 두 개의 분리된 실행 환경으로 설계됐다. 집에 있는 Mac과 GitHub이 운영하는 클라우드 서버(GHA)다.

Mac이 하는 일 — 데이터 수집: 네이버 카페(wgang · dlxogns01)에서 글·댓글·지표를 실제 브라우저로 수집해 우리 DB에 쌓는다. 네이버가 자동화 봇을 감지·차단하기 때문에 "눈에 보이는 진짜 브라우저"(headless:false)가 반드시 필요하다. 클라우드 서버 환경에서는 이걸 흉내낼 수 없어서 Mac에서만 실행된다.

GHA가 하는 일 — 콘텐츠 발행: Mac이 쌓아 놓은 데이터를 Claude AI로 재가공해 발행하고, 발행된 글에 봇 댓글을 순차적으로 단다. DB만 접근할 수 있으면 되므로 인터넷 연결만 있는 클라우드에서 24시간 자동으로 돌아간다. Mac 전원 상태와 완전히 무관하다.

핵심 설계 원칙 — 역할 분리의 이유: 수집(Mac)과 발행(GHA)을 분리했기 때문에 Mac이 꺼져 있어도 기존에 수집된 데이터로 발행·댓글이 계속된다. Mac이 하루 중 10시간만 켜져 있어도 GHA는 24시간 내내 콘텐츠를 발행한다. DB가 두 환경을 연결하는 유일한 다리다.

full 모드 5단계 상세 (11:30 KST)

크롤링은 하루 여러 번 실행되지만, 대부분은 글을 저장하고 끝나는 '빠른 수집'(crawl-only)이다. 11:30에는 전혀 다른 파이프라인이 실행된다. 단순 수집을 넘어, 오늘 카페의 분위기·욕망 트렌드를 AI가 직접 분석하는 3단계가 추가된다.

중복 수집 방지 3단계

크롤링이 하루 7회 실행된다는 건, 같은 카페를 하루에 7번 방문한다는 뜻이다. 같은 글이 중복으로 DB에 쌓이지 않을까? 실제로 이 문제가 생기면 동일한 원문으로 비슷한 큐레이션 글이 반복 발행되는 치명적 품질 문제가 생긴다.

3단계 방어 구조로 차단한다. 각 단계는 서로 다른 시점(수집 결정 전 → 저장 직전 → 인기글 동기화)에 작동하며, 한 단계가 뚫려도 다음 단계가 막는다.

단계 A — articleId 증분 크롤 (수집 단계)

crawler.ts는 전체글보기 페이지에 접근하기 전, DB에서 해당 카페의 MAX(articleId)를 조회한다. 그보다 articleId가 작거나 같은 글은 크롤링 자체를 하지 않는다.

결과: 이미 저장된 글은 Playwright로 페이지를 방문하지도 않는다. 트래픽 절약 + 처리 시간 단축.

코드 보기 — crawler.ts 증분 크롤 
// crawler.ts — 증분 크롤 핵심 로직
const latestArticleId = await prisma.cafePost.aggregate({
  where: { cafeId: cafe.cafeId },
  _max: { articleId: true },
})
const maxKnownId = latestArticleId._max.articleId ?? 0
// maxKnownId보다 큰 articleId를 가진 글만 수집

단계 B — postUrl unique 차단 (저장 단계)

혹시 articleId 체크를 통과한 글이 있더라도, savePosts 함수에서 postUrl로 DB 조회 후 이미 있으면 저장하지 않는다.

코드 보기 — crawler.ts savePosts 중복 차단 
// crawler.ts savePosts
const existing = await prisma.cafePost.findUnique({ where: { postUrl: post.postUrl } })
if (existing) continue  // 이미 있으면 저장 건너뜀

단계 C — 인기글 popular-sync: 갱신 vs 신규 분기

popular-sync.ts는 인기글 탭을 별도로 수집한다. 이미 DB에 있는 글이면 INSERT 없이 지표만 갱신하고 isPopular=true로 표시한다.

경우처리
Case A: DB에 이미 있는 글 killerScore 재계산 + isPopular=true 업데이트. INSERT 없음.
Case B: DB에 없는 새 글 isUsable=true 강제 설정 후 신규 INSERT. (인기글 = 품질 보증)
6

점수 체계 (qualityScore / killerScore)

수집된 카페 글이 모두 큐레이션 후보가 되는 건 아니다. 크롤 즉시 두 가지 점수를 계산해 글의 운명을 결정한다. qualityScore는 "이 글을 DB에 저장할 가치가 있는가, 큐레이션 후보로 쓸 수 있는가"를 판단한다. killerScore는 "이 글이 오늘 카페에서 터진 핫한 주제인가"를 측정해, 큐레이션 주제 선택 시 최우선으로 올린다.

qualityScore — 저장·후보 기준

크롤 시점에 즉시 계산. 저장 여부와 isUsable 여부를 결정한다.

좋아요·댓글·조회수(40%) + 본문 길이(20%) + 게시판 우선순위(20%) + 최신성(15%) + 미디어(5%) 합산

계산식 보기 — qualityScore 가중치 상세 
qualityScore =
  engagement   × 0.40  (좋아요+댓글+조회)
  contentLen   × 0.20  (본문 길이)
  media        × 0.05  (이미지/영상 수)
  boardPriority× 0.20  (게시판 우선순위)
  recency      × 0.15  (최신성)
qualityScore처리
< 20저장 자체 안 함 (minSave)
20 ~ 29저장하되 isUsable=false
≥ 30isUsable=true → 큐레이션 후보 풀 진입

killerScore — 킬러글 기준

인게이지먼트 지표에 집중한 별도 점수. 재크롤 시 갱신됨.

댓글 수(55%) + 좋아요(35%) 중심. 70점 이상이면 킬러글로 큐레이션 최우선 대상.

계산식 보기 — killerScore 가중치 상세 
killerScore =
  commentCount×10÷100 × 0.55
  + likeCount×20÷100 × 0.35
  + (기타 보너스)
  × 100
killerScore처리
≥ 70킬러글 — 큐레이션 시 selectedTopics 앞에 최우선 삽입
< 70일반 처리
킬러글 처리 방식 해당 카페 글을 그대로 복사하지 않는다. 글의 제목을 주제로 삼아 Claude가 새 글을 생성한다.
7

isUsable 완전 가이드 ⚠️ 오해 주의

isUsable은 큐레이션 시스템의 입장 티켓이다. isUsable=true인 글만 45분마다 실행되는 큐레이션의 후보 풀에 들어간다. false면 DB에 저장돼 있어도 발행 대상에서 완전히 제외된다.

이 필드를 잘못 이해하면 "왜 발행량이 줄었지?", "왜 후보 풀이 비어 있지?" 같은 문제를 엉뚱한 방향에서 디버깅하게 된다. 특히 이전 문서에 두 가지 오류가 있었기 때문에 이 섹션을 정확히 읽어야 한다.
⚠️ 이전 문서(v9.4)의 두 가지 오류 — 반드시 읽을 것 ❌ 구버전: "qualityScore 60점 이상에 isUsable=true" → ✅ 실제: 30점 이상
❌ 구버전: "crawl-only 모드에서 신규글은 isUsable=false 상태로 저장" → ✅ 실제: savePosts에서 크롤 즉시 설정 (crawl-only 포함)

isUsable의 생애 주기

1
savePosts() — 크롤 즉시 설정
crawler.ts의 savePosts 함수에서 qualityScore를 계산하고 즉시 isUsable을 결정한다.
isUsable = qualityScore >= QUALITY_THRESHOLDS.minUsable (minUsable = 30)
crawl-only 모드도 동일하게 적용됨. 11:30까지 기다리지 않는다.
2
trend-analyzer.ts — 11:30 full 모드에서 재설정
전체 CafePost를 대상으로 QUALITY_THRESHOLDS.minUsable(=30) 기준 재계산. savePosts에서 설정된 값을 업데이트한다.
이 단계는 "처음 설정"이 아니라 "검증·정정"이다.
3
content-curator.ts — 큐레이션 후 usedAt 설정
발행에 사용된 CafePost는 usedAt에 현재 시각이 기록된다. usedAt이 NULL이어야 큐레이션 후보 풀에 포함된다. (isUsable은 변경 안 됨)
4
popular-sync.ts — 인기글은 isUsable=true 강제
인기글 탭에서 수집된 글(Case B: DB 신규)은 qualityScore 무관하게 isUsable=true로 저장. 인기글 = 품질 보증.

한 줄 요약

isUsable 규칙 qualityScore ≥ 30이면 크롤 즉시(savePosts) isUsable=true. crawl-only도 포함. 11:30 분석 없어도 적용됨.
8

7일 재크롤 (refreshRecentPosts)

오늘 오전 7:40에 수집한 글의 댓글 수가 30개였다고 해보자. 같은 날 오후에 댓글이 80개로 늘었다면, DB에는 여전히 30개로 저장돼 있다. 큐레이션 시스템은 "이 글은 반응이 별로 없다"고 잘못 판단한다.

더 결정적인 문제는 killerScore다. 처음 수집 시엔 점수가 낮아서 일반 글이었던 게, 댓글이 급증하면서 킬러글 기준(≥70)을 넘어설 수 있다. 이걸 갱신하지 않으면 "오늘 카페에서 터진 글"이 그냥 묻혀버린다.

이 문제를 해결하기 위해, 모든 크롤링(crawl-only + full) 직후에 최근 7일 인기 글 상위 50건을 다시 방문해 지표를 갱신한다. 변경이 있으면 killerScore를 재계산하고, 승격 조건을 충족하면 킬러글로 올린다.

동작 방식

refreshRecentPosts()는 crawl-only와 full 모드 모두에서 크롤링 직후 실행된다. 최근 7일 이내 인기 글 상위 50건을 Playwright로 재방문해 지표를 갱신하고, 변경된 경우에만 killerScore를 재계산한다.

동작 흐름 보기 — refreshRecentPosts 단계 
refreshRecentPosts()
  → DB에서 최근 7일 이내 CafePost 상위 50건 조회 (commentCount 내림차순)
  → 각 글의 실제 페이지 재방문 (Playwright)
  → likeCount / commentCount / viewCount 재수집
  → 변경된 경우에만 killerScore 재계산 후 UPDATE

killerScore 재계산 공식

공식 보기 — killerScore 재계산 
killerScore = (commentCount × 10 ÷ 100 × 0.55 + likeCount × 20 ÷ 100 × 0.35) × 100
승격 시나리오 처음 크롤 시 killerScore=50 (킬러글 아님) → 3일 후 댓글이 급증 → 재크롤에서 killerScore=78 → 킬러글(≥70) 승격 → 다음 큐레이션에서 최우선 주제로 선택됨.

제약사항

항목값
대상 기간최근 7일 이내 CafePost
대상 건수상위 50건 (commentCount 내림차순)

업데이트 조건likeCount 또는 commentCount가 변한 경우에만

이 시스템 전체의 최종 산출물이다. Mac이 수집하고, AI가 분석하고, 스케줄러가 관리한 모든 과정이 여기서 결실을 맺는다.

큐레이션봇의 역할 — 정확한 범위

큐레이션봇(content-curator.ts)은 이름처럼 "글을 선정하고 가공해서 발행"하는 것이 핵심이다. 그런데 Wave 댓글 예약까지 트리거한다. 대댓글은 현재 미구현이다. 세 가지 역할을 정확히 구분해야 한다.

Wave 댓글 시스템

큐레이션 글이 발행되는 순간, 댓글이 하나도 없는 빈 글이 된다. 처음 방문한 유저 눈에는 '아무도 관심 없는 글'처럼 보인다. 봇들이 순차적으로 댓글을 달아 "이미 대화가 이뤄지고 있는 글"로 만드는 것이 Wave 시스템의 핵심 목적이다.

'파도(Wave)' 방식을 쓰는 이유는 자연스러움 때문이다. 발행 직후 여러 봇이 동시에 댓글을 달면 누가 봐도 부자연스럽다. 1분 → 5분 → 30분 → 60분으로 나눠 달면, 실제 커뮤니티에서 반응이 퍼지듯 자연스럽게 보인다. 각 wave는 댓글 유형도 다르다 — 공감 → 질문 → 경험 공유 → 다른 관점 순으로, 실제 대화가 발전하는 흐름을 흉내낸다.

큐레이션봇 상세

우나어 콘텐츠의 핵심 공급원이다. 매일 최대 20건(45분 × 20슬롯)의 글을 자동 발행한다. 단, 실제 발행량은 후보 풀(isUsable=true + usedAt=null) 크기와 P1~P3 필터에 따라 달라진다. Mac이 꺼져 있어도 GHA가 실행되지만, Mac이 크롤을 못 하면 신규 후보가 보충되지 않아 발행량이 줄어든다.

발행 시 CommentWaveQueue에 wave1~4를 자동 등록한다. 댓글 직접 작성은 하지 않는다. 대댓글 기능은 현재 미구현이다.

Wave봇 상세

5분 간격으로 24시간 실행되면서 CommentWaveQueue와 UserPostWaveQueue를 처리한다. wave1~4 각각 다른 유형의 댓글(공감형·질문형·경험공유형·다른관점형)을 생성한다. Claude Haiku 모델 사용.

한 봇이 하루에 달 수 있는 댓글 수는 최대 3개(BOT_DAILY_COMMENT_CAP)로 제한된다. 한 글에 붙을 수 있는 봇 댓글은 최대 5개. 동일 봇이 같은 글에 이미 달았으면 스킵한다. 50명 페르소나 봇(@unao.bot 이메일 도메인)이 순환하며 댓글을 단다.

시드봇(seed scheduler) 상세

50명의 페르소나 봇 중 한 명이 자신의 일상 경험을 담은 창작글을 직접 발행하는 봇이다. 현재 ENABLE_SEED_POSTS=false로 글 생성이 차단돼 있다. 단, 이 플래그는 글 생성 단계(scheduler.ts line 426~428)만 막는다 — 댓글·좋아요 등 다른 기능은 이 플래그와 무관하게 계속 실행된다.

controversy-chain(논쟁 글)과 magazine-generator(매거진 글)도 같은 agents-seed.yml에서 실행되지만 ENABLE_SEED_POSTS 체크를 하지 않아 계속 동작 중이다.

오해 주의: "씨드봇이 OFF니까 GHA agents-seed.yml이 아무것도 안 한다" — 틀렸다. controversy-chain과 magazine-generator는 계속 실행된다.

FAQ — 자주 하는 오해들

isUsable=true가 되려면 11:30 full 크롤을 기다려야 하나?
아니다. isUsable은 savePosts()에서 크롤 즉시 설정된다. crawl-only 모드(00:30, 07:40 등)에서도 qualityScore ≥ 30이면 그 자리에서 isUsable=true가 된다. 11:30 full의 trend-analyzer는 재검증(업데이트)이지 최초 설정이 아니다.
isUsable 기준이 60점인가, 30점인가?
30점이다. config.ts의 QUALITY_THRESHOLDS.minUsable = 30. 이전에 60으로 하드코딩된 버그가 있었는데 Fix #2(SHA: e32fdf2, 2026-05-19)로 30으로 복원됐다. 어떤 문서에 60이라고 나와 있으면 그 문서가 틀린 것이다.
crawl-only 모드에서 댓글(topComments)이 수집되나?
수집되지 않는다. topComments는 11:30 full 모드(DEEP 크롤)에서만 수집된다. crawl-only에서는 제목/본문/지표만 저장되고 topComments는 null이다.
killerScore가 높으면 그 카페 글이 그대로 우리 서비스에 발행되나?
아니다. killerScore ≥ 70인 글의 제목을 DailyBrief 주제 목록 앞에 삽입한다. 이 "주제"를 참고해서 Claude가 새 글을 생성한다. 카페 원문을 복사하지 않는다.
같은 글이 하루에 여러 번 크롤되어 DB에 중복 저장되지 않나?
저장되지 않는다. §5에서 설명한 3단계 방어 구조가 막는다. ① articleId 증분 크롤(이미 아는 ID는 아예 수집 안 함) → ② postUrl unique 체크(저장 단계) → ③ popular-sync Case A(기존 글은 UPDATE만, INSERT 없음).
씨드봇 글이 지금도 발행되고 있나?
seed scheduler(50명 페르소나 창작글)는 차단됐다. agents-seed.yml에 ENABLE_SEED_POSTS=false 설정 (SHA: 3b8f70e). BotLog에서 SEED [SUCCESS] publishedCount=0으로 확인 가능.

ENABLE_SEED_POSTS=false인데 왜 cafePostId=null BOT 글이 생기나?
controversy-chain과 magazine-generator는 ENABLE_SEED_POSTS 영향을 받지 않는다.

agents-seed.yml은 3가지를 실행한다:
  1. seed scheduler — ✅ ENABLE_SEED_POSTS=false로 차단됨
  2. controversy-chain execute — ⚠️ 차단 안 됨. 트렌드 기반 STORY 글 계속 생성
  3. magazine-generator — ⚠️ 차단 안 됨. MAGAZINE 글 계속 생성
따라서 _pm2_audit.ts §6에서 "cafePostId=null BOT 글"이 0건이 아닐 수 있다. 이것은 버그가 아니라 두 시스템이 의도적으로 계속 실행되는 것이다. 모두 중단하려면 agents-seed.yml에서 controversy-chain 줄 제거 + 해당 워크플로우에서 magazine-generate 제거 필요.
원문 보존율 90%/95%에서 AI가 실제로 바꾸는 게 정확히 뭐야?
바꾸는 부분은 매우 제한적이다. §10 "원문 보존율" 섹션에 상세 표가 있다. 요약:
  • 일반 큐레이션 (90% 보존): 첫 문장 말투 + 마지막 문장 마무리만 재가공. 중간 내용(지명·수치·경험·스토리) 완전 보존.
  • 킬러글 (95% 보존): 첫 문장 표현 1개 + 지명 익명화("OO구") + 어미 1~2개. 이야기 순서·핵심 수치·갈등 포인트 절대 변경 금지.
실제 확인: age-doesnt-matter.com에서 발행된 글의 본문과 원문 카페 글을 비교하면 내용은 동일하고 어미/시작·끝 문장만 다른 것을 확인할 수 있다.
Mac이 꺼져도 되는 시간과 켜야 하는 시간은?
§3 스케줄 섹션의 "Mac 켜짐/꺼짐 경계" 표를 참고. 요약:
  • Mac 필요 (하루 10회): 00:30, 07:40, 09:30, 10:30, 11:30★, 14:30, 16:00, 17:30, 21:15, 21:30 — Playwright 크롤링 (headless:false 필수, GHA에서 대체 불가)
  • Mac 불필요: 큐레이션 발행, wave 댓글, 시드봇 댓글/좋아요, DailyBrief fallback, brief-monitor — GHA가 24시간 자동 처리
해당 시각에 Mac이 꺼져 있으면 그 회차 크롤만 건너뜀. 나머지 GHA 자동화는 영향받지 않는다.
큐레이션이 적게 발행되는 원인은?
isUsable=true + usedAt=null인 후보가 부족하거나, 오늘 발행 cap에 도달했거나, P1~P3 필터로 후보가 모두 skip됐을 수 있다. 확인 방법: _pm2_audit.ts의 §10 "isUsable=true 큐레이션 후보 풀" + §11 "Silent Failure" 확인.
wave 댓글이 안 달리는 이유는?
CommentWaveQueue TTL(60시간) 초과 / 글당 봇 댓글 5개 cap 도달 / 봇 당일 3개 cap 도달 / 동일 봇 중복 방지 중 하나다. BotLog에서 botType=WAVE, status=FAILED 확인. _pm2_audit.ts §9로 오늘 FAILED 목록 확인.
