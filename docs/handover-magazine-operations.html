현재 판정

구현 완료 ≠ 운영 검증 완료 코드 연결과 설정은 끝났습니다. 하지만 실제 12:00/14:00 자동 실행으로 한 편이 제대로 발행되는 것은 아직 보지 못했습니다. "끝났다"가 아니라 "준비 끝, 첫 실측 대기"가 정확한 표현입니다.
항목상태

검색 노출(SEO) 성과는 이 문서로 판정할 수 없습니다. 구글이 새 글을 색인하고 순위를 매기는 데 시간이 걸리며, 성과는 추후 Search Console(GSC)로 봐야 합니다. 지금은 "발행 파이프라인이 도는가"까지만 봅니다.

AS-IS와 TO-BE — 무엇이 바뀌었나

핵심: 기존 시스템이 꺼진 게 아니라, "주제 선택"만 바뀌었습니다. launchd → runner → generator → Claude 글쓰기 → 이미지 → DB 발행 → Slack 보고 흐름은 그대로입니다. 그 흐름의 맨 앞 "오늘 무슨 주제로 쓸까"를 정하는 부분이 keyword queue로 교체됐습니다.

AS-IS (과거)

  • 주제 선택 = 시리즈 → GEO → 카페 트렌드 → 욕망지도
  • 하루 최대 3편(과거 기준)
  • 한 세션이 여러 편을 몰아 발행하기도 함

TO-BE (현재, 큐 모드)

  • 주제 선택 1순위 = keyword queue(검색 수요 기반 키워드)
  • 기존 시리즈/GEO/트렌드/욕망지도는 fallback(보조)으로 강등 — 삭제 아님
  • 하루 최대 2편 · 세션당 1편

즉 "하루 3편"은 과거 AS-IS 설명일 뿐, 현재 큐 모드 기준은 하루 2편입니다. 큐를 끄면(=false) 즉시 기존 AS-IS 방식으로 복귀합니다.

keyword queue 설명

keyword-universe.json — 구글 자동완성 등으로 모은 약 1,300개 키워드 우주(발행 후보 약 1,188개). 각 키워드에 클러스터·검색의도·민감도·발행정책(publishPolicy)·점수가 붙어 있습니다. gitignore 대상이라 커밋엔 안 들어갑니다.

selector (keyword-queue.ts) — 후보를 우선순위 순서로 정렬해 다음에 쓸 키워드를 고릅니다. 정렬 규칙:

규칙의미
클러스터 라운드로빈건강·관계·일자리 등 주제군을 번갈아 → 하루 2편이 서로 다른 주제군
클러스터 내 점수순같은 주제군 안에서는 수요·적합도 높은 것 우선
서브토픽 분산"국민연금 조기수령 ○○"처럼 같은 세부주제가 7일 안에 반복되지 않게 분산
publishPolicy 필터publish / publish_softened_title만 후보. no_publish_*는 절대 제외

publishPolicy 이중 가드

① 선택 단계: selector가 publish/softened만 반환. ② 발행 직전: generator가 다시 한 번 검사해 no_publish면 failed_no_publish_guard로 차단. 민감/부적합 키워드(보관용)는 절대 발행되지 않습니다.

state 파일 (소비 추적)

agents/magazine/data/keyword-queue-state.json — DB가 아니라 로컬 파일(gitignore). 무엇을 어디까지 썼는지 기록합니다.

  • consumed(소비, 재사용 안 함) = published + skipped_duplicate만
  • failed_* (생성·이미지·본문짧음·가드) = 미소비 → 다음 실행에서 다시 시도
  • 최근 이벤트 500개만 보관
  • ⚠️ state 파일 손상/백업 정책은 아직 미비 — 후속 과제(§11)

예외 처리 Q&A

Q. 주제(키워드)가 다 떨어지면?
consumed에 없는 키워드를 순서대로 씁니다. 발행 후보가 약 1,188개라 하루 2편 기준 1.6년치 분량입니다. 그래도 비면 기존 fallback(시리즈/트렌드/욕망지도)으로 넘어가 빈손으로 끝나지 않습니다.
Q. Mac이 꺼져 있으면?
그 시각엔 발행이 안 됩니다. 중요: 키워드는 소비되지 않고 다음 실행에서 이어 씁니다. 단, 빠진 분량을 다음 날 몰아서 보충 발행하지는 않습니다(여전히 세션당 1편). "다음 날 자동 재시도로 2배 발행" 같은 동작은 없습니다.
Q. 이미지가 실패하면?
인물(PERSON_REAL)은 외국 스톡 방지를 위해 fallback 없이 발행 보류(failed_image). 음식·장면·사물(FOOD/SCENE/OBJECT)은 Unsplash로 대체 가능. ChatGPT 세션이 만료되면 이미지가 실패하므로 재로그인이 필요합니다(§7).
Q. 글 생성이 실패하면?
failed_generation 기록 후 다음 후보로 넘어갑니다. 소비되지 않아 다음에 재시도됩니다.
Q. 본문이 너무 짧으면?
1,500자 미만이면 failed_body_short로 건너뜁니다(미소비).
Q. 제목이 중복이면?
최근 7일 유사 제목·전체기간 쌍둥이·seoTitle 중복은 skipped_duplicate로 건너뛰고 소비 처리(다시 시도 안 함). 기존 중복가드(isSimilarTitle / isDuplicateAllTime / seoTitle / unique slug)가 그대로 작동합니다.
Q. state 파일이 깨지면?
읽기 실패 시 빈 state로 시작해 발행이 멈추지는 않습니다. 다만 그동안의 소비 기록을 잃을 수 있어, 같은 글은 중복가드가 막아주지만 백업 정책은 미비(후속 과제).
Q. no_publish 키워드가 섞이면?
선택·발행 직전 이중 가드로 차단됩니다. 발행될 수 없습니다.
Q. 하루 2편을 초과할 수 있나?
아니요. 일일 카운트가 2 이상이면 실행 자체가 스킵되고, 세션당 1편에서 멈춥니다.
Q. push 안 했는데 실행되나?
됩니다. 매거진은 창업자 Mac의 launchd 로컬 실행이라 GitHub/원격과 무관합니다. main 반영은 별도(cherry-pick) 과제입니다.
Q. 큐를 끄려면?
.env.local의 MAGAZINE_KEYWORD_QUEUE_ENABLED을 =false로 바꾸면 즉시 기존 방식(시리즈/트렌드, 과거 동작)으로 복귀합니다.

글 / SEO 품질

이번에 좋아진 것 (prompt.ts 보강)

  • 핵심 키워드를 제목·첫 문단·FAQ에 자연스럽게 반영
  • 첫 문단에서 검색 질문에 곧바로 답하기
  • 확실하지 않은 통계·수치·출처를 지어내지 않기(환각 방지)
  • 과장·단정 표현 줄이고 사실 위주
  • FAQ를 실제 검색 질문형으로 작성
  • 제품 문구에 쓰지 않기로 한 표현(예: "시니어")은 쓰지 않고 "우리 또래/50·60대"로

아직 남은 것

  • topic hub / 시리즈 페이지 같은 SEO 구조 고도화
  • Article JSON-LD의 inLanguage 등 구조화 데이터 보강
  • 실제 글 품질의 사람 검수 루틴
검색 성과는 오늘 판정 불가글이 잘 나오는 것과 검색 상위 노출은 다른 문제입니다. 색인·순위는 시간이 걸리며, 성과는 추후 GSC로만 확인합니다.

이미지 운영

현재 이미지는 ChatGPT입니다. launchd morning/late 둘 다 IMAGE_GENERATOR=chatgpt로 실행됩니다. ChatGPT 웹 UI를 자동화해 이미지를 만든 뒤 Cloudflare R2에 업로드합니다.

이미지 타입처리
PERSON_REAL (인물)ChatGPT 전용. 실패 시 fallback 없이 발행 보류(외국 스톡 방지)
FOOD / SCENE / OBJECT실패 시 Unsplash 대체 가능
ILLUSTRATION현재 비활성(레거시 DALL-E 경로)
Gemini는 레거시(과거)과거에는 Gemini를 썼으나 현재 값은 ChatGPT입니다. (참고: morning plist 주석에 옛 gemini 표기가 남아 있으나 실제 환경변수 값은 chatgpt라 동작엔 무관 — 경미한 주석 정리 대상일 뿐.)
ChatGPT 세션 만료 시이미지가 실패해 인물 글은 보류되고 Slack 알림이 옵니다. 그때 chatgpt.com에 다시 로그인하면 됩니다(프로필 ~/.chrome-chatgpt-profile).

창업자가 알아야 할 것

지금 직접 할 것

  • Mac 켜두기 — 시스템 잠자기만 피하면 됨(화면 꺼짐 무관). 꺼져 있으면 그 시각 발행은 건너뜀
  • Slack #매거진 보기 — 매 실행 결과가 옴. 알림이 없으면 그날 미발행
  • 이미지 실패 알림이 오면 chatgpt.com 재로그인

지금 하지 말 것

  • 수동 발행/DB 직접 수정 — 자동 실행으로 관찰
  • launchd/plist 임의 변경
  • poc/admob-test 전체를 main에 올리기(섞인 커밋 多 — cherry-pick만)
실패가 떠도 당황하지 마세요대부분의 실패(이미지·생성·짧음)는 그 키워드를 소비하지 않고 다음에 다시 시도합니다. 중복은 의도된 스킵입니다. 끄고 싶으면 env 플래그 =false 한 줄이면 즉시 기존 방식으로 돌아갑니다.

Claude Code read-only 점검 프롬프트

12:00 morning 결과 점검

read-only로 12:00 매거진 morning 실행 결과를 점검해줘.
금지: 코드 수정·커밋·수동 발행·DB write.
