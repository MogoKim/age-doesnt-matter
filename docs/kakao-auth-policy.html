1 아키텍처 개요

파일 구조 — 2개 파일로 분리 (절대 합치지 말 것)

파일실행 환경Prisma역할
src/lib/auth.config.ts Edge Runtime ❌ 사용 불가 Kakao provider 설정, 세션/쿠키 설정, authorized 콜백
src/lib/auth.ts Node.js (Serverless) ✅ 사용 DB 조회/생성, JWT 토큰 생성, signIn 콜백
src/middleware.ts Edge Runtime ❌ JWT 쿠키 직접 복호화 (getToken()), 온보딩 리다이렉트
왜 두 파일로 나뉘는가 미들웨어는 Edge Runtime에서 실행되는데 Prisma는 Edge Runtime에서 동작하지 않습니다. NextAuth v5는 이를 해결하기 위해 authConfig(Edge OK)와 전체 auth(Node only)를 분리하는 패턴을 권장합니다. middleware.ts는 auth()를 호출하지 않고 getToken()으로 JWT 쿠키만 복호화합니다.

model User {
  id           String     @id @default(cuid())
  providerId   String     @unique   // 카카오 숫자 User ID (문자열로 저장)
  nickname     String     @unique   // 임시: "user_{providerId 끝 8자리}", 온보딩 완료 시 교체
  profileImage String?              // 카카오 프로필 사진 URL
  email        String?              // 카카오 선택 동의 이메일
  birthYear    Int?                 // 카카오 선택 동의 생일(년도)
  gender       String?              // 카카오 선택 동의 성별
  phone        String?   @unique   // 카카오 선택 동의 전화번호, 정규화: "01012345678"

  role         Role      @default(USER)
  grade        Grade     @default(SPROUT)
  fontSize     FontSize  @default(NORMAL)

  status         UserStatus @default(ACTIVE)
  suspendedUntil DateTime?  // SUSPENDED 상태일 때 정지 해제 시각
  withdrawnAt    DateTime?  // WITHDRAWN 상태일 때 탈퇴 요청 시각
  isOnboarded    Boolean   @default(false)  // DB 저장용 (런타임은 nickname 패턴으로 판단)

  lastLoginAt    DateTime  @default(now())
  nicknameChangedAt DateTime?
  createdAt      DateTime  @default(now())
}

5 로그인/가입 플로우

사용자 → /login 페이지 → "카카오로 로그인" 버튼 클릭 → /api/auth/signin/kakao (NextAuth 시작) → 카카오 인증 서버 (kauth.kakao.com/oauth/authorize) → 사용자 동의 화면 → 카카오 콜백 → /api/auth/callback/kakao → NextAuth: oauth4webapi → token 교환 kauth.kakao.com/oauth/token ↑ conform 함수 실행 (Content-Type 정규화) → NextAuth: signIn 콜백 → BANNED → /auth/error → SUSPENDED (기간 중) → /auth/error → WITHDRAWN → 복구 후 통과 → 나머지 → 통과 → NextAuth: jwt 콜백 → 신규 유저: DB create, nickname="user_{last8}", needsOnboarding=true → 기존 유저: lastLoginAt 업데이트 → JWT 쿠키 발급 → middleware: needsOnboarding=true → /onboarding 리다이렉트 → middleware: needsOnboarding=false → 원래 경로 or /

Token 교환 시 핵심 처리 (conform 함수)

카카오 비호환성 문제 3가지 카카오는 Content-Type: application/json;charset=UTF-8을 반환하지만 oauth4webapi v3는 application/json만 허용합니다. 또한 WWW-Authenticate 헤더도 제거해야 합니다. conform 함수 없이는 token 교환 자체가 실패합니다.

6 온보딩 플로우

신규 유저 첫 로그인 → JWT에 needsOnboarding=true → middleware: 모든 요청을 /onboarding으로 리다이렉트 → /onboarding 페이지: 닉네임, 나이, 지역 설정 → 완료 시: unstable_update() 호출 → JWT의 needsOnboarding=false, nickname 갱신 → token.tokenRefreshedAt = 0 (다음 요청에 DB 재조회 강제) → 이후 원래 경로 or / 로 이동

8 핵심 코드 설명

auth.config.ts — Kakao Provider 설정 핵심

Kakao({
  clientId: process.env.KAKAO_CLIENT_ID ?? '',
  clientSecret: process.env.KAKAO_CLIENT_SECRET ?? '',
  checks: ['state'],   // ← 카카오 PKCE 미지원, 반드시 state만 사용
  token: {
    url: 'https://kauth.kakao.com/oauth/token',  // ← 명시 필수 (없으면 authjs.dev fallback)
    conform: async (response: Response) => {
      // Content-Type charset 정규화 + WWW-Authenticate 제거
    },
  },
})

JWT 콜백 — 30분 DB 조회 최적화

const TOKEN_REFRESH_WINDOW_MS = 30 * 60 * 1000  // 30분

// 30분 이내 갱신된 토큰은 DB 재조회 건너뜀
const lastRefresh = (token.tokenRefreshedAt as number) ?? 0
if (Date.now() - lastRefresh < TOKEN_REFRESH_WINDOW_MS) {
  return token  // DB 스킵
}
// 30분 경과 시 DB 재조회 → token.tokenRefreshedAt = Date.now()

전화번호 정규화

// 카카오 제공 형식: "+82 10-1234-5678"
// 저장 형식: "01012345678"
function normalizeKakaoPhone(raw: string | undefined): string | null {
  if (!raw) return null
  return raw.replace(/^\+82\s?/, '0').replace(/[^0-9]/g, '') || null
}

사고 3 2026-05-18 (하루 종일)

KOE010 invalid_client Bad client credentials

증상
카카오 token 교환 단계에서 KOE010 에러 → 전체 로그인 불가
원인
Vercel Production의 KAKAO_CLIENT_ID가 Default REST API Key(3a0c2b...)였는데, KAKAO_CLIENT_SECRET은 "우리 나이가 어때서" 플랫폼 키(4ec06c8a...)의 시크릿. ID와 Secret이 서로 다른 키에 속함.
해결
vercel env rm KAKAO_CLIENT_ID production --yes
printf "4ec06c8a15dbb67bd92701f51c38ec2e" | vercel env add KAKAO_CLIENT_ID production
vercel --prod --force
교훈
  • 카카오 authorize 엔드포인트는 client_id를 검증하지 않음 (가짜 ID도 동의 화면 표시)
  • token 엔드포인트에서만 ID+Secret 쌍 검증 → curl 직접 테스트 필요
  • 반드시 vercel env pull --environment=production으로 실제 프로덕션 값 확인 후 작업
  • 콘솔 스크린샷만 보고 "맞겠지" 가정 금지
사고 4 2026-06-02

카카오톡 공유 P0 — Service Worker가 Kakao SDK를 차단

증상
게시글 상세에서 공유 → 카카오톡 클릭 시 fallback toast 발생. 여러 환경에서 window.Kakao === undefined, SCRIPT_ERROR, SDK_NOT_READY_AT_CLICK 확인. 일부 시크릿/비로그인 환경에서는 talk-apps.kakao.com 다운로드 안내 또는 about:blank이 보였으나, 이는 별도 브릿지/세션 단계로 분리.
목표
카카오 SDK 로드, 초기화, sendDefault() 호출까지 정상화하고 일반 모바일 Chrome, 데스크탑 Chrome, TWA 실사용 경로에서 공유 가능 상태를 복구.
진짜 원인
public/sw.js의 fetch handler가 request.destination === 'script' 조건으로 Kakao SDK 요청(https://t1.kakaocdn.net/kakao_js_sdk/.../kakao.min.js)까지 인터셉트함. Service Worker 실행 컨텍스트의 CSP connect-src에는 t1.kakaocdn.net이 없어 SW 내부 fetch(request)가 net::ERR_FAILED로 실패. 결과적으로 Kakao SDK 미로드 → window.Kakao undefined → fallback 발생.
잘못된 가설
SRI 해시 불일치, crossOrigin CORS 강제, 도메인 미등록, Vercel 키 누락도 순차적으로 의심했으나 최종 P0 원인은 Service Worker의 타사 스크립트 인터셉트였다. 카카오 Developers 도메인/앱키 문제는 로그인 상태 웹 공유 성공 및 모바일/TWA 성공으로 사실상 배제.
해결
커밋 25478fd: public/sw.js에서 cross-origin 요청은 fetch handler 초반에 즉시 return. 동일 출처 정적 리소스만 cache-first 처리. static fetch chain에 .catch()를 추가해 rejected promise 방치 방지. 
const url = new URL(request.url)

if (url.origin !== self.location.origin) {
  return
}
검증
Playwright before/after로 확인. Before: Kakao SDK net::ERR_FAILED, SCRIPT_ERROR, window.Kakao undefined. After: Kakao SDK 200 OK, SCRIPT_LOADED, INIT_OK, SCRIPT_READY. 이후 iPhone Chrome 로그인 상태, 데스크탑 Chrome 로그인 상태, TWA에서 실제 공유 성공 확인.
UX 정리
커밋 2816740: fallback toast를 카카오 공유가 안 되어 링크를 복사했어요에서 카카오톡을 열 수 없어 링크를 복사했어요로 변경. 카카오톡 앱 없음/로그인 안 됨/업데이트 필요/시크릿 제한은 카카오 화면이 처리하므로, 우리 서비스가 전체 공유 실패처럼 표현하지 않는다.
교훈
외부 SDK가 로드되지 않을 때는 SRI/CORS/env/도메인만 보지 말고 Service Worker가 타사 리소스를 인터셉트하는지 반드시 확인한다. 특히 SW 내부 fetch()는 페이지의 script-src가 아니라 SW 응답 헤더의 connect-src 영향을 받는다.

AccessDenied — app-handoff Credentials provider 차단

맥락 — 앱 handoff는 별도 구조 앱 로그인 = 시스템 브라우저에서 기존 카카오 OAuth(남성차단·상태검증 그대로) 통과 → 1회성 handoff token → 앱 WebView에서 Credentials('app-handoff') provider로 세션 재발급. (신규 구조, 커밋 8e47896)
증상
/api/auth/callback/app-handoff에서 AccessDenied → 앱 세션 발급 실패
원인
auth.ts의 signIn 콜백 첫 줄 if (account?.provider !== 'kakao' || !profile) return false가 Credentials app-handoff까지 차단
해결
signIn 콜백 맨 위에 if (account?.provider === 'app-handoff') return true 분기 추가 (커밋 f65a70a). app-handoff는 authorize에서 HMAC 서명 + nonce 1회 소비 + status 검증을 이미 통과하므로 signIn은 통과만. kakao 남성차단/상태검증 그대로 유지, auth.config.ts 무변경.
교훈
  • signIn 콜백의 provider 화이트리스트 가드(provider !== 'kakao' → return false)는 새 provider 추가 시 함께 갱신. 누락 시 해당 provider 로그인 전면 차단.
  • Credentials provider는 자체 authorize에서 검증을 마치므로 signIn은 통과시켜야 한다(이중 차단 금지).
  • 앱 handoff는 세션 발급 레이어만 추가 — 웹 카카오 로그인 코드(남성차단·온보딩 판정)는 그대로 재사용.
