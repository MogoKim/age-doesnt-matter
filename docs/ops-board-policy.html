🪞 거울 보드(M05) 운영 정책

손 갱신 없이 git·CI·HTTP·DB를 비춰 운영 현황을 보여주는 실측 칸반의 동작 규칙
대상: 거울 보드를 보거나 수정하는 모든 사람·AI 세션 · 최종 갱신: 2026-06-05 · 코드: scripts/ops-board/

1 제1원칙 — 철학

손으로 갱신하는 문서는 100% 썩는다(drift). 그래서 보드를 “AI가 쓰는 문서”가 아니라 “git·CI·HTTP·DB를 조회해 자동 판정하는 거울”로 만든다. 주장이 아니라 증명이라 안 썩는다.
  • 카드 목록 = 손 정의 (거의 안 바뀜)
  • 카드 상태(컬럼) = probe 자동 (자주 바뀌는 건 시스템이 잡음)
  • drift는 항상 “상태”에서 난다 → 상태를 probe가 잡으면 문서가 늙어도 무해

2 컬럼 = 창업자 5분류

probe가 카드를 아래 5개 중 하나에 자동 배치한다. 정적 카드는 baseCategory 값으로 고정된다.

완료됨 배포완료·적용확인 지금가능 백로그 제외

6 3층 staleness — drift 방지 (사활)

최악의 시나리오 = probe 정의가 썩어 “틀린 GREEN”을 표시하는 것. 사람은 문서를 의심하지만 자동 GREEN은 신뢰하므로, 이게 손문서보다 더 위험하다. 3층으로 방어한다.
  • ① 데이터 신선도 — 각 결과의 checkedAt 타임스탬프
  • ② probe 가정 신선도 — probeReviewedAt 90일 초과 시 🔍 “판정로직 재검토” 배지 (틀린 GREEN의 유일한 방어선)
  • ③ 수동확인 신선도 — 눈검수 항목 (후속)

7 보안 — 절대 준수

  • server는 127.0.0.1 only bind (0.0.0.0 금지) — 로컬 전용 도구
  • DB는 read-only role만 — write 권한 자격 사용 금지. 권한이 구조적으로 write 거부(permission denied 확인됨)
  • SSE payload에 토큰·env·DB URL 비노출 — detail은 화이트리스트된 값만
  • gh는 읽기 명령만 (merge·api 임의호출 금지)
  • probe는 절대 write 안 함

9 확장 · 유지 규칙

  • 새 백로그 = 정적 카드로 추가 (목록은 손)
  • 기능 배포 완료 = probe 부착해 자동 판정으로 승격
  • DB proof 카드 = OPS_BOARD_READONLY_URL 필요 (없으면 git/ci까지만 판정)
  • 카드 정의는 TS(cards.ts) — 타입체크가 정의 오류를 컴파일에 잡음
  • 메인 src/ 침투 0, 의존성 추가 0 (SSE는 node 내장)
  • 코드 변경 후 npm run board:check 통과 필수

10 금지 사항

  • DB write
  • src/ 수정 / agents/ 수정
  • 의존성 추가
  • 0.0.0.0 바인딩
