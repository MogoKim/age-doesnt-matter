기획 구멍 11개 — 코드 실증 확정

G01
전체글보기 전환 — 크롤러 전면 재작성
crawler.ts 200줄+

crawler.ts:341-477 — 현재 개별 board.menuId 루프 방식. V4 기획의 /menus/0 전체글보기 전환은 이 구조 전면 교체를 의미. 기존 board별 maxPages·delay 설정, 게시판 누락 감지(0건 경고), QUICK 모드 HIGH게시판 필터 모두 재설계 필요.

핵심 변경: collectPostUrls() 함수 → crawlAllArticles(cafeId, { maxArticles:200, stopOnConsecutiveExisting:3 }) 으로 교체. 기존 코드는 config-v3-backup.ts로 보존 (롤백 대비).

주의: 전체글보기에서도 articleId는 URL에서 파싱 가능(crawler.ts:392-404 로직 재사용). 새 방식에서 같은 정규식 그대로 사용.

G02
articleId DB 컬럼 없음 — MAX 비교 불가
schema.prisma

schema.prisma CafePost 모델에 articleId 컬럼 없음. 코드는 URL에서 파싱하지만(crawler.ts:392-404) DB에 저장 안 함. V4 원칙 1(MAX(articleId) 비교)을 구현하려면 컬럼 필수.

Migration:
articleId Int? 추가 + @@index([cafeId, articleId])
백필: UPDATE "CafePost" SET "articleId" = CAST(regexp_replace("postUrl", '.*/articles/(\d+).*', '\1') AS INTEGER) WHERE "articleId" IS NULL;

G03
usedAt 필드 없음 — 발행 중복 방지 불가
schema.prismacontent-curator.ts

schema.prisma CafePost에 usedAt 없음. content-curator.ts:356-368에서 Post를 생성할 때 원본 CafePost의 usedAt을 갱신하지 않음. 동일 원문이 다음 큐레이션에서 다시 선택될 수 있음.

Migration: usedAt DateTime? 추가.
코드: content-curator.ts에서 발행 완료 후 prisma.cafePost.update({ where: { id: post.sourceId }, data: { usedAt: new Date() } })

G04
minUsable=60 — V4 기획(30)과 불일치
config.ts:L221

config.ts:221 minUsable: 60. V4 기획서에 30으로 변경했으나 실제 코드는 아직 60. isUsable 기준이 60이면 qualityScore 30~59 글은 모두 isUsable=false로 저장되어 발행 대상에서 제외됨.

수정: config.ts minUsable: 60 → 30. 단, 기존 저장된 글은 isUsable=false 그대로 유지. 신규 수집분부터 30 기준 적용. 기존 글 일괄 재처리는 하지 않음(G06 참조).

G05
수미상관 90% — content-curator 프롬프트 미전환
content-curator.ts:302-306

content-curator.ts:302-306 — 현재 "반반이" 방식(원본 구체 상황 50% + 페르소나 감정 50%). V4 일반글 수미상관 구조: 첫문장 공감 1줄 + 원문 90% + 마지막 질문 1줄. 화제성글: 원문 95% 그대로.

수정 범위: generatePost 프롬프트 전면 교체. 일반글·화제성글 분기 추가. 발행 후 preservationRatio 자동 검증 로직 추가 (85% 미만 시 재생성).
