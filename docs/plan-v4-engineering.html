02

기획 구멍 15개 — 상세 분석

G01
4회 DEEP = 비용 2.5배 폭증 — 현실화 필요
비용plist

현재 구조: 08:30 DEEP(풀파이프라인) + 12:55 QUICK(심리분석 스킵) + 20:40 CRAWL(크롤만). 비용 절감을 위해 의도적으로 설계된 3-tier 구조. V4 "4회 DEEP" 전환 시 심리분석(Haiku × 배치) 비용이 현재 대비 2.5배 이상 증가.

수정 제안: 11:30을 DEEP+분석으로 이전 (현재 12:55 QUICK을 대체), 15:30 신규 슬롯은 DEEP+큐레이션(심리분석 스킵), 21:30은 CRAWL 유지. 실제로는 2회 DEEP + 1회 DEEP-큐레이션 + 1회 CRAWL이 현실적.

영향 파일: agents/cafe/run-pipeline.ts 모드 분기 / ~/Library/LaunchAgents/ plist 3개 수정 + 15:30 신규 1개 추가

G02
articleId 필드 DB에 없음 — 중복 방지 로직 구현 불가
DB Migration코드

V4 중복방지 원칙 1은 MAX(articleId) 비교에 의존. 현재 CafePost 테이블에 articleId 필드 없음. URL(postUrl)에서 파싱 가능하나 별도 컬럼 없으면 인덱스 불가 → O(n) 풀스캔.

필요 Migration:

ALTER TABLE "CafePost" ADD COLUMN "articleId" INTEGER;
CREATE INDEX idx_cafepost_cafeId_articleId ON "CafePost"(cafeId, articleId);
기존 데이터 백필: UPDATE "CafePost" SET articleId = CAST(regexp_replace(postUrl, '.*/(\d+)$', '\1') AS INTEGER);

Prisma 추가: schema.prisma CafePost 모델에 articleId Int? + @@index([cafeId, articleId])

G03
usedAt 필드 DB에 없음 — 발행 중복 방지 불가
DB Migration

V4 중복방지 원칙 4는 usedAt IS NULL 조건으로 미발행 글만 선택. 현재 CafePost에 usedAt 필드 없음. 같은 원문을 여러 번 발행하는 것을 막을 수단이 없음.

필요 Migration: ALTER TABLE "CafePost" ADD COLUMN "usedAt" TIMESTAMP;
Prisma: usedAt DateTime? + content-curator에서 발행 완료 시 prisma.cafePost.update({ where: { id }, data: { usedAt: new Date() } })

G04
댓글 파동 1분/5분 — GHA cron으로 구현 불가
GHA아키텍처

V4 일반글 댓글 파동: 1분(1) · 5분(3) · 30분(3) · 1시간(1). 현재 GHA 최소 단위는 1분 cron이나 실제 실행 지연으로 정확한 1분 보장 불가. GHA 워크플로우 실행 자체에 20~40초 오버헤드 존재.

해결 방안 2가지:
(A) Process 내 타이머: 킬러 포스트 발행 직후 setTimeout 체인으로 Wave 1(즉시), Wave 2(5분), Wave 3(30분) 동일 Node.js 프로세스 안에서 처리. Mac이 켜진 채로 유지 필요.
(B) 현실적 타이밍 완화: 1분→발행 직후, 5분→10분, 30분→30분 유지. 사용자가 체감할 수 없는 수준의 차이.

권장: (B) 방식으로 V4 기획 수정. "1분" → "발행 직후", "5분" → "10분 이내"로 표현 완화.

G05
전체글보기 전환 — 기존 65개 board URL config 전면 교체
코드대형 변경

현재 agents/cafe/config.ts에 2개 카페 × 약 65개 게시판 URL이 하드코딩. V4 전체글보기(/menus/0) 단일 URL 전환 시 이 구조 전면 삭제. 단순해 보이지만 게시판별 크롤링 우선순위 로직, boardName 필터, 재시도 로직이 이 config 구조에 의존.

신규 config 구조:
{ cafeId: '29349320', menuUrl: '/menus/0?viewType=L', maxArticles: 120, stopOnConsecutiveExisting: 3 }
기존 board별 config는 완전 삭제. boardName은 크롤링 시 각 글에서 읽어 DB 저장 (메타데이터용).

영향 범위: config.ts CAFE_CONFIGS 전면 재작성 / crawler.ts 게시판 루프 → 단일 페이지 루프로 교체 / 약 200줄 변경 예상.

G06
isUsable 30 전환 시 기존 DB 글 호환성
DB품질

현재 config.ts minUsable=60. V4에서 30으로 낮추면 기존 isUsable=false(qualityScore 30~59)인 글들이 자동 재활용 대상이 됨. 이 글들의 품질을 검증 없이 발행하면 저품질 글이 커뮤니티에 올라갈 위험.

처리 방안:
(1) 기존 DB 글은 isUsable 재계산 않고 그대로 유지 (기존 글 안전)
(2) 신규 수집 글부터 qualityScore ≥ 30 적용
(3) 기존 글 중 qualityScore 30~59인 것: 별도 수동 검토 후 재활성화
권장 (1)+(2): config 변경은 앞으로 수집되는 글에만 적용.

G07
content-curator vs scheduler — 75건 발행 역할 불명확
아키텍처

V4 기획은 "75건/일 = 09~23시 매 시간 5건"이라 명시. 그런데 content-curator는 카페 원문 기반 AI 생성 글이고, scheduler(seed agent)는 79명 페르소나가 직접 새 글을 쓰는 것. 이 둘은 완전히 다른 파이프라인.

현재 구조:
- Scheduler(seed): 하루 ~30~40건 페르소나 직접 발행 (원문 없음, 페르소나 경험 기반)
- content-curator: 하루 3건 카페 원문 기반 글 발행 (hotTopics 상위 3개)
75건은 Scheduler 발행량을 의미하는 것. content-curator는 별도 3건/일 유지.

필요 명확화: scheduler의 post type activity 합계 계산 + maxPostsPerSlot 조정으로 75건 목표 달성 여부 검증. 현재 scheduler SCHEDULE 배열 기준 post type 합산 필요.

G08
LIFE2 게시판 발행 로직 없음
코드 누락

content-curator.ts의 killerDesireToBoardType()은 HUMOR/ENTERTAIN → HUMOR, 나머지 → STORY만 매핑. LIFE2(2막준비) 발행 로직이 없음. V4 발행 비율 35%(2막준비)를 달성하려면 이 함수 수정 필수.

수정 필요:
RETIRE·MONEY·JOB·HOBBY·DIGITAL·PRIDE → 'LIFE2'
HEALTH·FAMILY·DIGNITY·CARE·GRIEF·BEAUTY·FOOD·RELATION·MEANING·LEGACY·FREEDOM·TRAVEL → 'STORY'
ENTERTAIN·NOSTALGIA → 'HUMOR'

G09
날짜 오염 필터 — 구현 위치 미정 + 오탐 위험
코드QA

"오늘 24일인데 본문에 23일 명시된 글 발행 금지" 정책. 구현 위치와 정규식 패턴이 미정. 너무 엄격하면 "지난 화요일에", "다음 주 월요일" 같은 정상 문장도 필터됨(오탐).

구현 위치: content-curator.ts 발행 직전 필터 (수집 시가 아닌 발행 시). 수집은 하되 발행만 막는 방식이 안전.

필터 로직:
const todayDate = new Date().getDate(); // 오늘 날짜(숫자)
const todayDay = ['일','월','화','수','목','금','토'][new Date().getDay()];
// 제목에만 적용 (본문 전체는 오탐 과다)
const titleHasStaleDate = /(\d{1,2})일/.test(title) && !title.includes(`${todayDate}일`);
// 특정 요일 언급: "오늘 (요일)"이 본문 첫 줄에 있을 때만
권장: 제목 필터만 적용. 본문 전체 날짜 필터는 오탐 과다로 제외.

G10
GHA 스케줄 V4 후 타이밍 충돌
GHA

현재 GHA 09:00 trend-analysis는 08:30 DEEP 완료(약 30분)를 기다린 구조. V4에서 핵심 분석을 11:30으로 이전하면 GHA 09:00 trend-analysis가 데이터 없는 상태에서 실행됨. 또한 GHA 13:30 content-curate가 12:55 QUICK → V4 11:30 DEEP+분석으로 전환 시 타이밍 맞지 않음.

조정 필요 GHA 크론:
- 09:00 trend-analysis → 12:00로 이동 (11:30 DEEP+분석 완료 후)
- 09:30 brief-monitor → 12:15로 이동
- 13:30 content-curate → 12:30으로 이동 (11:30 분석 결과 활용)
- 20:30 trend-analysis → 유지 (저녁 크롤 보완)

G11
욕망 20개 추가 시 기존 DB 글 호환성
DB마이그레이션

현재 psych-analyzer 욕망 분류 13개(HEALTH, FAMILY, MONEY, RETIRE, JOB, RELATION, HOBBY, MEANING, DIGNITY, LEGACY, CARE, FREEDOM, ENTERTAIN). 신규 7개(BEAUTY, DIGITAL, NOSTALGIA, PRIDE, GRIEF, TRAVEL, FOOD) 추가 시 기존 수집된 글은 새 카테고리로 분류 안 됨.

처리 방안:
(1) 기존 글 재분석: psych-analyzer를 WHERE desireCategory IS NULL OR desireCategory IN ('FREEDOM','MEANING','HOBBY') 조건으로 재실행 → 새 카테고리로 재태깅. 비용 발생.
(2) 새 글부터만 20개 적용: 기존 13개 유지, 신규 수집분에만 20개 적용.
권장 (2): 기존 DB 재분석 비용 절감. 단, DESIRE_PERSONA_MAP은 20개 기준으로 즉시 업데이트.

G12
수미상관 방식 — 품질 검증 방법 없음
QA코드

V4 핵심 변경: 반반이(50%) → 수미상관(90%). content-curator 프롬프트 수정 후 "실제로 90% 보존되는가"를 검증할 자동화 방법이 없음. AI가 프롬프트를 따르지 않고 원문을 더 많이 변형할 가능성 존재.

검증 방법:
import { diffChars } from 'diff';
발행 후 원문과 생성 글 ratio 계산: similarity = 1 - (editDistance / maxLen)
임계값: 일반글 ≥ 0.85, 화제성글 ≥ 0.92 미만이면 재생성(최대 2회) 또는 스킵.
BotLog에 preservationRatio 기록 → Slack 주간 리포트에 평균값 표시.

G13
categoryMap → desireCategory 매핑 전환 시 scheduler에도 적용 필요
코드

generator.ts categoryMap은 boardType별 서브카테고리를 무작위 선택. V4의 desireCategory→subcategory 매핑은 content-curator뿐 아니라 scheduler가 호출하는 generatePost()에도 동일 적용 필요. 누락 시 시드봇 발행 글(~75건/일)의 카테고리는 여전히 랜덤.

수정 위치: generator.ts categoryMap 로직 → DESIRE_TO_SUBCATEGORY 매핑 테이블로 교체. getPersonaDesire(personaId)로 욕망 조회 → 서브카테고리 결정. scheduler.ts는 수정 불필요 (generator 내부 처리).

G14
2일 수집 창 — 카페 원문 풀이 급격히 줄어듦
품질

14일 → 2일 수집 창으로 좁히면 content-curator가 선택할 수 있는 CafePost 풀이 대폭 감소. 하루 수집 글 약 160건(2카페 × 80건)에서 2일치 = 최대 320건. 이미 usedAt 있는 것, qualityScore < 30인 것 제외하면 실제 발행 가능 풀이 50~80건 수준. 화제성글 기준(commentCount ≥ 5 + qualityScore ≥ 50)에 맞는 글은 하루 5~10건에 불과할 수 있음.

안전장치: 화제성글 풀이 부족할 경우 자동으로 7일 이내로 확장하는 fallback 로직 추가. if (viralCandidates.length < 2) extendWindow(7)

QA 방법론 — 3단계 검증 체계

원칙: 크롤러 변경은 실제 사이트에서 검증해야 함. 모의 데이터·목 불가. 각 단계는 독립적으로 검증 후 다음 단계 진입.

1
크롤러 단독 검증 (DB Migration 전)

검증 항목방법통과 기준
전체글보기 접근npx tsx agents/cafe/crawler.ts --dry-run --mode deep200 응답, 글 목록 파싱 성공
articleId 추출URL에서 숫자 파싱 확인10개 글 중 10개 articleId 추출
본문 50자 이상 필터dry-run 결과에서 50자 미만 글 수 확인전체의 5% 이내만 탈락
공지 필터제목에 [공지]·필독 포함 글 수 확인공지 글 0건 저장
날짜 오염 필터어제 날짜 명시된 테스트 글로 검증해당 글 발행 제외 확인

파이프라인 통합 검증 (스테이징 DB)

검증 항목방법통과 기준
articleId 중복 방지같은 글 2회 크롤 시도2번째 크롤에서 이미 있는 글 DB insert 0건

05

리스크 헷징 — 롤백 & 안전장치

리스크발생 확률영향도헷징 방법
전체글보기 크롤 차단
네이버가 /menus/0 bot 감지		 중 치명 기존 65개 board URL config를 별도 파일(config-v3.ts)로 보존. 차단 시 환경변수 CRAWL_MODE=board로 즉시 롤백. 로그인 없이 접근 가능한지 매일 health check 추가.
DB Migration 실패
articleId/usedAt 컬럼 추가 중 락		 저 치명 ALTER TABLE ... ADD COLUMN IF NOT EXISTS 사용. Supabase에서 컬럼 추가는 무중단. 단, 인덱스 생성은 CREATE INDEX CONCURRENTLY로 논블로킹 실행.
수미상관 품질 저하
AI가 원문을 너무 많이 변형		 중 높음 G12 preservationRatio 검증 로직 필수. 85% 미만 시 자동 재생성(최대 2회). 재생성도 실패 시 해당 글 발행 스킵(강제 발행 없음). BotLog에 generationAttempts 기록.
2일 풀 고갈
발행 가능 글 부족		 중 높음 G14 fallback 로직: viralCandidates < 2 → 7일 창으로 자동 확장. 일반글 풀 부족 시 scheduler 페르소나 직접 글로 보완 (content-curator 0건도 가능한 구조 유지).
Mac 미가동 크롤 실패
특정 시간대 plist 미실행		 중 중간 GHA daily-brief-fallback 이미 존재(09:00 KST). 11:30 슬롯용 fallback 추가: GHA 12:30에 brief-monitor로 DailyBrief 미생성 감지 시 전날 데이터로 폴백 운영.
날짜 오염 필터 오탐
정상 글이 필터됨		 저 중간 필터 적용 범위를 제목만으로 제한. 본문 전체 스캔 안 함. 필터 조건을 BotLog에 기록 → 1주 후 false positive 비율 확인. 오탐 > 10% 시 필터 비활성화 옵션.
