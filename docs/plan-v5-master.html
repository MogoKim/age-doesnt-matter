01

V4 기획 검수 — 문서 내 불일치 3가지 (개발 전 확정 필요)

⚠ 불일치 #1 — isUsable 임계값: Section 06에서 "≥30", P0에서 "60→40"

Section 06 일반글 선별기준: isUsable = true (qualityScore ≥ 30)
Section 12 P0 개발항목: QUALITY_THRESHOLDS: isUsable 60→40
코드 현재값: config.ts L224: minUsable: 60

→ 창업자가 명시한 "qualityScore ≥ 30" 기준을 채택. P0에서 minUsable: 60 → 30으로 수정. Section 02 표도 "40" → "30"으로 정정 필요.

⚠ 불일치 #2 — 페르소나 수: V4 "79명" vs 실제 확인 "~63명"

V4 Section 02: "79명 전체"
generator.ts DESIRE_PERSONA_MAP 실제 집계: A~BH = 58명 + EN1~EN5 = 5명 = 63명
persona-data.ts에 별도 추가된 페르소나가 있을 수 있으나 현재 DESIRE_PERSONA_MAP 기준 63명.

→ P3 개발 전 persona-data.ts 전체 카운트 재확인. 79명 달성하려면 16명 추가 필요. 추가 페르소나 욕망 배정표 별도 작성.

⚠ 불일치 #3 — HUMOR vs ENTERTAIN: generator.ts vs psych-analyzer.ts 명칭 충돌

psych-analyzer.ts (13개): ENTERTAIN (연예·드라마)
DESIRE_PERSONA_MAP (12개): HUMOR — ENTERTAIN 없음, FOOD·FREEDOM 있음
psych-analyzer: HUMOR·FOOD 없음, DIGNITY·LEGACY·ENTERTAIN 있음
두 파일이 서로 다른 욕망 카테고리를 사용 중 → 매핑 불가.

→ 통합 기준: psych-analyzer 13개를 기준으로 DESIRE_PERSONA_MAP을 정렬. HUMOR → ENTERTAIN 리네임. FOOD·FREEDOM은 psych-analyzer V4 확장 시 추가 (FOOD는 V4 신규 7개에 포함됨). 상세: Section 07 G06.

⚠ 불일치 #4 — MAGAZINE 네이버 블로그 연동: V4 bcard에 언급됐으나 P0~P3 로드맵에 없음

MAGAZINE 게시판 bcard-note: "네이버 블로그 자동 발행 연동" — 현재 agents/naver-blog/ 디렉토리만 존재하고 파이프라인 연결 없음. P0~P3에 개발항목 없음.

→ V5 로드맵에서 P4로 별도 추가. V4 본체와 분리해서 진행. agents/naver-blog/ 현황 파악 후 기획 필요.

기획 갭 완전 분석 — G01~G26 (코드 실증)

G01
전체글보기 크롤러 전면 재작성 CODE
Critical — P0 핵심, 이게 없으면 V4 전체 불가

현재: crawler.ts L353-477 — 게시판 루프. menuId=0은 L354-360에서 명시적 SKIP.

V4 요구: https://cafe.naver.com/f-e/cafes/{cafeId}/menus/0?viewType=L 단일 URL로 전체 최신글 수집.

구현 스펙:
1. collectPostUrls() 함수 → collectAllArticles(cafeId, maxArticles, stopOnExisting)로 교체
2. 목록 URL: /menus/0?viewType=L 페이지네이션 (스크롤 또는 페이지 파라미터)
3. 목록에서 추출: articleId, 제목, 게시판명, 댓글수, 조회수
4. 중복 체크: DB MAX(articleId) 조회 → 이하면 중단 (연속 3개 기존 articleId 시 중단)
5. 2일 초과 글 스킵 (postedAt < NOW()-2day)
6. 공지 필터: 제목에 "[공지]", "필독", "이벤트 당첨" 포함 시 스킵
7. 본문 진입: 신규 글만, 상세 수집 (본문+이미지+좋아요+댓글+대댓글)
8. config.ts: boards 배열 전체 삭제 → allArticlesUrl, maxArticlesPerCrawl: 200 구조로 교체

G02
articleId DB 컬럼 없음 DB
Critical — 중복 수집 방지 불가

현재: schema.prisma CafePost 모델에 articleId 없음. postUrl UNIQUE로만 중복 방지.

V4 요구: articleId Int 컬럼 + 카페별 MAX(articleId) 비교로 효율적 중단.

Migration 스펙:
articleId Int? // 전체글보기 전환 전 수집분은 null 허용
@@index([cafeId, articleId]) — 카페별 MAX 조회용
기존 데이터 백필: postUrl에서 /articles/(\d+) 정규식 추출 후 UPDATE

G03
usedAt DB 컬럼 없음 DB
Critical — 동일 원문 중복 발행 방지 불가

헷징: 첫 문장(공감) + 마지막 문장(질문) 반드시 추가 → 변형 창작물로 처리. 원문 출처 카페명은 내부 DB에만 유지, 발행 글에 미표시.

11:30 심리분석 일괄 실패 시 DailyBrief 미생성

psych-analyzer: BATCH_SIZE=5, 실패 시 1개씩 재시도. 전체 실패 시 fallback 없음.

헷징: DailyBrief fallback 로직 — 전날 DailyBrief 재사용. Slack "fallback 운영 중" 알림. 15:30/21:30 큐레이션은 fallback DailyBrief로 계속 실행.

R6 — 발행 비율 불균형

LIFE2 발행글 인기 없으면 비율 조정 필요

LIFE2 35% = 하루 26건. 초기엔 LIFE2 인풋 카페글이 충분하지 않을 수 있음.

헷징: LIFE2 인풋 부족 시 STORY로 자동 채움. scheduler.ts에 "LIFE2 인풋 최소 5건" 체크 후 미달 시 STORY 대체 로직.

R7 — articleId 연속성 문제

글 삭제 시 articleId 불연속 → 중단 로직 오작동

"연속 3개 기존 ID 시 중단" 정책. 삭제글 많으면 조기 중단 발생.

헷징: 연속 조건을 3→5로 완화. 또는 "연속 3개" 대신 "최근 10개 중 7개 이상 기존" 조건으로 변경. 운영 초기 로그 분석 후 결정.

R8 — 전환 기간 데이터 공백

크롤러 재작성 중 신규 글 수집 중단

P0 개발 기간(1~2일) 동안 크롤러가 비활성화되면 당일 인풋 없음.

헷징: 개발은 기존 크롤러와 병렬로 진행. 새 크롤러 crawlAllArticles() 완성 후 feature flag로 전환. 전환 실패 시 기존 코드로 즉시 롤백.
