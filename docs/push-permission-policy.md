# 푸시 알림 권한 요청 정책 (우나어)

> 단일 진실의 원천. "어떤 상황에서 누구에게 무엇을 보여줄지"를 모두 정의한다.
> 구현 전 정책 확정용. 타겟: 40·50·60대 여성. 목표: 구독자 0 → 확보 → 리텐션.

---

## 📊 한눈에 (AS-IS → TO-BE)

**한 줄**: 지금은 푸시 다 만들어놨는데 **권한을 거의 안 물어봐서 구독자 0명** → 알림 0통. 바꾸면 **자연스러운 순간마다 물어봐 구독 확보** → 답글 알림(리텐션) + 합법 광고.

| 항목 | AS-IS (현재) | TO-BE (변경 후) |
|---|---|---|
| 권한 요청 시점 | "댓글 작성 시" 1곳뿐 | ③글쓰기·댓글·②방문(정독후)·①가입 **4곳** |
| 트리거 동작 | **구조 버그로 안 뜸**(상주 토스트 1회만 읽음) | **이벤트 재평가로 수정** → 정상 노출 |
| 구독자 | **0명** (실유저 94명 중) | 점진 확보 |
| 발송 | 0통(파이프 놀고 있음) | 서비스(답글·전원) + 광고(동의자만+(광고)+야간제한) |
| UI | — | 새 페이지 X, 하단 토스트 "받을게요/나중에" |

### 고객 입장 시뮬레이션
| 고객 유형 | AS-IS | TO-BE |
|---|---|---|
| 신규 가입자 | 안 물어봄 → 구독 0 | 가입 직후 "🌱 환영해요! 알림 받으시겠어요?" → 구독 |
| 글 쓰는 회원 | 안 물어봄 | "답글 오면 바로 알려드릴까요?" → 구독 → 내 글 댓글 시 폰 알림 |
| 댓글 다는 회원 | 트리거 있으나 **버그로 안 뜸** | 버그 수정 → 토스트 노출 → 구독 |
| 둘러보는 기존 94명 | 영영 안 물어봄 | 글 1개 읽으면 "내 글 소식, 알림 받으시겠어요?" → 구독 |
| 아이폰 사파리(미설치) | 안 뜸 | 여전히 안 뜸(거부감·오류 없이 자연 스킵) |
| 비회원(네이버 유입) | 무관 | 푸시 안 뜸 → 가입 유도 배너가 담당 |
| "나중에" 누른 사람 | — | 1일(2번째 30일) 대기 후 다시 |
| 허용자(며칠 뒤) | 알림 0 | 답글·등급·공지 + (동의 시)광고 알림 수신 → 재방문 |

## 0. 5대 원칙
1. **새 페이지 없음** — 기존 하단 토스트(pre-permission) 재사용.
2. **브라우저 권한창은 "네" 탭 시에만** — cold로 안 띄움 → 영구차단 방지.
3. **최고 전환 순간(③ 글쓰기 직후)을 보호** — 저품질 cold ask로 그 기회를 burn하지 않는다.
4. **시니어 안심 필수** — "언제든 끌 수 있어요" 서브문구. (⚠️ "광고 안 보내요"는 **광고 발송 의도가 생겨 삭제** — 거짓 금지)
5. **한 화면에 동의 요청은 1개만** — 팝업 도배 금지.
6. **광고는 합법적으로만** — "좋아요"=마케팅 동의(소식·혜택 명시) + (광고)표기 + 야간제한 + 수신거부. ⚖️ 정보통신망법 §50, 법무 확인.

## 1. 트리거 · 문구 · 우선순위 (언제 / 누구에게 / 뭐라고)

| 우선순위 | 트리거 | 발동 시점 | 대상 | 메인 문구 | 서브문구 | 버튼 |
|---|---|---|---|---|---|---|
| **1 (최고)** | ③ 글쓰기 직후 | 글 작성 성공 직후 | 로그인 회원·미구독 | 🔔 답글 오면 바로 알려드릴까요? | 우나어 소식·혜택도 함께 · 언제든 끄기 | 받을게요 / 나중에 |
| **1 (최고)** | 댓글 작성 후(기존) | 댓글 작성 직후 | 로그인 회원·미구독 | 🔔 답글 오면 바로 알려드릴까요? | 우나어 소식·혜택도 함께 · 언제든 끄기 | 받을게요 / 나중에 |
| **2** | ② 회원 방문 | 로그인 회원이 그 세션에 **글 1개 이상 열람(engagement)** 후 | 로그인 회원·미구독 | 🔔 내 글 소식, 알림 받으시겠어요? | 답글·소식·혜택 · 언제든 끄기 | 받을게요 / 나중에 |
| **3 (최저)** | ① 가입 직후 | 온보딩 완료 → 홈 첫 진입 | 신규 회원·미구독 | 🌱 환영해요! 알림 받으시겠어요? | 답글·새 소식·혜택 · 언제든 끄기 | 받을게요 / 나중에 |

- **구조**: 메인은 짧은 이득 훅 1줄, **"소식·혜택"은 작은 서브줄에 명시** → 가독성 ↑(동의율 ↑) + 마케팅 동의 근거(informed consent, §7) 유지. "광고 안 보내요" 류 거짓 문구 전면 삭제.
- "받을게요" 탭 = 푸시 허용 + 마케팅 수신 동의.
- 여러 트리거가 한 세션에 겹치면 **우선순위 높은 1개만** 노출(③ > ② > ①).

---

## 2. ① 가입 직후 — 핵심 결정 (트레이드오프)

**상황**: 신규 회원은 아직 쓴 글 0개 → "내 글에 댓글" 이득이 추상적. 게다가 가입 직후 ①에서 "나중에" 누르면 **1일 쿨다운** → 곧이어 첫 글 쓸 때(③, 최고 전환) 토스트가 안 뜸(= ①이 ③을 잡아먹음). 단, 그냥 무시하고 넘기면 쿨다운 없음(**무시 ≠ 거절**).

### ✅ 확정: (나) 환영 톤으로 띄움
- 문구: **"🌱 환영해요! 알림 받으시겠어요?"** / _답글·새 소식·혜택 · 언제든 끄기_ / [받을게요] [나중에]
- **③ 보호 장치(재해석)**:
  - ①을 **무시하고 넘기면 쿨다운 없음**(무시 ≠ 거절) → 같은 세션에서 첫 글 쓰면 ③ **정상 발동**.
  - ①에서 **"나중에"(명시적 거절)** 누르면 1일 쿨다운 → ③에서 또 안 조름. **이건 버그가 아니라 사용자의 'no'를 존중하는 올바른 동작**(5분 만에 두 번 조르기 방지).
  - 결론: 환영 톤으로 가볍게 띄우되 부담 0 → 손해 최소, ③ 대부분 보존.

## 3. 노출 조건 (AND — 전부 충족해야 토스트 노출)

1. **웹푸시 지원 환경** (`PushManager in window`) — iOS 미설치 사파리·일부 인앱 자동 제외
2. **로그인 회원** — 비회원 제외(구독 API가 401, 푸시 타겟 불가). 비회원은 가입 유도 배너가 담당
3. **미구독 상태** (`Notification.permission !== 'granted'` && `push_granted` 없음)
4. **브라우저 영구차단 아님** (`permission !== 'denied'`)
5. **쿨다운 경과** (마지막 거절 후 1일 / 2회+ 거절 후 30일)
6. **상위 팝업 미표시** (PWA 설치팝업 표시 중이면 숨김 — 기존 `POPUP_VISIBLE_KEY`)
7. **세션당 푸시 ask 0회** (이 세션에 아직 안 띄웠음)

하나라도 false → 토스트 미노출.

---

## 4. 충돌·우선순위 (홈 진입 시 동시 후보 정리)

홈 진입 시 뜰 수 있는 것들과 정리:

| 요소 | 대상 | 동의요청? | 푸시와의 관계 |
|---|---|---|---|
| WelcomeToast(가입 환영) | 신규 회원 | ✕(단순 환영) | 공존 OK(요청 아님) |
| PWA 설치 팝업(AddToHomeScreen) | web 회원 | ○ | **푸시보다 우선** — 표시 중이면 푸시 토스트 숨김 |
| **푸시 토스트** | 로그인 회원·미구독 | ○ | PWA팝업 없을 때만 |
| SignupPromptBanner / TwaEntryGate | **비회원** | ○(가입) | 회원/비회원으로 분리 → 푸시와 충돌 없음 |

- **회원**: PWA 설치팝업(web) → 그게 없으면 푸시 토스트. TWA는 PWA팝업이 없어 푸시가 바로.
- **비회원**: 푸시 토스트 안 뜸. 가입 유도 배너가 담당.
- 규칙: **한 화면에 동의 요청 1개.**

---

## 5. 빈도·쿨다운

- **세션당 푸시 ask 최대 1회.** 여러 트리거 겹치면 우선순위 1개만.
- **거절("나중에") 쿨다운**: 1회=1일, 2회+=30일(기존 `canAskPushPermission`).
- **허용 / 브라우저 영구차단** 후 = 다시 안 띄움.

## 6. 환경별 동작

| 환경 | 토스트 | 비고 |
|---|---|---|
| 갤럭시 TWA(앱) | ✅ | **주력 채널.** PWA팝업 없어 깔끔. ⚠️ OS 알림 실제 전달은 실기기 검증 필수 |
| 안드로이드 크롬(web) | ✅ | PWA 설치팝업과 순서 조정 |
| 아이폰 — 홈에 설치(PWA) | ✅ | iOS 16.4+ |
| 아이폰 사파리(미설치) | ❌ | PushManager 없음 → 자동 비노출(정상) |
| 카카오/네이버 인앱 | ❌(대개) | 푸시 미지원 → 비노출(graceful) |
| PC 크롬 | ✅ | 데스크탑 알림 |

## 7. 발송 — 서비스 알림 vs 광고 (2종 분리)

| 종류 | 받는 사람 | 표기·제한 | 예 |
|---|---|---|---|
| **서비스 알림** (댓글·답글·등급) | 푸시 구독자 **전원**(실고객·ACTIVE·본인행동 제외) | 없음 | "내 글에 댓글 달렸어요" |
| **광고/마케팅** (혜택·이벤트) | 구독 **+ `marketingOptIn=true`** 만 | **(광고) 표기** + **야간 21~08 차단** + 수신거부 안내 | "(광고) 우나어 이번 주 이벤트" |

- **공통 조건**: 수신자 구독 보유 + 실고객(봇 아님)·ACTIVE + `flags.webPush` ON. 미구독이면 종🔔만, OS 푸시 no-op.
- **광고 동의 출처**: 온보딩 마케팅 동의 **또는** "받을게요"(소식·혜택 명시) 탭 시 기록(`marketingOptIn=true` + `Agreement(MARKETING)`).
- **자동 발송**: 댓글·답글·등급(서비스)만. 새 글·좋아요·스크랩 ❌(남발 방지). 광고는 **어드민 수동 브로드캐스트**.

### 7-1. ⚠️ 헷갈리기 쉬움 — "권한(기술)" vs "동의(법)"은 다르다
| 메시지 종류 | ① OS 푸시 권한(구독) | ② 마케팅 수신 동의(법) |
|---|---|---|
| **정보성/서비스** (답글·댓글·등급) | ✅ **필요**(기술적으로) | ❌ 불필요 |
| **광고성** (혜택·이벤트·할인) | ✅ 필요 | ✅ **필요**(정보통신망법 §50) |
| **앱 안 종(🔔) 알림** | ❌ 불필요 | ❌ 불필요 (앱 방문 시에만 표시) |

- **핵심**: 정보성(답글)은 **마케팅 동의 없이** 보내도 되지만, **"폰에 뜨는 OS 푸시" 자체는 정보성이든 광고든 권한(①)이 무조건 필요**. 권한 없이 폰 알림 띄우는 방법 없음(앱 안 종 알림만 예외).
- 우리 설계의 "받을게요" = ①(권한) + ②(마케팅 동의)를 한 번에 획득(법무 확인 완료).
- ⚖️ **경계 사례**(예: "오랜만이에요~ 새 글 보러 오세요")는 광고로 볼 소지 → **마케팅 동의자에게만**. 애매하면 법무 확인.
- 설계 상세: [push-permission-design-spec.md §5.5](./push-permission-design-spec.md).

## 8. 측정(구독률·효과)

- `PushSubscription` 행 수 추이(0 → 증가) = 구독률.
- 트리거별 노출/허용/거절 이벤트(EventLog) → 어느 트리거가 잘 먹히는지.
- (Phase 2) 발송 후 클릭률 → 리텐션 기여.

---

## 9. 범위 / 비범위

**포함**: 구독 수집 + **광고 컴플라이언스**(동의기록·서비스/광고 분리·(광고)표기·야간제한·marketingOptIn 필터·어드민 광고 발송).
**제외**: 새 글/좋아요 **자동** 푸시 ❌ / 재참여 **자동 스케줄** = Phase 2 / 서비스워커·VAPID·DB **스키마** 변경 ❌(marketingOptIn·Agreement는 기존 필드 write).
