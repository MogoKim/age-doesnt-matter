### 5-3. 보안 취약점

#### 🔴 P0: Rate Limit 실효성 문제 (2중 문제)

**문제 A: Upstash Redis env var 미설정 — 기존 적용 코드도 무효**
```
UPSTASH_REDIS_REST_URL  → 미설정
UPSTASH_REDIS_REST_TOKEN → 미설정
```
현재 rate-limit.ts가 in-memory 폴백으로 동작 중. Vercel 서버리스는 인스턴스별로 메모리가 분리되어 **분산 Rate Limit 효과 없음**. 즉 "/api/push/subscribe (10/분)" 등 이미 적용된 14개도 사실상 무효.

**문제 B: 미적용 API 21개 (정확한 목록)**
```
공개 API (인증 없음, Rate Limit 없음):
- /api/ad-click POST          ← 광고 클릭 조작
- /api/posts/[postId] GET     ← DB 부하
- /api/posts/[postId]/view POST ← 조회수 조작
- /api/health GET             ← 의도적 공개 (OK)
- /api/health/auth GET        ← 의도적 공개 (OK)
- /api/threads/auth GET       ← 외부 OAuth 트리거
- /api/threads/callback GET   ← 외부 OAuth 트리거
- /api/user/pwa-status POST   ← 상태 위변조

Bot API (Bearer Token 있음, 그래도 Rate Limit 권장):
- /api/bot/check GET
- /api/bot/jobs POST
- /api/bot/logs POST
- /api/bot/posts POST

약한 Admin API (x-admin-secret 헤더, Rate Limit 없어 무차별 대입 가능):
- /api/admin/check-post GET
- /api/admin/revalidate-deleted POST

기타 공개 엔드포인트:
- /api/popups GET (GET은 Rate Limit 없음, POST에만 있음)
```
**우선 조치: Upstash Redis env var를 Vercel에 추가 → 기존 14개 즉시 실효화 + 21개 추가 적용**
